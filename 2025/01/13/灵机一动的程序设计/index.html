怎样用远端缓存实现近端缓存的亲和性

类似 diamond 的天河多维度写入，怎样在不同机房里执行特定的策略：

  1. 给配置按key配置行-多版本带来多行
  2. 给机器配置行-机器各自激活
  3. 给总体策略配置行-如果只有当前行，则这个策略行可以写到配置行里
  4. 机器配置关联行-中控主程序主动更新

然后机器各自读取机器配置关联行得到最新配置。

怎样让配置按机器下发，实现机器级的 configuration 下发

机器不要有 inactive 和 active 状态，这样就不需要专门的定时任务基于时间更新状态。

服务器查询配置信息通常先走缓存，然后再走数据库，而管理员通过界面查询配置则需要直接走数据库。

注册可以固定时间定时执行，而根据注册信息来处理拓扑信息则必须选主。

模仿 git 的逻辑，每个不同的json在一个环境里有一个单独的 version，类似 git 在不同的链上，即使做了 cherry-pick，也仍然有不同的 hash 一样。

对新旧模板进行对比

使用一个开关值表明当前使用的模板类型，如果当前使用新模板，则拉取旧模板进行比对；反之，则拉取新模板进行比对。

好的设计应该尽量采用旁路安插新配置的方法

这样老流程在切换中浑然不觉，而新流程把老流程当做自己的一个子环节，提供了兼容性，可前进可后退。

比如如果我们要切换插件的功能，不如在插件前面放一个更高优的插件，这样插件读写的数据不用变，更高优的插件使用更新的存储。

所有线上正在被读写的新接口和配置，都是最好使用旁路更改的模式升级的，否则只能停机硬切。

redis 在没有使用事务的时候应该遵循的原则

  1. 如果有嵌套的多个 set 或者 kv 要写，尽可能先写宏观的 数据结构，再写细节的。
  2. 如果新增了数据结构，正操作要增加，反操作也要增加。
  3. 强依赖的污染标识 kv 的 expire 一定要足够长，甚至无限期。

怎样让一个真实号针对许多个业务降级

  1. 使用一个与业务无关的 kv 来存储这个号码。
  2. 把这个号码放进业务开头的 set 里面-这样设计默认一个业务下有多个真实号需要降级。
