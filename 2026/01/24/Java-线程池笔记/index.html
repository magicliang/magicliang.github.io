Java 线程池笔记

线程池不是“帮你少 new 几个线程”这么简单。它真正解决的是一个更大的工程问题：任务如何被提交、包装、排队、执行、取消、回收，以及如何在有限线程资源下维持吞吐、延迟和边界控制。

这篇文章不再试图把 Java 并发世界里的所有高级概念都吞进来，而是专注于“执行模型”这一层。你会看到：Executor 只是契约，FutureTask 是任务包装桥梁，ThreadPoolExecutor 是经典生产者—消费者执行模型，ScheduledThreadPoolExecutor 是时间调度模型，ForkJoinPool 是分治并行模型，而 CompletableFuture 则更像建立在这些执行器之上的异步编排层。

所以本文的核心不是“记 API”，而是先分清这些组件分别在回答什么问题：

  • Executor / ExecutorService：任务提交契约是什么？
  • FutureTask：无返回值任务如何变成可等待、可取消、可获取结果的异步单元？
  • ThreadPoolExecutor：有限线程如何消费共享队列中的任务？
  • ScheduledThreadPoolExecutor：时间语义如何转成调度语义？
  • ForkJoinPool：为什么分治并行不适合用传统线程池思路？
  • CompletableFuture：如何把“单次异步执行”提升为“阶段图式编排”？

一、先分清：线程池到底在池化什么

1.1 不是“任务”，而是“执行资源”

线程池池化的不是任务，而是执行任务所需的线程资源和调度能力。任务本身仍然是一次性的，只不过它们被：

  • 统一提交；
  • 统一排队；
  • 由复用的工作线程取出执行；
  • 在有限资源约束下完成生命周期管理。

所以“线程池”的关键价值不是复用这个词本身，而是：

  1. 有界地消耗线程资源；
  2. 把任务提交和执行线程解耦；
  3. 允许调度、隔离、拒绝、监控和关闭形成一套统一模型。

1.2 线程池回答的是资源治理问题

如果没有线程池，最朴素的模型就是“来一个任务，起一个线程”。这在少量任务时看起来简单，但一旦任务规模增大，就会暴露出一连串问题：

  • 线程创建与销毁成本高；
  • 线程数量无限膨胀；
  • 没有统一排队与背压；
  • 无法集中做监控和关闭；
  • 无法表达“我宁愿拒绝，也不要把系统拖垮”。

从这个角度看，线程池首先是一个资源边界控制器，其次才是一个并发执行器。

二、Executor 体系：先定义契约，再定义参考实现

2.1 Executor：最小执行契约

Executor 只定义一件事：给我一个任务，我来执行。

这个接口最重要的价值，不是异步，而是解耦：

  • 调用方只描述“做什么”；
  • 执行器决定“在哪儿做、何时做、由谁做”。

所以 Executor 并不天然等于线程池，也不天然等于异步。一个 DirectExecutor 完全可以在当前线程里同步执行任务。

2.2 ExecutorService：从“执行”扩展到“生命周期与结果”

一旦引入 ExecutorService，事情就变得更完整了。它在 execute() 之上补齐了几个关键能力：

  • 提交带结果的任务；
  • 批量提交；
  • 取消与关闭；
  • 等待执行器终止。

这意味着执行器不再只是“把 Runnable 扔出去”，而是变成了一个可治理的异步执行框架。

2.3 AbstractExecutorService：把高级 API 串回 execute()

这是一个很值得讲清楚的层次：

  • ExecutorService 定义了很多高级接口；
  • AbstractExecutorService 用模板方法把这些接口的默认实现串起来；
  • 真正的底层实现者往往只要把 execute() 和生命周期关键方法补上。

其中最关键的桥梁是 newTaskFor()：它把 Runnable / Callable 包装成 RunnableFuture，从而让“提交任务”和“未来拿结果”连接起来。

一句话说，就是：

契约层定义能力，模板层把能力接上线，实现层再决定实际调度策略。

三、FutureTask：线程池世界里最重要的单任务桥梁

3.1 为什么 FutureTask 是关键桥梁

如果只看线程池，很容易误以为核心是工作线程和阻塞队列。但从接口演化上看，FutureTask 同样关键，因为它连接了两个世界：

  • 一边是 execute(Runnable) 这种“只管跑”的执行接口；
  • 另一边是 Future.get() 这种“我要等结果 / 看异常 / 做取消”的观察接口。

它让一个任务不再只是“一段要执行的代码”，而是一个带生命周期、状态机和结果通道的异步单元。

3.2 它解决了三件事

一个标准的 FutureTask 主要解决：

  1. 结果承载：任务完成后结果放哪；
  2. 终态表达：正常完成、异常完成、取消完成如何区分；
  3. 等待机制：调用方如何阻塞等待直到任务终态。

这也是为什么线程池里的很多高级能力，最后都会落回“任务对象要不要有状态机”。

3.3 execute 与 submit 的根本差异

这是工程里最容易混用的一组 API。

API 提交的是什么 返回什么 异常去哪儿
execute(Runnable) 原始任务 无返回值 更接近线程级未捕获异常路径
submit(...) 先包装为 FutureTask Future 被任务包装层拦住，等 get() 再取出

所以两者的关键差异不只是“有没有返回值”，而是：

submit 会先把任务包装成一个可观察的状态机对象，而 execute 更接近直接把任务交给执行器。

3.4 Future.get(timeout) 的语义边界

这也是本系列必须固定边界的地方：

  • get(timeout) 解决的是等待超时；
  • 它不自动等于“底层任务已经停止”。

这个差异会在《跨语言超时机制全解析》里展开说明。本文只保留执行模型层的结论：FutureTask 的超时主要作用于等待者，而不是自动终止工作本体。

四、ThreadPoolExecutor：经典的共享队列执行模型

4.1 它解决的是“有限线程消费共享任务流”

ThreadPoolExecutor 的思路非常经典：

  • 生产者持续提交任务；
  • 任务进入共享队列或直接交给新 Worker；
  • 有限数量的工作线程反复从队列中取任务执行。

所以它天然适合：

  • 任务来源多；
  • 任务彼此相对独立；
  • 需要统一做排队、隔离、背压和拒绝。

4.2 最核心的调度规则一定要写对

这是本系列事实核验里已经锁死的一条：

ThreadPoolExecutor 的核心调度顺序是：先补到 corePoolSize，再尝试入队，队列满后才继续扩到 maximumPoolSize，最后才拒绝。

这意味着两个非常重要的后果：

  1. 无界队列下，maximumPoolSize 基本不会起作用。
  2. 调线程池时，队列语义往往比 max 值更重要。

4.3 ctl：为什么线程池状态和线程数打包在一起

原文对 ctl 的分析很深入，保留下来很有价值，但不需要铺开太多位运算细节。更值得保住的是设计意图：

  • 线程池要同时关心池生命周期状态与当前工作线程数；
  • 这两个字段常常需要一起判断、一起 CAS 更新；
  • 所以 HotSpot/JUC 选择把它们打包进一个原子整型字段里。

这是一种非常典型的 field packing 设计：用一个原子字段承载多个逻辑位，降低锁需求，保证复合决策的一致性。

4.4 Worker：线程池里“线程”与“执行循环”的结合体

ThreadPoolExecutor.Worker 不是单纯的线程引用，而是：

  • 一个工作线程；
  • 一个执行主循环；
  • 一个围绕中断控制和任务边界的小型同步单元。

所以 ThreadPoolExecutor 管理的并不只是裸线程，而是“线程 + 执行语义”这一组合体。

4.5 为什么说它是生产者—消费者模型

因为它最本质的结构就是：

  • 外部线程不断往共享队列塞任务；
  • Worker 从共享队列取任务；
  • 队列负责缓冲瞬时流量波动；
  • 拒绝策略负责在边界超限时做最后决策。

这也是为什么 TPE 的核心关注点总是：

  • 队列怎么选；
  • 线程怎么增减；
  • 什么时候拒绝；
  • 关闭时哪些线程该被中断。

五、ThreadPoolExecutor 的工程关注点：队列、拒绝、关闭、监控

5.1 队列不是附属品，而是线程池行为的核心塑形器

很多人调线程池只盯线程数，这是不够的。队列类型几乎决定了线程池的整体行为：

  • 是否缓冲大量任务；
  • 是否让 maximumPoolSize 生效；
  • 是否容易堆积出长尾延迟；
  • 是否会在高峰期把系统拖成大排队。

最典型的结论就是：

无界队列很容易把线程池从“限流器”变成“堆积器”。

5.2 拒绝策略不是异常分支，而是系统边界的一部分

当任务无法继续接纳时，线程池必须明确回答：我现在怎么办？

内置拒绝策略分别对应不同态度：

  • AbortPolicy：明确失败；
  • CallerRunsPolicy：把压力回推给调用方；
  • DiscardPolicy：静默丢弃；
  • DiscardOldestPolicy：牺牲更早排队的任务。

其中最值得强调的是：

CallerRunsPolicy 的本质是回压，不是“换个线程也异步跑”。

5.3 shutdown() 与 shutdownNow() 解决的是两种不同问题

  • shutdown()：停止接收新任务，但尽量把已提交任务跑完；
  • shutdownNow()：进入更激进的停止路径，尝试中断执行中的线程，并清空未执行任务。

这里需要和前一篇线程原语文衔接：

  • 线程池可以发出中断；
  • 但任务是否真正停下，仍取决于任务代码如何响应中断。

所以“关闭线程池”从来不是一句话能搞定的动作，它依赖执行器状态、Worker 状态和任务响应能力共同配合。

5.4 监控与命名不是附加美化，而是可运维性的前提

线程池如果没有：

  • 明确线程名；
  • 队列长度监控；
  • 活跃线程数监控；
  • 拒绝次数监控；
  • 任务耗时分布；

那么线上一旦出问题，定位会非常痛苦。

从工程角度看，线程池不是“配个参数就完”，而是一个长期暴露在负载波动中的可观测对象。

六、ScheduledThreadPoolExecutor：把时间语义转成调度语义

6.1 它不是“普通线程池 + sleep”

ScheduledThreadPoolExecutor 的本质，是在执行器模型上再叠加一层时间维度：

  • 某个任务多久后执行；
  • 是按固定频率还是固定延迟重复执行；
  • 周期任务异常后是否还会继续。

因此它不是简单的“线程池里手动睡一会儿”，而是一个专门处理延迟与周期语义的调度器。

6.2 scheduleAtFixedRate 与 scheduleWithFixedDelay 的差别必须讲清

这两者最容易被误当成“差不多”。其实它们表达的是两种不同的时间观：

  • fixed rate：按理论时间轴推进，更关心节拍；
  • fixed delay：以上一轮真正结束的时刻为起点，更关心间隔。

一句话概括：

fixed rate 更像“按时钟打点”，fixed delay 更像“干完休息一会再干下一轮”。

6.3 周期任务异常为什么会“静默停掉”

这是工程里非常高频的坑。周期任务不是天然会无限重试；如果任务抛出未处理异常，后续调度可能直接终止。

所以周期任务里最重要的防御性规则之一是：

周期任务必须自己兜住异常，否则你以为它还在跑，其实它早就悄悄停了。

6.4 cancel 后是否立刻从队列清掉，是一个很现实的问题

原文对 setRemoveOnCancelPolicy(true) 的强调是对的，因为：

  • 任务被取消并不自动等于它立刻从延迟队列里消失；
  • 如果不及时清理，积累多了会影响队列体积和监控认知。

这说明 ScheduledThreadPoolExecutor 的问题不只是“按时跑没跑”，还包括“取消后队列怎么保持干净”。

七、ForkJoinPool：分治并行不是共享队列消费问题

7.1 它不是通用线程池的替代品，而是另一种执行模型

ForkJoinPool 经常被误当成“更高级的线程池”，但它真正针对的是分治并行任务树：

  • 父任务拆出子任务；
  • 子任务之间大多独立；
  • 父任务经常要等待子任务结果；
  • 任务倾向于计算密集，而不是 I/O 阻塞。

这和传统的“共享任务流 + Worker 消费者”不是一类问题。

7.2 为什么它用工作窃取，而不是共享阻塞队列

ForkJoinPool 的核心设计是：

  • 每个工作线程有自己的本地队列；
  • 本地优先处理自己拆出来的任务；
  • 空闲时再去窃取别人的任务。

这样做的关键收益是：

  • 大多数操作落在本地队列，竞争低；
  • 更贴合任务树结构；
  • join() 场景下能通过协作执行减少传统线程池饥饿死锁。

7.3 TPE 与 FJP 的差异，核心不在 API，而在任务形态

维度 ThreadPoolExecutor ForkJoinPool
典型任务形态 独立任务流 分治任务树
队列模型 共享队列 工作线程私有队列 + 窃取
主要问题 背压、排队、拒绝 负载均衡、窃取、协作等待
更适合 普通业务任务、服务端请求处理 CPU 密集型分治计算

所以不能因为它们都实现了 ExecutorService，就把它们当成“一个东西的不同参数版本”。

7.4 FJP 里最危险的误用：拿它跑阻塞 I/O

ForkJoinPool 的设计前提是：工作线程应该尽量持续参与计算和窃取。如果你把大量阻塞 I/O 扔进去：

  • 工作线程会被长期挂住；
  • 工作窃取优势被削弱；
  • 公共池里的其他计算任务也可能被连带拖慢。

这也是为什么 FJP 更像“并行计算池”，而不是“通用业务线程池”。

八、CompletableFuture：执行器之上的异步编排层

8.1 它解决的不是“怎么建线程池”，而是“怎么拼异步阶段图”

把 CompletableFuture 放在线程池文章里讲，很容易误导读者以为它只是“Future 的升级版”。更准确地说，它是在问另一个问题：

当异步任务不再是一个，而是一串带依赖关系的阶段时，我怎么描述它们的组合、传递和汇合？

所以 CompletableFuture 更接近编排层，而不是底层执行器本体。

8.2 它仍然依赖执行器，但不等于执行器

CompletableFuture 的很多异步阶段最终还是要落到某个执行器上：

  • 显式传入的线程池；
  • 默认的公共池；
  • 某些场景下的调用线程。

也就是说，它不是取代执行器，而是站在执行器之上做阶段编排。

8.3 它和 FutureTask 的关系

可以用一句很清晰的口径区分：

  • FutureTask：单任务结果句柄；
  • CompletableFuture：异步阶段图节点。

前者解决“一个任务将来有个结果”，后者解决“多个阶段如何组合成一张流图”。

8.4 超时、取消、异常传播不要在这里讲成总论

这也是本次系列裁剪的硬边界之一：

  • CompletableFuture 可以承载超时与异常传播逻辑；
  • 但超时语义本身应回到《跨语言超时机制全解析》；
  • 取消传播与任务作用域应回到《Java 结构化并发》。

本文只讲一层：它如何作为编排工具，挂接在线程池和执行器之上。

九、选型与实践：别把不同层级的问题混成一个答案

9.1 一个够用的执行器选择顺序

可以先用下面这条线索做第一轮选择：

  1. 普通业务任务流：先看 ThreadPoolExecutor。
  2. 需要延迟或周期调度：看 ScheduledThreadPoolExecutor。
  3. CPU 密集型分治计算：看 ForkJoinPool。
  4. 需要异步阶段组合：在前面执行器之上再看 CompletableFuture。

9.2 最常见的误用

下面这些误用在生产里都非常常见：

  1. 用无界队列，却以为 maximumPoolSize 会保护系统。
  2. 用 newCachedThreadPool() 跑不可控流量，最后线程爆炸。
  3. 周期任务不兜异常，结果调度 silently stop。
  4. 把 FJP 当通用业务线程池，塞进大量阻塞 I/O。
  5. 以为 Future.get(timeout) 超时就等于任务已经终止。

9.3 线程池实践的核心不是“参数口诀”，而是边界意识

线程池真正要回答的是：

  • 我最多愿意吃下多少任务？
  • 排队能排多久？
  • 压力大时我是降级、回压还是拒绝？
  • 关闭时我希望已提交任务怎样收束？

没有这些边界意识，调参就只是在做数字体操。

  • 《Java 结构化并发》负责：任务作用域、取消传播、失败收束，以及它和线程池之间的层级区别。
  • 《跨语言超时机制全解析》负责：timeout、deadline、cancellation、interruption 的统一语义框架。

这样分工以后，本文不再承担“中断总论”“超时总论”“结构化并发总论”，而是专注于“任务如何被执行器管理”。
