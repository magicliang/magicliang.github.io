无锁队列

很多人第一次接触无锁队列，往往会把它理解成“用 CAS 代替锁实现一个队列”。这个说法不能说错，但远远不够。真正值得理解的，不是“把 synchronized 去掉了”，而是：我们到底想让队列在什么并发形态下继续前进，它靠什么不变量维持 FIFO 语义，以及它把原本由锁承担的复杂度转移到了哪里。

所以本文不再把 SPSC、SPMC、Disruptor、MPMC 当成几段并排代码来展示，也不把一个手写的可疑 MPMC 示例当成学习中心。更合理的顺序是：先讲无锁队列到底在解决什么，再讲非阻塞进展保证，再讲不同生产者/消费者模型的差异，最后再回到 Michael-Scott 队列与 Java 工业实现 ConcurrentLinkedQueue。

本文拥有以下主题的定义权：blocking / lock-free / wait-free / obstruction-free 的差别、SPSC / MPSC / SPMC / MPMC 队列模型、Michael-Scott 队列的不变量、helping、head/tail lagging、ABA 与内存回收问题、Java 中 ConcurrentLinkedQueue 的工程边界。 JMM、volatile、happens-before 的规范定义已在《JVM 的内存模型与线程》中给出；CAS 与锁、AQS 的一般取舍已在《线程安全与锁优化》中给出，本文只在需要时回链，不再重复开课。

一、无锁队列到底在解决什么问题

1.1 无锁不是“没有同步”

“无锁”这个名字很容易误导人，好像它描述的是一种“完全不需要协调”的状态。事实恰恰相反：无锁队列往往比加锁队列更依赖精密的同步，只不过这种同步不再表现为“进入一个互斥临界区”，而是表现为：

  • 原子更新共享指针或序号；
  • 在发布节点或槽位时建立正确的可见性边；
  • 在竞争失败时重试，必要时帮助其他线程推进结构。

所以“无锁”真正省掉的不是同步，而是把整个队列的前进能力交给某一个持锁线程来决定。

1.2 它的核心价值是改变进展模型

加锁队列的典型问题不是“语义不正确”，而是：一旦某个线程持锁后被挂起、长时间停顿、或者在临界区里被阻塞，其他线程的推进就会被连坐。无锁队列试图改变的是这一点：

它不承诺每个线程都会立刻成功，但会尽量保证系统整体持续前进，而不是被某个持锁线程卡死在临界区边界。

这就是为什么讨论无锁队列时，第一件事不该是“CAS 怎么写”，而该是“它提供什么 progress guarantee”。

1.3 这不是“无锁一定比锁快”

这里必须先把一个最常见的误解掐掉：lock-free 并不自动意味着更快。

它只是换了一种成本模型：

  • 锁的成本主要在阻塞、唤醒、调度与等待队列管理；
  • 无锁的成本主要在 CAS 失败重试、缓存一致性流量、自旋与更复杂的不变量维护。

低竞争、短临界区时，锁可能已经足够便宜；高竞争下，无锁也可能因为重试风暴而吞掉大量 CPU。所以是否选择无锁，必须回到负载形态与进展需求，而不能只看名字。

二、先把术语摆正：什么叫 non-blocking progress

2.1 blocking、obstruction-free、lock-free、wait-free 不是一回事

并发文章里经常把“非阻塞”说成一个笼统好词，但这些术语的强度其实不同：

级别 含义 直观理解
blocking 线程可能因为等待锁或条件而停住 一个线程卡住，别人也可能被它连带卡住
obstruction-free 若线程独占运行足够久，它能完成操作 没竞争时能走通，但不保证竞争下整体推进
lock-free 系统整体持续取得进展 总有人能成功，但某个具体线程可能一直失败
wait-free 每个线程都能在有限步内完成操作 最强保证，也最难普遍做到

这里最重要的一点是：大多数通用无锁队列在工程上讨论的是 lock-free，而不是 wait-free。

2.2 为什么 lock-free 已经很强，但仍不是“人人公平”

如果一个队列是 lock-free，意味着在持续竞争中，系统不会因为某个线程停顿就整体失去推进能力。但这不意味着每个线程都一定很快成功。某个线程完全可能：

  • 一次次读到过时的 tail 或 head；
  • 一次次 CAS 失败；
  • 长时间在高竞争下输给别的线程。

所以 lock-free 保证的是 global progress，不是线程级的公平完成。把它误读成“每个线程都不会饿死”，就会把语义说大了。

2.3 为什么 wait-free 在通用 FIFO 队列里更难

FIFO 队列不是单个计数器。它涉及：

  • 入队和出队两个方向的结构修改；
  • 多个线程对 head / tail / next 的竞争观察；
  • 保持顺序语义而不是“谁抢到谁先出”。

因此 wait-free FIFO 队列通常比 lock-free 版本复杂得多，也更难成为标准库默认选项。理解这一点之后，你就不会再把“标准库没有 wait-free 通用队列”误读成“库设计不够先进”。

三、队列设计的第一问不是“链表还是数组”，而是并发形态

3.1 SPSC、MPSC、SPMC、MPMC 是四种不同问题

很多材料把这四个缩写当成部署标签，好像只是线程数量不同而已。实际上，它们代表的是不同的正确性契约：

模型 生产者 消费者 复杂度直觉
SPSC 单生产者 单消费者 最容易，常可避免 CAS
MPSC 多生产者 单消费者 入队难，出队相对简单
SPMC 单生产者 多消费者 发布简单，消费协调更难
MPMC 多生产者 多消费者 最一般，也最难

这里的关键不是缩写本身，而是：线程所有权一旦从“唯一写者”变成“多人竞争写者”，算法的成本结构就完全变了。

3.2 为什么 SPSC 能大幅简化

SPSC 场景的美妙之处在于：

  • 只有一个线程推进写端；
  • 只有一个线程推进读端；
  • 头尾索引天然分别归属于两个线程。

这样一来，很多实现可以把复杂度集中在“如何正确发布元素”，而不是“如何仲裁多个写者或多个读者的冲突”。这也是为什么高性能环形缓冲常从 SPSC 起步：不是因为它“更高级”，而是因为它利用了 single-writer principle。

3.3 为什么模型一旦扩大，算法就不能线性照搬

一个在 SPSC 下成立的实现，直接拿去做 MPMC，通常会立刻暴露问题：

  • 多个生产者同时占用同一个槽位；
  • 多个消费者同时看到同一个头节点；
  • 发布顺序与消费顺序被并发竞态打乱；
  • 单纯依赖普通 volatile 索引已不足以维护结构不变量。

所以真正稳妥的顺序应该是：先确定队列模型，再选择算法；不要先写一个“看起来能跑”的队列，再事后给它贴上 MPMC 标签。

四、为什么 MPMC 才是无锁队列真正困难的地方

4.1 困难不只是“多了几个 CAS”

MPMC 的难点，不在于“把单线程代码外面再包几次 CAS”，而在于多个线程会同时观察并修改同一条链或同一个索引系统。线程看到的信息，在它准备行动时就可能已经过时。

这意味着算法必须在下面这些现实条件下仍然正确：

  • tail 可能落后于真实尾部；
  • head 可能已经被别的消费者推进；
  • 某次 CAS 失败不代表结构损坏，只代表别人先一步完成了同类工作；
  • 线程不能假设“我观察到的局面，在我提交更新时仍然成立”。

4.2 正确性依赖不变量，而不是依赖“指针永远最新”

很多初学者实现队列时，会把 head 和 tail 当成必须时刻精确的“真相指针”。Michael-Scott 这类算法更成熟的地方恰恰在于：它不要求每个线程都瞬时看见全局最新状态，而是要求即使 head / tail 有滞后，结构不变量仍然成立，其他线程也能继续把系统往前推。

这就是“helping”存在的理由：

  • 如果你发现别人的入队已经把节点挂上了，但 tail 还没跟上；
  • 你不必等待原线程回来善后；
  • 你可以帮助推进 tail，让结构回到更有利于后续线程观察的状态。

4.3 无锁队列把复杂度从临界区转移到结构维护

加锁队列的正确性往往来自一句简单的话：同一时刻只有一个线程在改内部结构。无锁队列没有这把大伞，于是必须用更细粒度的机制补回来：

  • 逻辑删除与物理推进分离；
  • 元数据允许短暂滞后；
  • 线程之间互相帮助完成结构修复；
  • 正确性证明依赖严密不变量，而不是“经验上应该没事”。

因此，学习无锁队列时最该保留的敬畏感是：看懂几段代码，远不等于真的理解了算法的正确性边界。

五、Michael-Scott 队列：通用 lock-free FIFO 的经典参照物

5.1 为什么它是学习中心，而不是某个随手写的 MPMC 示例

Michael & Scott 在 1996 年给出的 non-blocking queue 算法之所以经典，不是因为“它很老”，而是因为它抓住了通用 lock-free FIFO 队列最核心的一组问题：

  • 多生产者如何并发追加节点；
  • 多消费者如何并发移除节点；
  • head 和 tail 允许滞后时，如何仍然保持 FIFO 语义；
  • 如何通过 helping 保证系统整体推进。

这也是为什么 Java 标准库 ConcurrentLinkedQueue 的官方文档会明确指出，它基于 Michael-Scott non-blocking queue 算法。

5.2 先理解 dummy node，很多推理才会顺

Michael-Scott 队列通常以一个 dummy / sentinel node 开始。这样做的价值，不只是“代码好写一点”，而是它把队列边界条件大幅收敛了：

  • head 永远指向一个占位节点，而不是直接指向“当前第一个真实值”；
  • 真正的首元素通常是 head.next；
  • 入队与出队对“空队列”“只剩一个元素”等边界情况的处理更统一。

如果没有这个 dummy node，队列在空与非空之间切换时会出现更多特殊分支，正确性推理也会变得更脆。

5.3 入队的关键不是立刻改 tail，而是先把节点真正接进链表

Michael-Scott 风格的入队逻辑里，最关键的一步不是“把 tail 改成新节点”，而是：

  1. 先确认当前尾节点的 next == null；
  2. 用 CAS 把新节点挂到这个 next 上；
  3. 挂接成功后，再尝试把 tail 往前推进。

这里的顺序非常重要，因为真正决定“新元素已进入队列”的，是它已经成为链表末尾节点的 next，而不是某个线程是否已经把 tail 指针更新到了最前面。

这也是“tail lagging”在算法里被允许存在的根本原因。

5.4 出队的关键不是“删值”，而是推进 head

与很多初学者直觉不同，出队不是对“第一个值节点”做原地删除。更稳妥的理解方式是：

  • 读取当前 head 与它的 next；
  • 如果 next == null，说明逻辑上为空；
  • 否则尝试把 head 向前推进到 next；
  • 成功推进后，原 next 持有的值就被视为出队结果。

这种写法的本质，是把“移除头元素”转化成“让 dummy node 向前滑动”。这样做不仅边界统一，也更适合无锁并发下的正确性证明。

5.5 helping 不是附加优化，而是进展保证的一部分

如果某个线程看到：

  • tail.next != null，说明有线程已经成功把新节点接进链表；
  • 只是 tail 自己还没被推进到最新尾部；

那么当前线程可以帮忙把 tail 往前推。这个 helping 动作不是“顺手优化一下缓存命中率”，而是 lock-free 设计的重要组成部分：它避免队列结构的推进依赖于最初那个线程必须回来完成最后一步。

这类设计背后的思路很值得记住：

在 lock-free 算法里，线程不是各干各的；它们经常要在竞争中顺便替彼此完成一部分结构维护，以换取系统整体前进。

六、无锁队列真正微妙的地方：ABA、内存回收、滞后指针与重试

6.1 ABA 不是一句“Java 里没有”就能打发掉

ABA 问题的经典描述是：某线程先看到值 A，稍后再次看到还是 A，于是误以为“中间没人改过”，但实际上它可能经历了 A → B → A。

在 Java 里，因为对象由 GC 管理，很多低层无锁算法里最棘手的“节点被释放后地址复用”问题会被大幅缓解，这和 C/C++ 手工内存回收环境确实不同。但这并不等于可以粗暴地下结论说“Java 没有 ABA”。更准确的说法是：

  • GC 改变了 ABA 与内存回收的风险形态；
  • 它让某些“节点被回收并复用同一地址”的问题不再那么尖锐；
  • 但并发观察与状态误判仍然需要谨慎建模。

所以本文保留 ABA 作为概念边界，但不把它讲成一句过度绝对化的口号。

6.2 内存回收为什么在托管运行时和非托管运行时里差别很大

Michael-Scott 原论文与很多后续无锁结构讨论，会把大量篇幅放在 memory reclamation 上。这是因为在非托管语言里，节点什么时候可以安全释放，本身就是正确性的一部分；释放早了会造成悬垂指针，释放晚了会泄漏。

而在 Java 中，GC 帮你接管了这件事的一大部分，所以 ConcurrentLinkedQueue 之类的实现可以把复杂度集中在并发结构本身，而不是 hazard pointers、epoch-based reclamation 这类回收协议上。

这也是为什么：

  • 学无锁算法时，必须知道 memory reclamation 是一等公民问题；
  • 但写 Java 工程文时，不应把大量篇幅投入到不属于当前运行时主要痛点的回收机制上。

6.3 head / tail 滞后不是 bug，而是算法允许的中间态

Michael-Scott 风格队列里，head 与 tail 都可能短暂落后于“逻辑上更准确的位置”。这不是实现粗糙，而是一种刻意接受的中间状态，因为：

  • 强迫所有线程实时同步到“绝对最新指针”，代价很高；
  • 只要关键不变量仍成立，滞后并不破坏正确性；
  • 其他线程可以通过 helping 把这些指针逐步推向更新的位置。

因此阅读这类算法时，思维习惯需要切换一下：不要问“为什么这里的 tail 不总是最准”，而要问“即便它暂时不准，为什么 FIFO 与进展保证仍不被破坏”。

6.4 CAS 失败不是异常，而是正常流量的一部分

在锁语境里，失败往往意味着出了问题；在无锁语境里，CAS 失败通常只是“别人刚刚已经完成了同类更新”。

所以正确的阅读姿势不是把每次失败都当成异常路径，而是明白：

  • 失败重试本来就是算法主路径的一部分；
  • 真正需要警惕的是高竞争下的重试风暴；
  • 这也是为什么 lock-free 不必然优于 lock-based。

七、为什么 Java 工程里通常应优先参考 ConcurrentLinkedQueue

7.1 因为“理解算法”和“手写生产实现”是两件事

对 Java 开发者来说，一个非常重要的现实判断是：学习无锁队列当然值得，但这不等于你应该默认自己手写一个通用 MPMC 队列投入生产。

原因并不神秘：

  • 正确性证明难；
  • 边界条件多；
  • 性能表现强依赖负载形态；
  • 错误往往只在高竞争、极端时序、长时间运行后暴露。

所以实践里最稳妥的默认值，通常是先看标准库是否已经给出成熟实现。

7.2 ConcurrentLinkedQueue 官方文档给了哪些关键信息

ConcurrentLinkedQueue 的 Javadoc 里有几条非常值得直接记住的事实：

  1. 它是一个 unbounded thread-safe queue。
  2. 它基于 Michael & Scott non-blocking queue algorithm。
  3. 其迭代器是 weakly consistent，不是强一致快照。
  4. size() 不是 O(1)，而且在并发修改下可能不精确。
  5. 官方给出了并发容器常见的 memory consistency effects：线程在把元素放入队列前的操作，对另一个线程成功访问或移除该元素后的操作可见。

这些信息的价值在于：它们把“这是个高性能并发队列”重新落回到了规范化边界，而不是停留在口耳相传的印象里。

7.3 为什么它更适合作为工业参照物

把 ConcurrentLinkedQueue 作为工业参照物，并不是因为“标准库永远最快”，而是因为它已经替你承接了最容易出错的那些问题：

  • Michael-Scott 家族算法的结构不变量；
  • 垃圾回收环境下的实现取舍；
  • 弱一致迭代语义；
  • 文档化的行为边界。

对于大多数 Java 场景，真正需要做的不是“从零发明一个更厉害的 MPMC 队列”，而是先回答：你到底需要的是通用 non-blocking FIFO，还是一个更窄、更专用的模型。

7.4 什么时候甚至不该优先选无锁队列

无锁队列不是默认答案。下面这些场景里，锁队列或阻塞队列往往更合适：

  • 你需要明确的阻塞等待与背压，而不是空转轮询；
  • 你需要容量边界；
  • 你更关心可维护性而不是极致低停顿；
  • 你的竞争水平并不高，锁的成本已经足够低；
  • 你需要与条件等待、超时、中断等控制流紧密配合。

换句话说：无锁队列不是“更现代的队列”，它只是更适合某一类成本模型。

八、这篇文章不再重复解释什么

到这里，系列边界必须再次收紧，否则无锁队列文章很容易膨胀成“并发大全”。

  • 如果问题是为什么 volatile 能建立可见性边，回到《JVM 的内存模型与线程》。
  • 如果问题是 CAS 与锁、AQS、synchronized / ReentrantLock 该怎么取舍，回到《线程安全与锁优化》。
  • 如果问题是线程如何 park/unpark、interrupt、join，回到《Java 并发编程笔记》。
  • 如果问题是任务提交、线程池、FutureTask、CompletableFuture 的调度语义，回到《Java 线程池笔记》。

本文只保留一个中心：无锁 FIFO 队列如何在并发竞争中维持结构正确性与整体进展，以及 Java 工程里该如何看待它。

九、最该记住的结论

把全文压缩成几句话，其实就是：

  1. 无锁不是没有同步，而是不用互斥锁来承载进展保证。
  2. 先分清 SPSC / MPSC / SPMC / MPMC，再谈算法。
  3. 通用 lock-free FIFO 队列的经典参照物是 Michael-Scott 队列，而不是随手写的示例代码。
  4. head / tail 可以滞后，helping 是算法的一部分，不是额外补丁。
  5. Java 里学习无锁队列很重要，但生产默认值通常应是 ConcurrentLinkedQueue，而不是自研通用 MPMC。

如果这些结论稳定了，再回头看具体实现细节，你会更容易分辨哪些是算法本质，哪些只是某个实现版本的局部技巧。
