线程安全与锁优化

《JVM 的内存模型与线程》回答的是“多线程程序为什么会观察到奇怪结果”，《Java 并发编程笔记》回答的是“线程怎么等待、唤醒和结束”。而这篇文章要回答的是另一个更工程化的问题：当共享可变状态无法避免时，我们究竟该如何把它管住，以及这些手段在 JVM 里要付出什么代价。

很多并发文章一上来就讲对象头、Mark Word、偏向锁、AQS 源码，但那样读者很容易只记住一堆实现碎片，却没先理解：线程安全到底在保护什么，锁和 CAS 分别在解决什么，为什么有些场景该用互斥、有些场景该用不可变或无锁结构。这篇文章会把这个顺序拉正：先讲策略，再讲工具，最后讲 HotSpot 如何把这些工具优化成不同的锁形态。

本文拥有以下主题的定义权：线程安全分层、不可变与安全发布、synchronized/Lock/CAS 的工程取舍、ObjectMonitor、Mark Word、锁升级与锁优化、AQS 作为显式锁框架的抽象。 JMM、volatile、happens-before 的规范定义已经在《JVM 的内存模型与线程》中给出，本文只在需要时回链，不再重复开课。

一、线程安全到底在保护什么

1.1 线程安全不是“加了锁就安全”

线程安全的本质，不是“某个关键字出现了”，而是：

当多个线程访问同一份共享状态时，程序依然能维持它承诺的正确性。

这个“正确性”可以是：

  • 值不会丢；
  • 不变量不会破；
  • 状态转移不会乱序；
  • 读者不会看到半初始化结果。

所以线程安全真正想保护的，不是线程本身，而是共享可变状态及其不变量。

1.2 线程安全问题的根源是“共享 + 可变”

只共享但不变，通常不是问题；只可变但不共享，也通常不是问题。最危险的是两者叠加：

  • 多个线程都能访问；
  • 且至少有线程会修改；
  • 且这些修改之间存在语义约束。

这也是为什么并发设计的第一原则永远是：能不共享，就别共享；能不修改，就别修改。

1.3 线程安全的几种层次

原文里这部分很有价值，适合保留下来，但需要把它讲得更像一套判断框架：

层次 含义 典型例子
不可变（Immutable） 状态一旦构造完成就不再变化 String、不可变配置对象
绝对线程安全 不管调用方怎么组合操作，都天然安全 很少见，要求极高
相对线程安全 单个操作线程安全，但多步组合仍需调用方协调 Vector 这类同步容器
线程兼容 本身不线程安全，但调用方可通过外部同步保证安全 大多数普通类
线程对立 即使调用方想同步，也难以安全使用 某些全局可变 API、废弃线程控制方法

这张表的现实意义是：不要看到“线程安全”四个字就以为万事大吉。很多库最多只是相对线程安全，并不意味着你可以无脑把多个操作串起来当成原子事务。

二、最优先的策略不是“加锁”，而是“减少共享”

2.1 不可变：最省心的线程安全

如果一个对象一旦构造完就不再改变，那么多个线程同时读取它，天然不会互相踩踏。

不可变策略的优势不是“快”，而是：

  • 推理最简单；
  • 不依赖运行时竞争；
  • 不需要考虑锁顺序、死锁、中断、唤醒。

这也是为什么配置对象、值对象、快照对象尽量做成不可变，会让并发设计一下子轻很多。

2.2 线程封闭：把状态关进线程里

另一种比加锁更好的办法，是根本不让状态跨线程共享。常见方式包括：

  • 局部变量
  • 栈内中间状态
  • ThreadLocal
  • actor / mailbox 风格的单线程拥有状态

从工程上看，这种方法的价值在于：你绕开了“共享可变状态”这个问题本身。

2.3 安全发布：不是不可变，也至少要有正确交接

当对象必须在多个线程之间传递时，光“构造完了”还不够，还要考虑它是否被正确发布。

这正是 JMM 负责定义的部分：

  • 通过锁发布；
  • 通过 volatile 发布；
  • 通过线程启动 / 任务提交 / 并发容器发布；
  • 依赖 final 的初始化安全。

这部分规范边界已经在《JVM 的内存模型与线程》中定义完毕。本文只保留一个结论：对象不是“new 出来”就天然对别的线程可见，而是要通过正确的发布边界交出去。

三、当共享不可避免时，有哪些基本武器

3.1 volatile 不是锁，它解决的是“信号”和“发布边”

虽然 volatile 不属于本文的定义中心，但它必须在这里被重新摆正位置。

它适合做的事是：

  • 传递状态位；
  • 作为一次性发布标志；
  • 守卫一批普通写入的可见性。

它不适合做的事是：

  • 保护 i++ 这类复合操作；
  • 维护多个变量共同构成的不变式；
  • 代替互斥进入临界区。

也就是说，volatile 是“让别人看见”，不是“禁止别人同时改”。

3.2 synchronized：最基础也最可靠的互斥同步

synchronized 是 Java 最基础的互斥手段。它同时提供：

  • 互斥进入临界区；
  • 锁边界上的可见性；
  • 可重入语义；
  • 异常路径上的自动释放。

它的优势在于：

  • 语法直接；
  • JVM 深度优化；
  • 表达“这段代码就是一个临界区”非常自然。

它的局限也很明显：

  • 等锁不可中断；
  • 没有超时获取；
  • 条件队列只有一个 Wait Set；
  • 公平性不可配置。

3.3 ReentrantLock：更强控制力，代价是更高心智负担

和 synchronized 相比，ReentrantLock 的价值不在于“总是更快”，而在于它暴露了更多工程控制面：

  • 可中断获取 lockInterruptibly()；
  • 超时获取 tryLock(timeout)；
  • 公平 / 非公平可选；
  • 多个 Condition 条件队列；
  • 更容易和复杂调度、取消、限时等待组合。

因此它更适合：

  • 需要可取消等待的代码；
  • 一个锁对应多个业务条件；
  • 明确要做公平性或限时控制的场景。

但如果只是简单互斥，synchronized 往往仍是更合适的默认值。

3.4 CAS 与原子类：不阻塞，不代表没有代价

CAS（Compare-And-Set / Compare-And-Swap）是无锁同步的核心基石。它适合解决的，是单点状态的原子更新：

  • 计数器加减；
  • 指针切换；
  • 一次性状态迁移；
  • 无锁链表 / 队列里的关键更新步骤。

但 CAS 的代价也不能被浪漫化：

  • 失败时要重试；
  • 高竞争下可能疯狂烧 CPU；
  • 有 ABA 问题；
  • 只对单点原子更新友好，不天然解决复杂不变式。

因此“无锁”不是“总比锁快”，而是“把等待代价从阻塞切换成了重试和自旋”。

四、synchronized 背后的运行时模型：Mark Word、Lock Record、ObjectMonitor

4.1 先记住一句总规则：对象头里装的是状态入口，不是整套锁系统

理解 synchronized 底层实现时，最容易混乱的地方是把对象头、Mark Word、Monitor、线程队列全搅在一起。更稳妥的理解方式是：

Mark Word 更像锁状态的入口或指针容器，而真正的等待管理结构在更外部的数据结构里。

4.2 Mark Word 到底装了什么

Mark Word 是对象头里最“多态”的一块区域。不同状态下，它承载的信息不同：

  • 无锁时：可能存 hashCode、年龄等运行时信息；
  • 偏向锁时：记录偏向线程信息；
  • 轻量级锁时：指向栈中 Lock Record；
  • 重量级锁时：指向 ObjectMonitor。

这也是为什么它常被形容成一种“像 union 一样复用同一块位空间”的结构。

4.3 轻量级锁时，原始头信息去了哪里

原文里有一个非常值得保留的洞察：当锁进入轻量级路径时，Mark Word 不再直接存原始信息，而是把原始头信息“搬到别处”——也就是线程栈帧里的 Lock Record。

因此可以把三种典型状态理解为：

锁状态 Mark Word 主要内容 原始头信息存放位置
无锁 / 偏向锁 直接存自身状态信息 仍在 Mark Word
轻量级锁 指向 Lock Record 线程栈帧中的 displaced mark word
重量级锁 指向 ObjectMonitor ObjectMonitor 的 _header 等位置

4.4 重量级锁时，真正的锁管理在哪儿

一旦膨胀为重量级锁，真正承担锁管理职责的是 ObjectMonitor。它才是这几个问题的宿主：

  • 谁当前持有锁（owner）？
  • 谁在等待进入（EntryList / cxq）？
  • 谁在 wait() 之后等待条件（WaitSet）？
  • 重入了多少次？

所以一句话概括就是：

重量级锁阶段，Mark Word 只是 monitor 的入口，monitor 才是锁管理本体。

4.5 这和前一篇的管程语义正好接上

前一篇《Java 并发编程笔记》讲的是管程语义：Entry Set、Wait Set、wait/notify、Mesa 模型。这里讲的是实现层：这些抽象在 HotSpot 里如何落到 ObjectMonitor 的字段和队列上。

两篇文章的关系是：

  • 前一篇定义线程怎么等、怎么醒；
  • 本篇解释JVM 把这些等待和唤醒结构放在哪儿、怎么管理。

五、锁优化不是魔法，而是“尽量别太早进阻塞”

5.1 为什么 JVM 要做锁优化

真正昂贵的不是“锁”这两个字，而是：

  • 线程阻塞与唤醒；
  • 用户态 / 内核态切换；
  • 调度和缓存失效；
  • 在高竞争下维护等待结构。

所以 HotSpot 的优化目标并不是取消锁，而是：

在语义不变的前提下，尽量用更便宜的路径完成同样的互斥。

5.2 偏向锁、轻量级锁、重量级锁分别在优化什么

虽然这套叙述带强烈 HotSpot 版本色彩，但作为理解材料仍然有价值。

形态 主要优化目标 适用直觉
偏向锁 单线程反复进入同一锁时避免重复竞争 几乎没有竞争
轻量级锁 用 CAS + 栈记录 + 短暂自旋避免立即阻塞 低竞争、临界区短
重量级锁 明确维护等待队列并阻塞线程 竞争明显或等待较长

这里最该记住的不是“锁一定按某个永恒流程升级”，而是：

  • 这些都是 HotSpot 的实现策略；
  • 不同 JDK 版本会调整甚至移除其中某些路径；
  • 你不该把它们当成 Java 语言语义本身。

5.3 JDK 15+ 以后，对偏向锁要加版本围栏

这是本系列事实核验里最重要的一条之一：

“默认无锁 → 偏向锁 → 轻量级锁 → 重量级锁”的讲法，只能作为特定 HotSpot 版本历史理解，不能当成所有现代 JDK 的无条件事实。

因此本篇保留偏向锁，但只以“历史实现和理解材料”的身份出现。

5.4 自旋为什么既是优化，也是风险

轻量级锁和 AQS 在很多场景下都会先尝试一段短暂自旋，因为：

  • 如果持锁线程很快就要退出；
  • 那么比起把当前线程挂起再唤醒，继续在用户态多试几次更划算。

但自旋的风险也很直接：

  • 竞争高时，所有线程都在空转；
  • CPU 被烧掉，却没人真正推进；
  • 最终还是得回到阻塞。

所以“自旋优于阻塞”只在低竞争 + 短临界区下成立。它从来不是无条件真理。

六、锁消除、锁粗化：JIT 不只是加速业务代码，也会改写同步形态

6.1 锁消除：证明没有共享，就把锁拿掉

如果 JIT 通过逃逸分析能够证明某个对象不会逃出当前线程，那么围绕它的同步就可能被直接消除。

这背后的逻辑非常合理：既然对象根本不共享，那锁的语义价值就消失了，只剩下开销。

所以锁消除不是“JVM 不尊重你的同步”，而是“JVM 证明了这段同步在当前上下文里没有必要”。

6.2 锁粗化：如果总在短距离内反复加解同一把锁，不如合并

和锁消除相反，锁粗化做的是另一件事：

  • 如果一段连续逻辑中，反复对同一对象做很细碎的加锁 / 解锁；
  • JIT 可能会把这些局部锁边界合并成一个更大的锁区域。

其动机是减少反复进入退出同步路径的成本。

6.3 这两种优化都在提醒同一件事

JVM 关心的不是“有没有写 synchronized”，而是“在保持语义正确的前提下，最省成本的同步形态是什么”。

所以观察并发程序性能时，不仅要看源码里的锁，还要知道 JIT 可能已经把它变形了。

七、AQS：显式锁家族的统一抽象

7.1 AQS 最值得记住的不是源码细节，而是它抽象了哪些职责

原文里这部分信息密度很高，但如果不先收束，读者容易掉进源码迷宫。更值得先记住的是：AQS 其实把“一个同步器需要管理什么”抽象得非常清楚。

任何锁 / 同步器大致都要处理这些职责：

  1. 谁拥有资源？
  2. 当前状态是什么？
  3. 谁在等待获取？
  4. 谁在等待条件？
  5. 什么时候该阻塞，什么时候该唤醒？

7.2 AQS 怎么把这些职责落成数据结构

在 AQS 里，这些职责分别映射成：

  • state：同步状态的统一抽象；
  • exclusiveOwnerThread：独占模式下的持有者；
  • 同步队列：等待获取锁或资源的线程节点；
  • ConditionObject 队列：等待条件的线程节点；
  • LockSupport.park/unpark：最终的阻塞与唤醒原语。

这也是为什么可以把 AQS 看成“显式锁世界里的 ObjectMonitor 框架化版本”：

抽象职责 AQS ObjectMonitor
持有者 exclusiveOwnerThread _owner
状态 / 重入计数 state _recursions 等
获取等待队列 同步队列 EntryList / cxq
条件等待队列 ConditionObject WaitSet
阻塞原语 park/unpark 底层 monitor / parker 路径

7.3 ReentrantLock 为什么比 synchronized 更灵活

因为 ReentrantLock 不是语言关键字，而是构建在 AQS 之上的一个可编排同步器，所以它天然更容易暴露更多控制面：

  • 公平 / 非公平；
  • 可中断获取；
  • 超时获取；
  • 多个 Condition；
  • 与其它同步器模式统一建模。

但别忘了，灵活性的另一面就是更重的 API 和更高的误用成本。

7.4 Condition 为什么值得单独提一句

前一篇文章讲过：对象监视器只有一个 Wait Set，所以 wait/notify 只适合比较简单的条件协作。

而 Condition 的工程价值就在于：

  • 一个锁可以派生多个条件队列；
  • 生产者只唤醒消费者，消费者只唤醒生产者；
  • 减少 notifyAll() 式惊群。

这也是 ReentrantLock + Condition 常被用于更复杂阻塞队列和调度器实现的原因。

八、什么时候用锁，什么时候用 CAS，什么时候干脆别共享

8.1 一条非常实用的选型顺序

在工程实践里，可以用下面这个顺序做第一轮判断：

  1. 能否不可变？ 能的话优先不可变。
  2. 能否线程封闭？ 能的话别共享。
  3. 只是单点原子状态更新吗？ 是的话考虑 CAS / 原子类。
  4. 涉及复合不变量或多步事务吗？ 优先互斥锁。
  5. 需要复杂等待、可中断、多个条件队列吗？ 考虑 ReentrantLock / AQS 同步器。

8.2 锁和 CAS 不是敌对关系，而是不同成本模型

可以把两者的区别看成：

  • 锁：冲突时让线程排队 / 阻塞，以空间和调度结构换确定性；
  • CAS：冲突时让线程重试，以 CPU 自旋换避免阻塞。

没有哪一边天然更“高级”。真正要看的只有两件事：

  1. 共享状态到底复杂到什么程度；
  2. 竞争在实际流量下到底高不高。

8.3 最常见的误判

下面这些说法都值得保持警惕：

  1. “用了原子类就不需要再思考线程安全。”
  2. “无锁一定比加锁快。”
  3. “对象头和锁状态图就是 Java 语义本身。”
  4. “看懂了 Mark Word，就等于会设计并发程序。”

实现细节很重要，但如果脱离“我要保护什么不变量”这个问题，它们只会变成高密度噪音。

九、这篇文章与系列其他文章的分工

  • 本文负责：线程安全层次、共享状态治理策略、synchronized / Lock / CAS 的取舍、Mark Word、Lock Record、ObjectMonitor、锁优化、AQS 的抽象地位。

这样分工以后，本文不再重复讲完整 JMM 教程，也不承担线程原语的全部定义，而是专注于“共享状态如何被正确保护，以及这些保护机制在 JVM 里如何被实现和优化”。
