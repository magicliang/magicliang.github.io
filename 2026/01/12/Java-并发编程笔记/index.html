Java 并发编程笔记

如果说《JVM 的内存模型与线程》回答的是“并发为什么会错”，那么这篇文章回答的就是另一个同样基础的问题：线程到底是怎么协作、等待、唤醒和结束的。

很多并发 bug 不是出在内存模型本身，而是出在对线程原语的误解：wait() 什么时候释放锁，notify() 为什么不能当成“立刻切换”，park() 和 wait() 到底有什么差别，interrupt() 为什么常常让人越用越乱，join() 为什么本质上也是一种条件等待。这些问题如果不讲清楚，后面的锁、线程池、结构化并发都会越学越混。

这篇文章只保留线程原语相关的定义权：管程语义、线程生命周期、wait/notify、LockSupport.park/unpark、interrupt、join，以及虚拟线程下这些原语的边界变化。 JMM、锁优化、线程池、结构化并发都会在它们自己的文章里展开，这里只做必要回链。

一、并发与异步：先别把问题混在一起

本文聚焦并发（Concurrency）：多个任务如何在同一时间段内交错或并行执行，以及它们之间如何竞争资源、等待条件、传递结束信号。

**异步（Asynchronous）**讨论的是另一维度：调用方发起操作后，是否必须阻塞等待结果。异步可以建立在单线程事件循环上，也可以建立在多线程并发上。

所以两者的区别可以收束成一句话：

  • 并发关心“多个执行单元如何协调”；
  • 异步关心“调用方是否同步等待”。

后文会提到 Future、join()、中断、取消，但本文不会展开线程池、CompletableFuture 和响应式编排；这些属于《Java 线程池笔记》的范围。

二、为什么线程协作需要管程模型

2.1 光有锁，只解决了一半问题

锁的价值是互斥：同一时刻只让一个线程进入临界区。但现实里的并发协作并不只有“不能同时改”，还有“现在条件还不成立，所以你先等等”。

例如生产者—消费者问题里：

  • 缓冲区满了，生产者不能继续放；
  • 缓冲区空了，消费者不能继续取。

这时候只靠锁是不够的。你还需要一种机制，让线程能够：

  1. 在持有锁时检查条件；
  2. 条件不成立时主动等待；
  3. 条件变化时被别人唤醒；
  4. 醒来后重新竞争锁并继续检查。

这就是管程（monitor）要解决的问题。

2.2 管程的四个组成部分

从抽象上看，一个完整的管程至少有四部分：

  1. 共享数据：被保护的数据结构。
  2. 互斥机制：保证同一时刻只有一个线程执行临界区代码。
  3. 条件变量：让线程在条件不满足时进入等待。
  4. 入口等待区：让还没拿到锁的线程排队竞争进入。

Java 并没有把这些概念用一个 Monitor 类显式暴露出来，但它们都客观存在：

管程概念 Java 中的对应物
共享数据 被保护的对象状态
互斥机制 synchronized 对象监视器
条件等待 wait() / notify() / notifyAll()
入口等待区 monitor 的竞争队列（常被称为 Entry Set）

2.3 Mesa 模型是理解 Java 协作语义的关键

Java 采用的是 Mesa 风格的管程语义，而不是 Hoare 风格。它最重要的结论只有一个：

当线程调用 notify() 或 notifyAll() 时，发通知的线程不会立刻让出锁；被唤醒线程只是从等待区回到竞争区，之后还要重新竞争锁。

这意味着：

  • “被唤醒”不等于“立刻执行”；
  • “收到通知”不等于“条件一定成立”；
  • 唤醒线程重新拿到锁时，条件完全可能已经变了。

这就是为什么 Java 中的 wait() 必须永远和 while 配对，而不是 if。

2.4 别把 Mesa 和 MESI 混在一起

这是并发文章里最常见的术语误伤之一：

  • Mesa 是软件层的管程语义；
  • MESI 是硬件层的缓存一致性协议。

它们都重要，但讨论层次完全不同。Mesa 解决“线程怎么等和怎么醒”，MESI 解决“多核缓存如何保持一致”。

三、Entry Set、Wait Set 与“醒来以后去哪儿”

3.1 两类等待，含义完全不同

线程在管程里最容易混淆的，是这两种等待：

  1. 等待获取锁：线程想进入临界区，但还没拿到监视器。
  2. 等待条件成立：线程已经进入过临界区，但发现条件不满足，于是主动释放锁并等待通知。

它们看起来都叫“等”，但语义不同：

等待类型 典型位置 代表状态
锁竞争等待 Entry Set BLOCKED（synchronized）
条件等待 Wait Set WAITING / TIMED_WAITING

3.2 为什么叫 Entry Set，而不是 Entry Queue

一个非常实用的理解是：Java 不希望你对 monitor 的等待顺序产生 FIFO 幻觉。

叫 Entry Set 的潜台词是：

  • 这里是一群在门口等的人；
  • 但谁先拿到锁，不是你能依赖的公共语义；
  • notify() 唤醒谁、锁释放后谁先抢到，也不能想当然推成“排队最早的一定先走”。

所以用 Set 比用 Queue 更能提醒你：这里存在等待集合，但不要对顺序作出语义承诺。

3.3 一个线程相对于管程只有几种位置

从某个具体 monitor 的视角看，一个线程大体上只会处于这几种位置：

  • Outside：当前和这个 monitor 没关系。
  • Entry Set：想进，但还没拿到锁。
  • Owner：已经持有锁，正在执行临界区。
  • Wait Set：调用 wait() 释放锁，等待条件变化。

但要注意，这只是“相对于某个管程的位置”，并不是 Java 线程生命周期的完整状态机。

四、Java 线程生命周期：六种状态各自代表什么

4.1 Java 线程状态是生命周期视角，不是管程视角

Thread.State 定义了六种状态：

  • NEW
  • RUNNABLE
  • BLOCKED
  • WAITING
  • TIMED_WAITING
  • TERMINATED

它们描述的是线程在生命周期中的状态，而不是“相对于某个 monitor 的位置”。所以两套模型不能一一硬映射，但可以互相帮助理解。

4.2 六种状态的直观含义

状态 含义 常见触发
NEW 线程对象已创建但尚未启动 new Thread(...)
RUNNABLE JVM 可运行，包含 OS 的 ready/running start() 之后的正常执行
BLOCKED 等待进入 synchronized 保护区 monitor 锁竞争
WAITING 无限期等待某个事件 wait()、join()、park()
TIMED_WAITING 带超时时间的等待 sleep()、wait(timeout)、join(timeout)、parkNanos()
TERMINATED 线程执行结束 run() 返回或异常终止

一个很容易踩坑的点是：

RUNNABLE 不等于“此刻正在占用 CPU”。它在 JVM 视角下同时覆盖了 OS 的 ready 和 running。

4.3 BLOCKED、WAITING、TIMED_WAITING 最值得区分

这三个状态最容易在排查问题时被混为一谈，但它们背后的阻塞原因不同：

  • BLOCKED：等锁，尤其是 monitor 锁。
  • WAITING：主动等待一个条件或信号。
  • TIMED_WAITING：主动等待，但带时间上限。

所以看线程 dump 时，先分清“它是在抢锁，还是在等条件，还是在纯睡眠”，后面分析才不会跑偏。

4.4 一个常见但重要的差异：synchronized vs ReentrantLock

同样是“等锁”，两者在线程状态上常表现不同：

  • synchronized 竞争失败，线程通常表现为 BLOCKED；
  • ReentrantLock 及其 AQS 队列通常借助 LockSupport.park() 挂起，线程常表现为 WAITING (parking)。

这也是为什么在线程 dump、JVisualVM 或 jstack 里，你会看到 monitor 和 parking 两套线索。

五、wait/notify：条件等待的经典原语

5.1 wait() 做了两件事，而不是一件

当线程在持有 monitor 的前提下调用 wait()，它会同时做两件事：

  1. 释放当前持有的 monitor；
  2. 把自己放入该 monitor 的等待集合中并挂起。

这和“只是在临界区里暂停一下”完全不同。wait() 的本质是：我先把锁让出来，因为现在条件不成立，我必须允许别的线程进来改变条件。

5.2 notify() 不会立刻让出锁

notify() 和 notifyAll() 最容易被误读成“马上把另一个线程切过来”。但 Java 的 Mesa 语义不是这样的。

更准确的理解是：

  • notify()：从等待集合中挑一个线程，告诉它“你可以回去竞争锁了”；
  • notifyAll()：把所有等待线程都转回竞争路径；
  • 当前持锁线程继续执行，直到它退出同步块或再次 wait()，别人才能真正拿到锁。

5.3 为什么 wait() 必须放在 while 里

这是并发代码里最该形成肌肉记忆的一条规则：

原因至少有两个：

  1. Mesa 语义：你被唤醒后重新抢到锁时，条件可能已经被别人改掉了。
  2. 伪唤醒（spurious wakeup）：线程可能在没有收到你期望通知的情况下返回。

所以“醒了就继续”从来不是 Java 条件等待的正确写法；正确写法一定是“醒了先重查条件”。

5.4 notify() 还是 notifyAll()

在只使用对象监视器的场景里，Java 的 wait() 只有一个 Wait Set，没有按条件拆分成多个独立队列。所以一旦存在多种等待条件混在一个 monitor 上，notify() 很容易唤醒“错的人”。

这也是为什么很多生产代码里更稳妥的做法是：

  • monitor 场景优先 notifyAll()；
  • 如果需要精确条件队列，转向 ReentrantLock + Condition。

六、LockSupport.park/unpark：JUC 的统一阻塞原语

6.1 park/unpark 和 wait/notify 不是一回事

LockSupport 是 JUC 世界里非常核心的一层。很多高级同步器最终都会落到它之上，包括 AQS、FutureTask、Condition 等。

它和 wait/notify 的差异，可以先抓住两个点：

  1. park() 不要求事先持有某个 monitor；
  2. unpark(thread) 可以先于 park() 调用。

6.2 permit 语义是理解 park/unpark 的关键

可以把每个线程想象成带着一个 0/1 的 permit：

  • unpark(thread)：把 permit 设为 1；
  • park()：如果 permit 是 1，就消费掉它并立即返回；如果是 0，就真正挂起。

这个模型带来一个非常重要的性质：

unpark() 先发生，信号不会丢；但 permit 也不会累积，多次 unpark() 仍然只保留一个许可。

6.3 为什么 JUC 喜欢 park/unpark

它最大的好处是更底层、更统一，也更适合框架封装：

  • 不必依赖某个特定对象监视器；
  • 更容易和队列节点、同步器状态一起使用；
  • 能表达“先唤醒后阻塞”的场景；
  • 更适合作为 AQS 这类框架的基础阻塞原语。

6.4 典型上层使用者

组件 底层阻塞方式
AQS 同步队列 park/unpark
ReentrantLock 通过 AQS 间接使用
Condition.await() 最终也会落到 park
FutureTask.awaitDone() 直接 park 等待完成

所以 LockSupport 不只是一个冷门工具类，它是 JUC 世界的阻塞地基。

七、interrupt：协作式中断，而不是强制终止

7.1 interrupt() 的本质是“发信号”

Thread.interrupt() 做的不是“杀死线程”，而是：

  1. 给目标线程设置中断状态；
  2. 如果目标线程正处于某些可中断阻塞点，则尝试把它唤醒。

这是一种协作式取消，不是抢占式终止。

也正因为如此，单独调用 interrupt() 并不保证线程立刻停下。线程是否真正退出，取决于：

  • 它是否在合适的位置检查中断；
  • 它所在的阻塞方法如何响应中断；
  • 它是否在 catch 中正确传播或恢复中断状态。

7.2 三个常用 API 的语义必须分清

方法 作用 是否清除中断标志
interrupt() 给目标线程发送中断信号 否
isInterrupted() 查询某线程是否已中断 否
Thread.interrupted() 查询当前线程是否已中断 是

最危险的地方在于 Thread.interrupted() 会顺手把标志清掉。它不是纯查询，而是“查询并消费”。

7.3 不同阻塞方式对中断的响应并不一致

这是 Java 中断机制最让人困惑、也最值得专门记住的一点：

阻塞方式 收到中断后的典型表现
wait() / sleep() / join() 抛 InterruptedException，并清除中断标志
LockSupport.park() 直接返回，不抛异常，通常保留中断标志
等待进入 synchronized 通常无法立即响应，只是把中断标志设上

所以不要把“中断”理解成一套行为统一的机制。它是一种统一的信号入口，但不同阻塞原语对它的兑现方式并不统一。

7.4 为什么说中断常常让人越用越乱

interrupt 最大的问题不是它没用，而是它的作用粒度和语义边界都比较粗：

  • 它中断的是线程，不是任务；
  • 中断标志是线程级共享状态；
  • 不同阻塞原语的表现不一致；
  • 第三方库可能吞掉 InterruptedException；
  • 子流程可能误用 Thread.interrupted() 把外层的中断信号清掉。

这也是为什么在线程池、嵌套任务和取消传播场景里，单靠中断常常不够，需要额外的任务级取消协议。结构化并发正是试图从更高层解决这些问题，但那是《Java 结构化并发》一文的主题。

7.5 中断的三条实践规则

规则一：永远不要吞掉 InterruptedException

如果你捕获了 InterruptedException 却什么也不做，中断信号就相当于被你偷偷吃掉了。

规则二：默认优先用 isInterrupted()，谨慎使用 Thread.interrupted()

除非你明确需要“消费掉一次中断”，否则不要随手用 Thread.interrupted()。

规则三：区分“取消条件”和“唤醒机制”

很多时候更稳妥的做法是：

  • 用显式任务状态表达“我想取消”；
  • 用 interrupt() 去唤醒阻塞线程；
  • 在线程醒来后，再根据任务状态决定是否真正退出。

这样语义会比“把所有含义都压在一个线程级中断标志上”清楚得多。

八、join()：线程终止等待，本质上也是条件等待

8.1 join() 并不是魔法 API

很多人把 join() 当成一个特殊黑盒，但从本质上看，它做的事情非常朴素：

当前线程在目标线程对象上等待，直到目标线程终止。

也就是说，join() 其实是把“线程是否已经结束”当作一个条件，然后在这个条件上做等待。

8.2 join() 的协作式逻辑

你可以把它理解成这样一段模式化逻辑：

这里有三个角色：

  1. 锁对象：目标线程对象本身；
  2. 等待线程：调用 join() 的那个线程；
  3. 条件：目标线程是否仍然 isAlive()。

它的妙处在于：join() 并没有发明一套新的等待原语，而是建立在 monitor 的等待语义之上。

8.3 join() 为什么也要配合循环理解

因为它本质上仍然是“条件等待”，所以和 wait() 一样，必须按“醒来后重新检查条件”来理解，而不是“被唤醒就说明线程必然结束”。

这也是为什么 join() 在底层同样是协作式等待，而不是某种抢占式暂停。

九、虚拟线程来了以后，哪些地方的边界变了

9.1 虚拟线程没有让旧原语失效，但改变了它们的成本模型

JDK 21 之后，虚拟线程让“每任务一线程”重新变得现实。但这并不意味着所有老的线程原语都能原样照搬，只是它们的性能成本和适用边界发生了变化。

9.2 synchronized 内阻塞时的 pinned 问题

虚拟线程最值得警惕的一个点是：如果它在某些 monitor 相关阻塞路径上被挂住，可能会把底层载体线程也一起占住，出现 pinned 问题。

一个足够实用的结论是：

  • 短小、纯互斥的临界区：synchronized 仍然可以用；
  • 临界区内还要做等待、I/O 或长阻塞：要格外小心 pinned 风险；
  • 需要显式条件等待：更适合转向 ReentrantLock + Condition 一类更可控的原语。

9.3 这不意味着 monitor 失效了

虚拟线程时代，wait/notify、join、中断、park/unpark` 这些原语并没有失效。真正变化的是：

  • 你更需要区分“互斥”和“阻塞等待”各自的成本；
  • 你更需要避免在 monitor 持有期间做长时间阻塞；
  • 你更应该把复杂取消和任务组织逻辑交给结构化并发，而不是继续堆在线程级 API 上。
