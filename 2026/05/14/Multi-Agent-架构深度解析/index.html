Multi-Agent 架构深度研究：从四种基础模式到「何时不该用多 Agent」的工程判断

Created2026-05-14|Updated2026-05-14|技术
|Word Count:7.5k|Reading Time:24mins|Post Views:

2026 年上半年，Multi-Agent 成为 AI 工程领域最热的架构关键词。Google 在 ADK 里把四种编排原语写进了框架层；Anthropic 在《Building Effective Agents》里用一整篇工程博客论证「多数情况下你不需要 Multi-Agent」；OpenAI 先发布了 Swarm 做教育性演示，随后把核心思想收进了 Agents SDK 的 Handoff 机制；Microsoft Research 的 Magentic-One 用双循环编排在 GAIA benchmark 上取得了不依赖特定模型的通用成绩。

表面上这些方案都在回答同一个问题——「怎么让多个 Agent 协作」。但把它们的设计文档摊开对比，会发现它们回答的其实是四个完全不同层次的问题，选错层次比选错框架造成的工程伤害大得多。

本文按五个切面展开：能力光谱的四层定位、Google 四种编排原语的工程语义、六个主流框架的实际架构选择（含它们不会写进 README 的 tradeoff）、Anthropic「更强模型优于更多 Agent」的底层逻辑、以及一个比「选哪个框架」更前置的判断——什么任务根本不该引入 Multi-Agent。

能力光谱：不是所有「多个 LLM 调用」都是 Multi-Agent

社区讨论中最常见的混淆是把「多个 LLM 调用」等同于「Multi-Agent」。一个 prompt chain 里有三次 LLM 调用，不意味着它是三个 Agent 在协作。在讨论架构模式之前，必须先锁定术语的准确层级。

四层能力定位

层级 形态 核心特征 谁控制执行流 典型代表
L1 Single Agent 一个 LLM 实例 + 工具调用循环 单一模型自主 Claude Code 单会话、Cursor Agent
L2 Agent + Skills 同一 Agent 通过外部能力（MCP tools、RAG、memory）扩展 单一模型自主 + 工具 Claude Code + MCP Server、Hermes skill 系统
L3 Multi-Agent Workflow 多个独立 Agent，编排逻辑由代码预定义 开发者编写的代码 CrewAI Sequential、LangGraph 图
L4 Autonomous Agent Teams 多个 Agent，编排逻辑由 Agent 自身动态决定 LLM 动态决策 Magentic-One、MetaGPT

这四层之间的核心区别不在于 LLM 调用的数量，而在于谁持有执行流的控制权。L1-L2 始终是单个模型在做决策；L3 的多 Agent 按预定义的代码路径协作（Anthropic 称之为 Workflow）；L4 才是真正意义上的 Agent Teams——Agent 自己决定子任务怎么分、谁来做、什么时候重新规划。

Anthropic 的分类法在这里做了一个重要区分：Workflow ≠ Agent。前者是「LLM 和工具通过预定义代码路径编排」，后者是「LLM 动态指挥自身过程和工具使用」。很多号称 Multi-Agent 的系统，按这个标准其实只是 Multi-LLM Workflow——没有任何一个参与者拥有自主决策权。

[PATTERN] 判断一个系统是 Workflow 还是 Agent：问一个问题——「如果删掉编排代码，各参与者能不能自己协调完成任务？」如果答案是否定的，它就是 Workflow，不管用了多少个 LLM 实例。Workflow 不是贬义词——Anthropic 的核心主张恰恰是绝大多数生产系统应该停在 Workflow 层。

为什么这个分层很重要

因为每向上一层，付出的代价都在指数级增长：

  • L1→L2：增加工具集成和状态管理的复杂性，但模型仍是单一决策点，出错时排障路径清晰
  • L2→L3：引入 Agent 间通信协议设计、状态同步、失败传播问题；好处是开发者仍控制编排逻辑
  • L3→L4：把编排权交给模型本身，获得最大灵活性，同时失去可预测性——「Agent 为什么做了这个决定」变成需要 trace 才能回答的问题

一个反复出现的工程错误是在 L2 能解决的问题上引入 L4 的复杂性。Claude Code 的实际架构就是一个反证：它是 L2（单 Agent + 强工具集），而非 L3 或 L4，但它在 SWE-bench 上的表现说明精心设计的 L2 系统可以匹配甚至超过草率搭建的 L4 系统。

Google ADK 的四种编排原语

Google Agent Development Kit 做了一件对行业有价值的事：把 Multi-Agent 编排的常见模式抽象为四个可组合的原子类型。所有更复杂的系统都可以视为这四种原语的嵌套组合。

SequentialAgent：流水线

子 Agent 按固定顺序逐一执行。前一个 Agent 的输出通过 Session State 的 output_key 机制传递给下一个。这是最简单、最可预测、也最容易调试的模式。

工程语义：当且仅当任务可以分解为有严格先后依赖的阶段，且每个阶段的输入输出契约可以预先定义时使用。

隐含假设：每一步都会成功；如果第二步失败了，没有内置的回退到第一步的机制——需要外层套 LoopAgent 来处理。

ParallelAgent：扇出-聚合

多个子 Agent 同时启动，各自独立处理互不依赖的子任务。所有子 Agent 共享同一个 Session State，但各自运行在独立的上下文分支中。

工程语义：子任务之间严格无依赖，且最终需要一个聚合步骤。典型场景是并行搜索多个数据源后由汇总 Agent 综合结果。

隐含假设：子任务真的独立。如果任务 A 和任务 B 之间有隐含的信息依赖（比如都在修改同一个文件），ParallelAgent 不会帮你发现这个冲突——它会让两个 Agent 各自写出一版修改，最后覆盖的那个赢。

LoopAgent：迭代直到收敛

子 Agent 序列被反复执行，直到满足三种终止条件之一：max_iterations 达到上限、某个子 Agent 发出 escalate=True 信号、或外部条件判定收敛。

工程语义：需要渐进式改善的生成任务——代码生成→测试→修复、写作→评审→修订、优化→评估→再优化。

隐含假设：每次迭代都会朝正确方向收敛。实际中一个常见的失败模式是 LoopAgent 进入「振荡」——Agent A 修改一处，Reviewer B 说应该改回去，下一轮 A 又改过来，循环不收敛。好的 LoopAgent 设计需要带 convergence detection（检测输出是否不再变化）或递减的修改权限。

LlmAgent 层级委派：动态路由

中央 LlmAgent 根据输入内容动态决定将子任务委派给哪个专家 Agent。两种委派机制：

  • transfer_to_agent()：控制权完全转移，类似 OpenAI Swarm 的 Handoff
  • AgentTool：把子 Agent 包装成工具同步调用，结果返回给父 Agent 继续处理

工程语义：输入类型不确定、需要运行时路由的场景。本质上是一个 LLM 驱动的 Router，比硬编码的 if-else 路由更灵活，但也更不可预测。

隐含假设：LLM 的路由判断是准确的。如果 Orchestrator 误判任务类型把请求送到了错误的子 Agent，整条执行链都会偏移。Anthropic 的建议是在 Orchestrator prompt 中提供极其清晰的子 Agent 能力描述，让路由决策基于充分信息。

组合是真实世界的常态

实际系统几乎不会只用一种原语。一个典型的生产级设计：

[PATTERN] 选择编排原语的决策树：先问「子任务之间有没有依赖？」——有 → Sequential；没有 → Parallel。再问「需不需要迭代改善？」——需要 → 外面套 Loop。最后问「下一步谁执行是固定的还是动态的？」——动态 → 加 LlmAgent 路由。不需要从头设计编排模式，只需要按这三个问题组合原语。

六个框架的真实架构选择

Anthropic：反 Multi-Agent 的 Multi-Agent 方法论

Anthropic《Building Effective Agents》（2024-12-19）是 2025-2026 年 agent 工程领域引用率最高的一篇工程博客。它的核心主张反直觉：在一篇讨论 Agent 的文章里，花了大部分篇幅论证你可能不需要 Agent。

Anthropic 定义了五种 Workflow 模式（按复杂度递增）：

  1. Prompt Chaining：顺序 LLM 调用，中间有程序化校验门
  2. Routing：分类输入 → 路由到专门处理器
  3. Parallelization：并行切片（Sectioning）或多次执行投票（Voting）
  4. Orchestrator-Workers：中央 LLM 动态分配子任务
  5. Evaluator-Optimizer：生成-评估循环

这五种全部归类为「Workflow」而非「Agent」。真正的 Agent——模型自主控制迭代和终止——被 Anthropic 限制在「开放式问题，无法预测所需步骤数」的场景中。

Anthropic 不说的但做的：Claude Code 本身就是一个极端精调的 L2 Single Agent。它没有用多 Agent 架构来解 SWE-bench，而是用一个模型 + 深度优化的工具集 + 精心设计的 system prompt 来解。Anthropic 工程团队的原话是「我们在工具设计上花的时间比在整体 prompt 上多」——这是 Skill 路线的工程实证：投资于 Agent-Computer Interface（ACI）的精度，比投资于 Agent 间编排的复杂度回报更高。

OpenAI Swarm：去中心化的极限

Swarm 是 OpenAI 2024 年发布的实验性教育框架（后续核心思想被收进 Agents SDK），它的设计哲学走到了 Anthropic 的对面——不是让一个强大的编排者管理所有 Agent，而是让每个 Agent 自己决定什么时候把控制权交给谁。

全部抽象只有两个：

  • Agent：instructions（系统提示）+ functions（可用工具）+ model
  • Handoff：function 返回另一个 Agent 实例 = 控制权转移

没有中央编排者。没有全局状态管理。没有消息广播。每个 Agent 只看到自己的上下文和从前一个 Agent 传来的消息历史。这种设计的 Token 效率极高——每次只有当前 Active Agent 在消费 API 调用。

Swarm 适用的精确场景：大量独立能力域，每个域的 instruction 和工具集足够不同，以至于塞进单个 prompt 会导致模型选工具选错。典型例子是客服系统：退款、技术支持、账单查询、投诉处理各走不同专家路径。

Swarm 不说的限制：

  • 没有全局进度跟踪——如果一个任务需要多个 Agent 协作完成一个目标，没有人监控「整件事做完了没有」
  • Handoff 是单向的——Agent A 把控制权给 Agent B 之后，B 没有内置机制回到 A 汇报结果（需要自己设计 handoff 回路）
  • 完全依赖模型的 tool calling 能力——如果模型在 function call 上不可靠，整个路由系统就不可靠

Microsoft Magentic-One：工业级层级编排

Magentic-One 是 Multi-Agent 工程化的另一个极端——它不追求极简，而是追求对长程复杂任务的鲁棒性。核心创新是 Orchestrator 的双循环机制：

外循环（Task Ledger）：

  • 维护全局认知：已确认的事实、当前假设、当前计划
  • 当内循环报告停滞时触发重新规划——不是从头来过，而是在已有认知基础上调整

内循环（Progress Ledger）：

  • 逐步分配子任务给专家 Agent
  • 每步执行后自省：完成了？在推进？还是卡住了？
  • 停滞计数超过 2 → 上升到外循环

这种设计回答了一个其他框架回避的问题：Agent 卡住了怎么办？ 大多数 Multi-Agent 框架对「Agent 做了几步但没进展」这种情况没有内置检测机制——LoopAgent 会继续 loop，SequentialAgent 会卡在当前步。Magentic-One 把「进展检测」作为一等公民引入编排层。

MetaGPT：用 SOP 对抗幻觉级联

MetaGPT 的出发点是一个被充分观察到但少有框架系统解决的问题：朴素地串联 LLM 会导致幻觉级联放大。Agent A 的小偏差传给 Agent B，B 在偏差基础上继续生成，到 Agent C 时偏差已经被放大到不可用。

MetaGPT 的解法是把人类组织的 SOP（标准操作流程）编码进 Agent 间的通信协议：

  • Agent 不做 free-form 对话，而是按 SOP 规定的格式生成结构化输出
  • 下游 Agent 不是「阅读上游的自然语言然后自由发挥」，而是「接收符合 schema 的上游输出并在约束内工作」
  • 每个阶段有明确的验证点——类似代码 review 中的 type check，在信息传递环节做格式和语义校验

这种设计在软件开发场景（MetaGPT 的主场景）中特别有效：Product Manager 输出的是 PRD 格式文档，Architect 输出的是系统设计文档，Engineer 输出的是代码——每一层的输出格式都有明确规范，偏差可以在传递前被检测到。

MetaGPT 不说的限制：SOP 本身是人类设计的，如果 SOP 设计有缺陷（步骤遗漏、约束过松），系统不会自己发现。SOP 驱动的系统灵活性也比纯 LLM 编排的系统低——遇到 SOP 没覆盖的情况时，系统倾向于硬套最近的 SOP 步骤而非灵活应对。

LangGraph：显式状态机

LangGraph 把 Multi-Agent 系统建模为可编程的状态机。这个决策背后有一个深思熟虑的设计哲学：不隐藏编排逻辑。

三种拓扑：

  1. Shared Scratchpad：所有 Agent 共享完整消息历史。信息充分但冗余——Agent C 不一定需要看到 Agent A 和 Agent B 之间的全部讨论。
  2. Supervisor + 独立记事板：各 Agent 拥有自己的记事板，只有最终结论上报全局。信息隔离减少噪音但增加了 Supervisor 的负担。
  3. 层级子图：Supervisor 管理的不是单个 Agent 而是子图（本身也是 Multi-Agent 系统）。适用于真正大型的系统。

LangGraph 的一句概括精准定义了它和 AutoGen 的区别：Multi-Agent 系统本质上是状态机——「独立 Agent 节点是状态，它们的连接方式是转移矩阵」。AutoGen 把重心放在「Agent 之间怎么对话」，LangGraph 把重心放在「控制权什么时候、在什么条件下从一个 Agent 转到另一个 Agent」。

AutoGen/AG2：对话协议的四种选择

AutoGen 提供了四种 GroupChat 模式，每种对应一种「下一个谁说话」的决策机制：

模式 决策机制 适用场景
RoundRobin 固定顺序轮询 每个 Agent 都必须参与、无需跳过
Selector LLM 根据上下文选人 需要动态路由但保持中央控制
Swarm 当前 Agent 通过 HandoffMessage 指定下一个 去中心化，各 Agent 自决
GraphFlow 图结构定义的转移规则 需要确定性控制流

AutoGen 的独特价值在于它是唯一一个把这四种模式放在同一个框架里让你按需切换的。这反映了一个实践洞察：真实项目在不同阶段可能需要不同的协调模式——原型阶段用 RoundRobin 快速迭代，生产阶段切到 GraphFlow 确保确定性。

[PATTERN] 比较 Multi-Agent 框架的正确维度：不比功能列表，比三件事——第一，它对「下一步谁执行」这个问题的默认回答是什么（中央选择 vs 自决 vs 固定顺序）；第二，Agent 之间怎么共享信息（全量广播 vs 按需传递 vs 完全隔离）；第三，系统卡住时的恢复机制是什么（replan vs retry vs escalate vs 没有）。这三个问题决定了框架的工程边界。

通信带宽：被低估的核心约束

Brooks 定律的 Token 版

Frederick Brooks 1975 年的洞察——「向延期项目增加人手只会使其更加延期」——背后的数学是沟通路径按 n(n-1)/2 增长。这条规律在 LLM Multi-Agent 系统中以更极端的形式呈现，因为 Agent 间每次通信都有 Token 成本。

一个 4 Agent 系统的沟通路径有 6 条。如果采用全量广播模式（所有 Agent 都能看到所有消息），每轮对话中每个 Agent 都要处理其他 3 个 Agent 的输出——context window 膨胀速度是 O(n × messages)。

实际中这意味着：

  • 5 个 Agent 全量广播 10 轮：每个 Agent 的 context 里堆积了约 40 条他人消息。如果平均每条 500 token，额外的 context 开销就是 20,000 token/Agent——还没算上 system prompt 和工具定义
  • 成本线性放大：以 Claude Sonnet 的输入价格计算，这 20k token 的 context 每次推理都要付费。10 轮 × 5 Agent × 20k token = 1M token 的额外通信开销

各框架的通信效率分级

框架 通信模式 Token 效率 代价
OpenAI Swarm 仅 Handoff 时传递 最高 无全局视图，复杂协作难
LangGraph Supervisor 独立记事板 + 摘要上报 高 Supervisor 可能漏信息
MetaGPT SOP 结构化传递 中高 灵活性受限
Magentic-One 分层（内循环局部，外循环全局） 中 架构复杂度高
AutoGen Selector 全量广播 + 智能选人 低 所有人看到所有消息
MoA 层间全连接 最低 设计上就是用 Token 换质量

降低通信成本的工程手段

  1. 信息隔离：Agent 只看到与自己相关的消息（LangGraph Supervisor 模式）
  2. 摘要传递：不传原始输出，传 LLM 压缩后的摘要（额外一次 API 调用换更短的 context）
  3. 结构化协议：约定固定格式减少冗余信息（MetaGPT 的 SOP）
  4. Handoff 而非广播：控制权转移时只传必要上下文（Swarm 模式）
  5. 输出键值存储：Agent 将结果写入共享 State 的特定 key，下游 Agent 只读需要的 key（Google ADK 的 output_key）

[PATTERN] 估算 Multi-Agent 系统的通信税：系统中 N 个 Agent、平均每轮通信 M 条消息、每条平均 T 个 token。全量广播模式的通信税 = N × (N-1) × M × T × 每轮数。如果通信税超过了有效工作 token（Agent 自身推理 + 工具调用）的 30%，说明架构选择有问题——要么减少 Agent 数量，要么切换到隔离式通信。

「更强的模型」vs「更多的 Agent」

这是 2025-2026 年 Multi-Agent 领域最核心的一场思想分歧。Anthropic 代表的立场是：模型能力提升的收益大于架构复杂度增加的收益。

Anthropic 的论证链

Anthropic 的推理路径是这样的：

  1. 起点：一个 Agent 的有效能力 = 模型本身能力 × 工具质量 × 上下文质量
  2. 观察：Multi-Agent 通过角色分离来绕过单模型的能力天花板，但引入了通信开销和协调失败风险
  3. 关键假设：模型能力在快速提升——今年需要三个 Agent 协作的任务，明年一个更强的模型可能直接搞定
  4. 推论：在模型快速迭代的时期，投资于「让单模型用好工具」（Skill Augmentation）比投资于「多模型间编排」（Role Decomposition）的半衰期更长

这不是空谈。Claude Code 在 SWE-bench 上的表现是实证——一个 L2 系统（Single Agent + 工具集）通过精调 ACI（Agent-Computer Interface）就达到了很多 L3/L4 系统追求的性能。Anthropic 工程团队在文章中明确说：「我们花在优化工具设计上的时间比花在 prompt 上的多。」

Skill Augmentation vs Role Decomposition

这两种路径的本质区别：

Skill Augmentation（技能增强）：保持单一 Agent，通过增加工具、知识、Memory 来扩展能力边界。Agent 始终是一个决策点，context 天然完整，不存在通信问题。瓶颈是模型的 context window 和 tool selection 能力。

Role Decomposition（角色分解）：将任务拆成多个角色，每个角色由独立 Agent 承担。每个 Agent 处理更简单的子问题，但引入了「怎么让它们协调」这个新问题。

Anthropic 的隐含判断是：context window 和 tool selection 能力的提升速度，比人类解决 Agent 协调问题的速度更快。换言之——工具越来越好用这件事会自动发生（模型升级带来），但 Agent 间协调的难度不会因为模型升级而自动降低。

何时这个论证会失效

Anthropic 的立场并非绝对正确。它在以下条件下失效：

  1. 模型能力停滞：如果模型迭代速度放缓，Skill Augmentation 路线就遇到天花板，Multi-Agent 的相对价值重新上升
  2. 任务确实不可单体化：某些任务天然需要对抗性观点（辩论、Red Teaming），单一模型无法同时持有互斥的立场
  3. 专业化要求极端：当每个子领域的 tool set 和 instruction 庞大到无法装进单一 context 时，分离为多 Agent 是物理必要的

[PATTERN] 判断是否需要从 Single Agent 升级到 Multi-Agent：问三个问题。第一，「把所有工具定义 + system prompt 塞进一个 context 后，模型还能可靠地选对工具吗？」——如果不行，需要分离。第二，「任务是否需要互斥的观点同时存在？」——如果是，需要多 Agent 辩论。第三，「单 Agent 的错误率是否超过 45%？」——如果单体成功率已经低于 45%，增加 Agent 数量能有效提升系统整体成功率；反过来，如果单体成功率高于 45%，增加 Agent 数量的边际收益急剧下降。

45% 法则：Multi-Agent 的收益边界

这是一个对架构决策有直接指导意义的经验规律：当单 Agent 对某类任务的成功率超过 45% 时，通过增加 Agent 数量来提升系统整体成功率的收益会快速递减。

直觉理解

假设一个任务的判定标准是「最终输出正确」。当单 Agent 成功率为 30% 时，三个 Agent 并行执行再投票（majority voting），系统成功率约为 44%（组合概率计算）。收益显著——增加了 14 个百分点。

但当单 Agent 成功率为 60% 时，同样的三 Agent 投票方案，系统成功率约为 65%。增加了仅 5 个百分点——而成本增加了 3 倍。

这背后的数学是组合概率的非线性特征：当单体已经足够强时，多数投票能翻转错误的概率空间变小了。45% 是一个经验阈值——在此之下，增加 Agent 数量有显著系统级收益；在此之上，同样的工程投入用在提升单 Agent 能力（更好的 prompt、更好的工具、更好的 RAG）上 ROI 更高。

对架构决策的影响

这条法则直接挑战了「问题复杂就上 Multi-Agent」的默认思维：

  • 如果你的单 Agent 在目标任务上已经有 50% 以上的成功率，先别加 Agent。把精力花在分析那 50% 失败案例的模式上——是工具不够用？是 context 不够？是 prompt 有歧义？——然后针对性修复
  • 如果单 Agent 成功率在 20-40% 的区间，Multi-Agent 架构（并行 + 投票或分工 + 验证）能带来系统级提升，值得投入
  • 如果单 Agent 成功率低于 20%，问题可能不在架构上——模型可能根本不具备完成此类任务的基础能力，Multi-Agent 也救不了

[PATTERN] Multi-Agent 投入的 ROI 评估：先在目标任务上跑 100 次单 Agent baseline，统计成功率 P。如果 P > 45%，投资方向应该是优化单 Agent；如果 P ∈ [20%, 45%]，Multi-Agent 架构有 ROI 空间；如果 P < 20%，先等更强模型或换任务定义。这三个区间决定了你的架构投资应该押在哪里。

任务类型决定最佳架构

没有普适的「最优」Multi-Agent 架构。任务的内在特征决定了应该使用哪种模式——选错模式比选错框架的伤害更大。

决策流程

被过度营销的部分：五个常见误区

误区一：「Multi-Agent 一定比 Single Agent 强」

这是最广泛的误解。大量 benchmark 表明，精心设计的 Single Agent 系统在中等复杂度任务上可以匹配甚至超越草率搭建的 Multi-Agent 系统。额外的 Agent 带来的不只是额外的能力，还有额外的通信成本、协调失败风险、以及调试复杂度。

Andrew Ng 展示的 GPT-3.5 在 agentic workflow 下达到 95.1% HumanEval 得分（vs GPT-4 zero-shot 的 67.0%）说明的不是 Multi-Agent 的优越性，而是迭代式架构的优越性——那个 workflow 本质上是 Single Agent + Loop（反思 + 重试），不是多个 Agent 在对话。

误区二：「框架 = 架构」

选了 CrewAI 不意味着你的架构就是好的。框架提供的是编排原语——Sequential、Hierarchical——但架构决策（用几个 Agent、怎么分角色、Agent 之间传什么信息、失败了怎么恢复）仍然是人做的。一个用 CrewAI 搭建的烂架构，不会因为 CrewAI 受欢迎就变好。

误区三：「Agent 越多越好」

Mixture-of-Agents 论文报告了「性能随 Agent 数量提升」的结论，但这有严格的前提条件：Agent 之间的分歧必须是信息性的（informative disagreement），而非噪声性的。如果 5 个 Agent 中有 3 个因为相同的 prompt 偏差犯了同一个错误，投票只会放大错误而非修正它。45% 法则从另一个角度约束了这一点。

误区四：「通信问题可以靠更大 Context Window 解决」

Context Window 的增大确实缓解了信息传递的截断问题，但不解决根本矛盾——Agent 看到了所有信息不等于它能正确利用所有信息。「Lost in the Middle」现象（模型倾向于忽略 context 中间部分的内容）在长 context 中依然存在。信息过载的问题比信息不足更难诊断。

误区五：「先搭架构后填能力」

正确的顺序是反过来的——先确认单 Agent 的能力边界在哪里（具体在哪类任务上失败、失败模式是什么），再决定是否需要 Multi-Agent 以及需要什么模式的 Multi-Agent。跳过单 Agent 基线直接搭多 Agent 系统，会导致你分不清性能问题来自模型能力不足还是架构设计缺陷——前者加 Agent 也没用，后者可能只需要修 prompt。

[PATTERN] 识别 Multi-Agent 营销中的可疑承诺：如果一个 Multi-Agent 框架的宣传里只谈「多少个 Agent 协作完成了什么」，不谈「和精心设计的 Single Agent 基线比提升了多少」，那它大概率没做过这个对比——因为对比结果可能不好看。真正有说服力的 Multi-Agent 展示必须包含 ablation：去掉一个 Agent 性能掉多少？只留一个 Agent 行不行？这些问题的答案决定了架构是必要的还是虚胖的。
