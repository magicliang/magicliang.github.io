七个 skill，一条端到端的投放流水线。读完它们最有意思的不是"agent 能投广告了"，而是这一套文档把"如何不让 LLM 在生产系统里自作聪明"写得有多明白。

这不是 agent 在写广告文案

把"agent + 广告"放在一起，多数人脑子里冒出来的第一个画面是 LLM 帮忙拟标题、改文案、生成创意图。tencentads-* 这一组 skill 不做这件事——它做的是把 agent 接到腾讯广告营销 API（api.e.qq.com）后面：从查广告主账号、配 API Key，一路到创建广告组、绑创意、改预算、看报表。换句话说，这一组 skill 让 agent 操作的是一个真实的、有钱进出的投放系统，不是一个生成内容的玩具。

这个区别决定了整套 skill 的设计取向。生成内容的工具可以容忍 LLM 的"自由发挥"，多写一句、改一个比喻，最差也只是文风问题。投放系统不行：marketing_goal 选错一个枚举，adgroups/add 直接报错；bid_amount 单位搞混，钱花得不是你预想的那个数量级；component_id 写成 0 占位，整条创意进不去。所以读这一组 SKILL.md，最值得看的不是它们各自能调哪些接口，而是它们怎么把 LLM 的决策面收窄到不会出事。

七个 skill 的角色摊开

先把对照系建起来。这一组目前有七个 skill，按职责可以分四类：

类别 skill 一句话职责
凭据 tencentads-auth 保存 / 检查 API Key，所有 tencentads-* 共用
寻址 tencentads-advertiser 列出当前 user 下有权限的广告主账号，按公司名模糊匹配
写入（建组） tencentads-delivery-smart-create 创建智投（AIM+）广告组
写入（建组） tencentads-delivery-standard-create 创建常规（标准）广告组
写入（创意） tencentads-creatives 动态创意 + 创意形式 + 素材上传
写入（修改） tencentads-delivery-standard-update 广告 / 智投项目通用更新（含批量）
读出 tencentads-management 跨账户报表、广告详情、创意详情查询

几条值得提前点出来的差别。

smart-create 和 standard-create 在 description 层就分流了。 两者步骤极其相似（都是"四元组 → 推广产品 → 版位 → 转化目标 → 定向 → 组装请求体"的 7 步流），但 standard-create 的 description 末尾写明：如果当用户明确提到智投、项目、AIM+、艾米等关键词，应使用 tencentads-delivery-smart-create 技能。这是把意图歧义消解放在 skill 选择这一层，而不是丢给单个 skill 内部去判断。原因不复杂：智投和常规的请求体差异大到不值得在一个 skill 里 if/else——智投强约束 smart_delivery_platform 必传、出价方式只有 oCPM、版位强制自动；常规则 CPC/CPM/oCPM/oCPC 全支持、版位手动可选、定向能力完整。

update 是单数职责，且支持智投项目。 description 里特意写明"智投项目 ID 等同于 adgroup_id，参数构造方式完全一致"——也就是说创建分两个 skill，更新合在一起。这个不对称是有道理的：创建时要管的"决策树"不一样，更新时改的字段（出价、预算、状态、定向、时段、深度转化、一键起量、创意增强）几乎是平的。

没有"删除"独立 skill。 状态变更（暂停 / 启用）走 update 的 configured_status: AD_STATUS_SUSPEND / AD_STATUS_NORMAL。这是符合广告系统真实运营习惯的——投放领域没有"硬删"的概念，只有"暂停"和"归档"。

没有"搜索创意（search_dynamic_creatives）"。 tencentads-creatives 的 description 里专门写了"排除搜索创意"。把它留给 management 那边的查询脚本去做，避免在创意管理 skill 里多塞一条职责。

顺一遍端到端流程

把这一组 skill 当成一条流水线来走：从一个新装的环境到投出去一条带创意的广告，再到改预算、看效果，七个 skill 分工很清楚。

先看一下流水线的"管道型 prompt"结构

在按步骤展开之前，有一件事值得先点出来——这一组 skill 的"七步"不是普通的"按顺序调七个接口"，而是一种很特定的结构。每一步内部都是一个**“先查事实 → 在事实上做选择 → 把选择喂给下一步”**的三段式：

把七步串起来看，七个步骤之间的"上下文传递"是有具体规律的——前一步的某几个返回字段会被后一步的入参原样消费，agent 不参与改写。举几个对应关系：

上一步返回字段 被下一步消费成什么
get-account-list 的 account_id 后续所有 skill 的必传入参
get-rules 的 combinations[i] 四元组 get-assets-by-rules 的过滤条件、create-adgroup 的请求体顶层字段
get-assets-by-rules 的 marketing_asset_id create-adgroup 的 marketing_asset_outer_spec 子字段
get-creative-templates 的 creative_template_id get-component-list 的 sub_types 过滤、create-creative 的 creative_template_id
get-component-list 的 component_id build-creative-params 的 creative_components[i].component_id
create-adgroup 的 adgroup_id create-creative 的归属字段、update-adgroup-general 的目标字段、query-adgroups 的过滤条件
update-adgroup-general 的 _verify agent 用来跟用户预期做反思比对，不喂下一步

这张表读下来会发现——整条流水线本质上是一个数据管道，agent 在这条管道上做的事不是"生成参数"，而是"在事实里挑选并搬运"。这是这套 skill 跟通用 agent prompt 工程的一个本质区别：通用 agent 让 LLM 自由组装参数，这套 skill 让 LLM 只在脚本返回的事实空间里做一次"挑哪个"的选择。

这种"管道型 prompt"带来三个工程后果：

第一，每一步都是可验证的。前一步返回什么、这一步消费什么，都在文档里写明，agent 想偷懒拍一个值进去会立刻在下一步的脚本校验里翻车（比如 marketing_target_type 不在 get-rules 的 combinations 里，create-adgroup 直接报错）。这跟"agent 自由组装一个 JSON 然后 API 收到才报错"是两个世界——后者错误反馈链条长，前者错误反馈链条短到几乎没有。

第二，步与步之间的 context 是结构化的，不依赖 LLM 自己的"记忆"。SKILL.md 反复强调的"字段必须可溯源"在这种结构下变得可执行——所谓溯源就是回答"这个字段是从哪个上一步的哪个返回字段抄过来的"。这把 LLM 在跨轮对话里最容易出问题的"我记得用户之前说过 X"完全替换成了"上一步脚本返回了 X"。

第三，步骤可以被 agent 平台层做"断点续跑"。每一步的输入输出都是 JSON 化的事实，理论上一个 agent 跑到第 4 步崩溃，下一个 agent 接手时只要拿到前 3 步的脚本返回就能从第 5 步继续。这条 skill 协议没有显式承诺，但结构上是支持的。

下面把每一步具体展开。

第 0 步：鉴权（tencentads-auth）

任何 tencentads-* 调用之前，得先把 API Key 落到磁盘。auth 这个 skill 整体很轻——一个 auth-status.mjs 检查状态，一个 auth-save-apikey.mjs 落盘。SKILL.md 里有两条值得拎出来：

第一，API Key 不在 JSON 入参里走：

其他 skill 的脚本统一用 node scripts/xxx.mjs '<JSON 参数>' 的形式，唯独 auth 用显式命令行参数。SKILL.md 给的理由是"避免不同操作系统和终端下的转义/引号兼容性问题"——这个解释在跑过几个其他 skill 后会非常容易共鸣（下面那一节会专门说）。

第二，安全原则写得直白：「Agent 收到 API Key 后应立即调用保存脚本，不要在回复中重复展示凭据内容。」这一句在 prompt 工程上是有意义的——LLM 默认行为是"复述用户给的内容以确认理解"，对密钥来说这就是泄漏。skill 把"不许回显"作为第一红线写进协议，比起依赖模型的安全直觉要可靠得多。

auth 还和后面所有 skill 形成一个对照：当任何 tencentads-* 脚本返回 未找到腾讯广告认证凭据 / AUTH_REQUIRED / AUTH_EXPIRED / Authentication is not valid 时，agent 应该回到这里走鉴权流程，再重试原始操作。这是把"凭据失效"做成了一个跨 skill 的事件，而不是让每个 skill 各自重新发明一套登录逻辑。

决策面之一：四元组只能从脚本返回里选

四元组指 marketing_goal、marketing_sub_goal、marketing_target_type、marketing_carrier_type。SKILL.md 用整整一节强调：“必须从脚本返回的可选范围中选取，禁止通过推理猜测”，理由列了三条——不同账号准入的组合不同、自然语言场景对应的 goal 不同账号上不一样、四元组错误会级联污染下游。

调用方式：

脚本内部解析 rules_json 嵌套 JSON 字符串，把四层嵌套树展平为 combinations 列表。返回多条时按"用户意图提取关键约束 → goal 匹配 → carrier 匹配"分层过滤，过滤后仍多于一条就让用户确认，禁止猜测。

更值得抄的是 smart-create 里的 1A 步——通过资产 ID 反推四元组：

如果用户给了"推广产品 ID / 应用 ID / marketing_asset_id"这种结构化信号，就用它去反向收敛 marketing_target_type，避免向用户多问一轮。这种"能不问就不问"的设计，是 skill 在和真实业务流摩擦之后才会写出来的细节。

决策面之二：枚举不能"近义词改写"

SKILL.md 反复强调一条规则：

marketing_sub_goal 只能复用 get-rules.mjs 返回值，禁止语义改写。例如用户说"注册"、"新客"时，不要自造 MARKETING_SUB_GOAL_USER_REGISTER；必须原样使用返回的枚举值。

这是写给 LLM 看的，因为 LLM 太擅长"按命名规律拼一个看起来对的枚举"。management 那个 skill 在 query-fields 那段里给了一个更直接的例子：用户说"关注数"，LLM 凭直觉拼出 wechat_official_account_follower_count，实际正确字段名是 scan_follow_user_count。所以 management 干脆要求：“如果字段不在常用映射表里，必须先用 --query-fields 查字典再构造请求”，否则"猜测几乎必错"。

这两个要求合起来读，是一种很明确的工程立场——结构化字段是事实，自然语言是诉求；事实必须来自接口，诉求只能用来过滤事实。

决策面之三：可选组件未经允许不许加

tencentads-creatives 在第二步查到组件列表后，文档上专门标了一条：

required=false 的组件（如浮层 floating_zone、标签 label、社交互动 social_skill、数据外显 show_data 等）未经用户许可禁止自行构建和添加。仅在用户明确要求该组件时才加入。

这是另一条"防 LLM 主动加戏"的约束。模型读到"可选 = 用户没明确不要 = 我可以主动给加上"是常态，但在投放语境下，多一个浮层、多一条数据外显，意味着创意构成、合规审核、跑量行为都会变。这条规则把"被动响应用户意图"和"主动补全字段"明确切开了。

决策面之四：查不到就跳过，绝不占位

tencentads-creatives 在 component_id 查询失败的处理上写得相当具体：

严禁用 0、null、-1 等无效值占位 component_id：查询失败的组件条目整体省略，不填虚假 ID。

video 组件查询失败时：绝对不可生成 [{}]（空对象数组）——这会让 API 收到无效的空视频条目。

这是把 LLM 在工具调用失败时的"找一个看起来合理的默认值填上"行为，明确禁掉了。背后的取向很清楚——宁可缺一个字段、让上层报错，也不要伪造一个能让请求通过但结果错乱的值。这条规则在 smart-create 的执行协议里也有对应表述：「每个脚本调用最多重试 1 次。如果 2 次仍失败，立即跳过该步骤，用已有数据继续推进。严禁反复重试同一脚本、读取脚本源码、或切换到非脚本方式（如 tencent-ads api）尝试绕过。」

把这四条放在一起看，整套 skill 在防的不是"agent 调错接口"，而是"agent 在不确定的时候自己编一个"。前者靠文档可以解决，后者要靠协议——这一组 skill 选了协议路线。

这个 skill 的设计有两点值得说。

第一，绝对值和相对调整解耦。bid_amount 是绝对值（元），bid_amount_adjustment 是表达式（"+20%" / "-10%" / "*2" / "+0.5"），二选一。这是直接把"出价加 10%"这种自然语言诉求映射到了脚本接口上，不让 agent 自己读当前值再算新值——前者是一次幂等的语义调用，后者要先 read 再 write，期间任何并发都可能把账打错。

第二，自动回查 + 反思比对：脚本执行后会输出 _verify 字段（包含广告更新后的实际值），Agent 必须基于 _verify 回查数据进行反思比对：对照用户预期、检查金额单位（API 返回分，用户期望元）、不一致时明确告知用户。这是把"写完之后默认成功"这种 LLM 习惯改成了"写完必须验"。

字段表里 4 个金额字段（bid_amount / daily_budget / deep_conversion_behavior_bid / auto_acquisition_budget）统一用元、脚本自动 *100 转分；deep_conversion_worth_rate 是比率不做转换。这种"在 skill 层统一单位、在脚本层转换"的做法，比让 agent 每次自己换算要可靠得多。

_verify 是分布式系统"写后读"模式的微缩版

_verify 这个字段单独看是一条工程实践——“写完别假设成功，再读一次回来对一下”——但放在分布式系统的语境里看，它对应的是一个有名字的模式：read-after-write（写后读）。这个模式在分布式存储里出现得很多，几乎所有"最终一致性"系统在写操作返回之后都会面临同一个问题：客户端怎么知道这次写真的生效了？

腾讯广告 API 这一层为什么也需要 read-after-write，原因有三条：

第一，API 返回的"成功"不等于"字段值就是你传的值"。投放系统里有很多"软兜底"——传的 bid_amount 超过了行业上限会被自动 clip 到上限值、传的 daily_budget 低于该出价方式的最小预算会被向上修正、传的 start_date 早于今天会被改成今天。这些 silent clamp 不会在响应里抛错，但写出去的值跟你以为的值不一样。_verify 把"我以为我写了 120 元"和"系统实际记下来是 100 元（因为撞到了上限）"的差异显式暴露给 agent。

第二，金额字段的元-分换算是个高发错位点。脚本入参用元、API 内部用分，转换发生在脚本层。但 _verify 里的回读值是直接从 API 拿的——也就是说回读到的是分。SKILL.md 专门提醒"检查金额单位（API 返回分，用户期望元）"，意思是 agent 拿到 _verify 时要自己除 100 再跟用户预期对照。这是一个非常具体的"两端单位不一致需要在比对时显式做单位换算"的场景，跟分布式系统里"客户端时间戳 vs 服务端时间戳"的对账问题是同构的。

第三，LLM 默认行为里没有"写后读"的概念。LLM 接到一个工具调用返回 {success: true} 的标准反应是"那就成功了，往下走"。这是被强化学习训练出来的"工具调用乐观假设"，在 chat 场景里基本没问题，在写操作场景里是危险的。_verify 协议把这个乐观假设改写成了"看到 success 之后必须再看一眼 _verify 字段"。这种把分布式系统经验下沉到 prompt 协议层的做法，是把"工程师默会的复核习惯"变成了 LLM 的可执行约束。

把这三点合起来读，_verify 不是一个"额外的安全网"，而是这套 skill 认认真真把 API 调用当作分布式写操作来对待的产物。它跟分布式数据库里 read-your-writes 一致性、Stripe API 的 idempotency_key 回查、甚至 etcd 的 compare-and-swap 都属于同一族——核心都是"不要相信自己刚刚写过什么，从权威源再读一次"。

值得注意的是这套 skill 没有把 _verify 做成强制重试机制：回读的值跟期望不一致时，协议要求的是"明确告知用户"，而不是"自动改回去"。这个克制是对的——投放系统里"我以为我写 120 但被 clip 到 100"的正确处理多半是问用户"是不是要换出价方式让 120 能生效"，而不是机械重试。read-after-write 解决的是可见性问题，不是补偿问题——这套 skill 把这条线划得很清楚。

第 5 步：看数据（tencentads-management）

最后是回看。management 是这一组里 SKILL.md 篇幅最长的一个，但它解决的事情其实很集中：让 agent 在"用户笼统说一句"的时候，能正确分流到报表查询、广告详情、还是创意详情。

它的分流规则写得相当严谨。两条核心规则：

关键规则：用户笼统说"查询广告数据"、"看下创意数据"等，**未明确提到指标关键词（消耗、曝光、点击等）**时，一律归入 B 类（实体查询），而非 A 类（报表）。

tencent_ads_type 是枚举类型，只允许 "smart" / "standard" / "all" 三个值，传其他值脚本会报错退出。

第二条尤其值得抄——tencent_ads_type 在脚本侧 fail-closed 校验，只接受三个枚举之一。这是把"agent 在调用前必须先把意图想清楚"做成了硬约束：你不能传 "project"，不能传 "ad"，不能传一个看起来合理但实际不存在的值，否则脚本直接拒。同时这个字段决定了返回字段的命名（smart 走 project_id / project_name、standard 走 adgroup_id / adgroup_name、all 保留原字段名），是一个有结构后果的开关。

management 还有一个让人印象深刻的细节：单创意自动解析。query-creatives.mjs 自动判断结果条数——只有 1 条创意时，脚本自动执行"提取 component_id → 拉取组件详情 → 提取 image_id / video_id → 获取预览 URL → 内联到 _component_detail + _preview"，一次返回完整结果；多条时只返回基本信息不解析组件，避免炸 API 配额。这种"按结果规模动态调整副作用深度"的设计，比起让 agent 自己判断"我要不要再展开一层"要省心得多。

management 是整套 skill 里最复杂的"分流器"

把 management 跟前面六个 skill 摆在一起对比，会发现一个不对称——其他 skill 的边界是按"动作类型"划的（鉴权、寻址、建组、建创意、改广告），一个 skill 干一件事；唯独 management 一个 skill 里塞了至少五类完全不同形态的查询：

  1. A 类报表查询：跨账户、按指标聚合、按时间维度切片，对应 daily_reports/get 这种重量级接口。
  2. B 类实体查询（广告 / 项目）：拉某个 account_id 下的 adgroup_id 列表、查某条广告的完整配置（定向、出价、版位）。
  3. B 类实体查询（创意）：跟广告查询是同一类意图但走的接口完全不同，且有"单条自动解析"的特殊路径。
  4. C 类组件 / 素材详情：拉 component_id 对应的图片视频原始信息、获取预览 URL。
  5. D 类字段字典查询：--query-fields，专门用来回答"我能查哪些指标"这个元问题。

这五类查询在用户嘴里的措辞高度重叠——"看下广告怎么样"既可以是 A 类（要看消耗 / 曝光），也可以是 B 类（要看配置）；"查一下创意"既可能要看创意列表（B），也可能要看创意素材（C）。这套分流要做对，靠的是 SKILL.md 里那张被反复引用的"动作 → 关键词 → skill 路径"映射表。

为什么这五类不像 create 那样拆成五个独立 skill？把它们摆在一起对照 smart-create / standard-create 的拆分逻辑，会得到一个相反但合理的答案：

  • create 拆分的本质是参数体不兼容。智投和常规的请求 schema 差异大到无法收敛在一个 SKILL.md 里讲清楚，硬合并会让 agent 在中段判断走错。
  • management 合并的本质是入口意图同质。读、查、看——五类查询在用户语言层面是一个动词族，强行拆成五个 skill 会让用户在"我要看一下广告效果"这种场景下不知道该触发哪个，反而把意图歧义消解的负担推给了用户而不是 skill 协议。

合并带来的代价是 SKILL.md 变得很长——这一篇是七个 skill 里篇幅最长的，因为它必须把"动作分类法"完整地写在 prompt 里。但代价值得：入口分流由 SKILL.md 文档承担一次，比让用户每次都先选 skill 要省心。

这套分流决策树的几条具体规则，已经能从 SKILL.md 摘出来对照了：

用户信号 路由结果
提到指标关键词（消耗 / 曝光 / 点击 / CTR / CPM） A 类报表，走 query-reports.mjs
没提指标但提到"广告" / “项目” / “怎么样了” B 类实体查询，走 query-adgroups.mjs
提到"创意" / “素材” / “图片” / “视频” B 类创意查询，走 query-creatives.mjs
提到一个不在常用映射表里的指标名 强制先走 --query-fields 字典查询，再回到 A 类
用户说"智投" / “项目” / “AIM+” tencent_ads_type=smart
用户说"常规广告" / “标准广告” tencent_ads_type=standard
用户说"全部" / 没指明 tencent_ads_type=all，返回字段保持原命名

最后一行尤其值得说——all 这个枚举值不是 union，而是一个保留原始字段命名的特殊标记。意思是：返回数据里 project_id 还是 project_id、adgroup_id 还是 adgroup_id，agent 拿到结果后要自己判断每条记录是哪种类型。这把"返回结构"和"过滤条件"做了一次解耦——tencent_ads_type 决定要查谁，但不决定返回时改不改名。这种"语义清晰的 fail-closed 枚举 + 保留歧义返回字段"的组合，在 API 设计上是一个很罕见但很实用的姿态。

一个绕不开的问题：无状态 skill 怎么安全拿到用户登录态

把流程走完一遍后，回过头会发现一个更基本的问题没被正面回答：每个广告主有自己的 API Key，每个 user 在腾讯广告平台后台是独立账号；而 skill 在设计上是无状态的、函数式的——每次调用都是一个新的 node scripts/xxx.mjs 进程，进程跑完就退出，下次调用不知道上一次发生过什么。这两件事怎么对齐？

更具体地说：

  • 用户的 API Key 不可能每次都让他在对话里重新发一遍——既不安全（密钥反复出现在 chat 历史里），也不现实（一次投放流程要走 7-8 个 skill）。
  • 但 skill 自己又不能"记住"密钥——SKILL.md 是协议文档，进程之间不共享内存。
  • agent（LLM）也不能把密钥放在自己的上下文里复用——LLM 上下文会被截断、会被压缩、会被记录进对话日志。

这套 skill 给出的答案值得拎出来单独说，因为它解决的是所有"agent 调真接口"场景都会遇到的问题，不是腾讯广告独有的。

拆解：把凭据从 LLM 上下文里"剪掉"

整个鉴权机制由三个角色构成：

角色 形态 是否接触凭据明文

tencentads-cli（npm 包） 共享 SDK，提供 callApi、TencentAdsAuth 每次调用读盘，封装在 callApi 内部

落盘策略：本地文件而不是环境变量、不是参数、不是 LLM 上下文

auth-save-apikey.mjs 的实际逻辑只有几行：

落盘位置由 SDK 写死在 ~/.tencent-ads/credentials.json。这个选择和几个常见的备选方案对比一下就看出取舍：

  • 环境变量：每次 agent 启动新 shell 都要重新 export，跨进程不可靠；而且环境变量在父子进程里默认继承，agent 自己的 LLM 后端进程也能看到，反而扩大了暴露面。
  • 每次调用作为参数传入：意味着 LLM 上下文里必须保留密钥才能在后续调用里继续传——这就回到了"密钥反复出现在对话历史"的问题。
  • agent 自己记着：LLM 的上下文是会被截断、压缩、在多 agent swarm 间传递的，密钥放在那里相当于公开。
  • 本地文件 + 用户家目录：唯一一个"agent 看不到、但下一个进程能读到"的位置。文件权限由 OS 管，scope 限定在当前用户，跨 skill 跨进程都能用。

这个选择把"密钥的可见性"和"密钥的可达性"做了一次切割——LLM 不可见、子进程可达。这是无状态函数式调用想拥有"会话级身份"时唯一干净的解法。

prompt 红线：明确禁止 LLM 复述密钥

光有落盘机制还不够，因为密钥进到这套系统的那一瞬间一定要经过 LLM——用户把它发到对话里，LLM 读到它，再决定调 auth-save-apikey.mjs。这个窗口期内有两个高危行为，SKILL.md 顶部直接写明禁掉：

安全原则: API Key 等敏感凭证不应在对话窗口中回显。Agent 收到 API Key 后应立即调用保存脚本，不要在回复中重复展示凭据内容。

这条规则很短，但它在防的是 LLM 的两个本能：

第一，复述用户输入以确认理解。“好的，我收到了你的 API Key mkt_xxxx，现在为你保存”——这是 LLM 的默认对话礼仪，对普通参数没问题，对密钥就是泄漏。SKILL.md 把"立即调用保存脚本"和"不要在回复中重复展示凭据"绑成一条强约束。

第二，在工具调用回显里把参数原样打印。很多 agent 框架会把 tool_call 的 arguments 直接打印到对话流里供用户审阅。这套 skill 选了一个更彻底的解法——让密钥不走 JSON 入参：

对比业务 skill 的统一姿态 node scripts/xxx.mjs '{"key":"value"}'，唯独 auth-save-apikey.mjs 用 --api-key 这种显式 CLI 参数。SKILL.md 给的明面理由是"避免不同操作系统和终端下的转义/引号兼容性问题"——这个理由在跑过几个其他 skill 后会非常容易共鸣（PowerShell 5.x 那段引号事故就是反例）。但它还有一个隐藏理由：CLI 参数 --api-key value 这种形态，在大多数 agent UI 的工具调用回显里更容易被识别为"敏感字段"做掩码处理，比埋在 JSON 里的某个 key 容易被前端拦截。

读源码会发现 auth-save-apikey.mjs 还兼容旧版的 JSON 入参——if (legacyArg && legacyArg.trim().startsWith("{")) 那一段。也就是说这个迁移是有历史痕迹的：早期版本把 API Key 当普通参数走，后来发现问题，专门改成显式 CLI 参数并把旧路径标为 不再推荐。这是一个真实在生产环境踩过坑后才会留下的代码形态。

失效兜底：把"凭据失效"做成一个跨 skill 的事件，而不是登录闭环

到这里凭据的"进"和"用"都解决了。还有一个问题——凭据会过期、会被吊销、会因为账号变更而失效。无状态系统怎么处理这种动态失效？

这套 skill 的做法是把它做成一组明文契约的错误码 + 一份明文契约的恢复流程，刻在每个 skill 的 SKILL.md 里。tencentads-auth/SKILL.md 列出了四个标准错误：

当 callApi 返回以下错误时，应触发本技能重新鉴权：

  • 未找到腾讯广告认证凭据 — 凭据未配置
  • AUTH_REQUIRED — 凭据未配置
  • AUTH_EXPIRED — 凭据已过期或无效
  • Authentication is not valid — 服务端拒绝认证

这四个错误码由 tencentads-cli 的 callApi 在不同情况下抛出（前两个是本地检查、后两个是服务端响应），但**"看到这些错误就回 tencentads-auth 重新走鉴权"是写在每个 SKILL.md 里的协议**，不是 SDK 内部硬编码的行为。

这个设计取向很关键。SDK 只负责"产生事件"，agent 负责"响应事件"。原因是无状态 skill 没法自己处理"重新拿凭据"——这件事必须经过用户（密钥不能让 agent 自己生成）。所以 SDK 不能阻塞重试，只能把信号抛给上层；上层（agent + SKILL.md 协议）拿到信号后，根据协议路由到 tencentads-auth，由它引导用户走一遍输入流程，存盘后再回到原任务。

恢复流程也写得很具体：

  1. 告知用户认证已失效，需要重新配置
  2. 引导用户提供新的 API Key
  3. 调用 auth-save-apikey.mjs 保存
  4. 鉴权成功后，自动重试引发错误的原始操作

第 4 步是关键——它要求 agent 在失效之前记得自己原来在做什么，鉴权完成后回到原任务而不是丢上下文。这是把"中断 → 鉴权 → 恢复"做成了一个跨 skill 的可恢复流程，而不是让用户在每次失效后从头说一遍需求。

这一套机制的边界

再诚实一点，这套机制并不是没有缝隙：

  • 本地文件路径是固定的。~/.tencent-ads/credentials.json 写死在 SDK 里。如果同一台机器上有多个用户共用一个 OS 账号（比如远程开发机），他们会共享同一份凭据——这个层面的隔离要靠 OS 用户来做，不是 skill 层能解决的。
  • 没有多账号 / 多租户的内置支持。一个 agent 同时为多个广告主操作时，要么频繁覆盖 credentials.json，要么得在 SDK 外面包一层多文件路由。SKILL.md 没有提到多 profile 机制，这是当前版本的一个能力空白。
  • 密钥仍然要经过 LLM 一次。用户把密钥发到对话里那一瞬间，密钥就进了 LLM 的上下文。SKILL.md 能约束 LLM 不复述、不回显，但没法让 LLM "看不见"这个输入——agent 后端如果把每轮 user message 都写日志，密钥仍然会被记录。要彻底干净，得靠 agent 平台层做敏感词识别和自动 redaction，不是 skill 单独能解决的。
  • storagePath 在保存成功后会回显给 agent。auth-save-apikey.mjs 的输出包含 "storagePath": "~/.tencent-ads/credentials.json"——这条返回会进 LLM 上下文。它不是密钥，但暴露了凭据的物理位置；如果 agent 环境里同时有其他工具能读任意路径，这个信息会成为攻击链的一环。

这些缝隙不是这套 skill 没想到，而是 skill 这一层本身能做的事就到这里——再往下需要 OS 权限模型、agent 平台的 secret manager 集成、或者 OAuth-style 的短期凭据机制来兜底。

抽象出来：无状态调用想拥有"会话身份"的通用范式

把腾讯广告这套机制脱水一下，会得到一个任何"agent 调真 API"场景都能复用的范式：

  1. 凭据通过专用入口一次性写入用户级安全存储——本地文件、OS keychain、平台级 secret manager 都行，关键是这个存储 LLM 不可见、子进程可达。
  2. 共享 SDK 封装"读凭据 + 拼请求"——业务 skill 完全不感知凭据的存在，看到的世界只有"调接口、拿返回"。
  3. 凭据写入入口和业务调用入口在 prompt 层面强分离——前者用专用 skill，后者用业务 skill，互不混淆。
  4. prompt 协议禁止 LLM 复述凭据、禁止把密钥放进可被回显的 JSON 入参——把"防 LLM 复述"做成红线。
  5. 失效用标准错误码 + 标准恢复流程跨 skill 广播——SDK 只产生信号，agent 负责响应，恢复后回到原任务而不是丢上下文。

这五条放在一起，就是这套 skill 给"无状态函数式 skill 如何承载用户级登录态"这个问题给出的工程答案。它不优雅——本地文件、家目录路径、prompt 红线，听起来都是"凑合做"的方案。但它比任何"让 LLM 自己管密钥"或者"每次调用都让用户重新输"的方案都更可落地。这一组 skill 在这件事上的取舍非常清醒：密钥安全这件事，不能靠 LLM 的自觉，要靠把 LLM 从凭据的处理路径上彻底剪掉。

几条横贯所有 skill 的姿态

走完一遍流程后，把 SKILL.md 拎起来翻，会看到几条几乎在每个 skill 里都重复出现的设计取向。

姿态一：脚本是黑盒，agent 只做业务决策

这是最被反复强调的一条。smart-create 的开头写：“Agent 专注于：从脚本返回结果中做出业务决策（选哪个组合、选哪个转化目标）。” 脚本负责"API 分支路由、返回数据裁剪、格式转换（rules_json 解析、geo 编码查找等）"。

这个分工的取向很清楚——SKILL.md 是给 LLM 读的协议，脚本是给执行环境跑的代码。能让 deterministic 代码做的事就别交给 LLM 概率性地做。例子非常多：get-rules.mjs 内部解析 rules_json 嵌套 JSON 字符串、get-assets-by-rules.mjs 自动对 marketing_target_type 去重、update-adgroup-general.mjs 自动把元转分、query-creatives.mjs 单条结果时自动展开组件——所有这些都不是 LLM 的活。

姿态二：跨平台引号事故学

有一段几乎一字不差地复制在 advertiser / creatives / smart-create / standard-create 几个 skill 里：

终端环境 正确写法
Linux / macOS (Bash/Zsh) node scripts/xxx.mjs '{"key":"value"}'
Windows CMD node scripts/xxx.mjs "{\"key\":\"value\"}"
Windows PowerShell 5.x node --% scripts/xxx.mjs "{\"key\":\"value\"}"

文档里直接写明：「PowerShell 5.x 是重灾区：单引号、反引号、反斜杠三种常见写法全部失败。」management 那个 skill 干脆给 PowerShell 单独走 --base64 通道：把 JSON 编码成 Base64 再传，彻底绕开引号问题。

这一段在每个 skill 里复制出现，本身是这套 skill 在生产环境踩过坑的痕迹。值得抄走的不是这张表，而是这件事本身——任何"从 LLM 输出落到 shell 命令"的链路都要假设引号会被吃掉。auth 那个 skill 索性把 API Key 改成 --api-key <value> 走显式参数，理由就是这个。

姿态三：fail-closed 而不是 fail-open

smart-create 和 standard-create 的"执行协议"里有一条很硬的规则：

进入步骤 7（智投）/ 步骤 6（常规）前必须能回答每个关键字段来自哪里。说不清来源的字段，不要猜。

合法来源只有三类——用户原始需求里明确给出、当前 session 前一步脚本返回、当前 session 前一步 API 返回。这是把"上下文溯源"做成了进入下一步的前置条件。LLM 默认会做的"我记得用户之前好像说过 / 我推测应该是这个值"，在这套 skill 里都不算合法来源。

加上前面提到的"查不到就跳过、不占位"、“重试上限 1 次”、“枚举值禁止改写”，整套 skill 在偏好上是非常一致的：宁可在上层暴露一个"我没拿到这个字段"的状态，也不要让请求带着伪造的字段去打 API。

姿态四：skill 的边界按"意图能不能被歧义消解"划

最后这一条更隐蔽，把七个 skill 摆在一起对照才看得出来。

smart-create 和 standard-create 拆成两个 skill，根本原因不是 API 不一样（七步骨架几乎一致），而是意图分流必须前置到 skill 选择那一层。让一个 skill 里既处理智投又处理常规，等于把"用户说的’拉新’到底是智投拉新还是常规拉新"的判断丢进 SKILL.md 中段——LLM 在中段做的判断错误率远高于在顶端做的判断。把它拆成两个独立 skill，让 description 关键词去做一次性的硬路由，是更保险的方案。

同样的逻辑可以解释 creatives 为什么明确说"排除 search_dynamic_creatives"——搜索创意是查询动作，跟"管理动态创意"不是同一组意图，硬塞在一起会让这个 skill 的 description 失焦、触发率下降。update 把广告和智投项目合并是反方向——它们更新时确实是同一类意图（“调一下出价 / 暂停一下”），合并反而让触发更稳。

这是这套 skill 给所有 skill 设计者的一条隐性建议：先想清楚用户在你这个 skill 边界上会说哪几类话，再决定要不要拆。

姿态五：并发安全模型不靠 LLM 自己想

读 SKILL.md 的时候很容易忽略一类规则——它们不在"主流程"里，而是散落在各个角落，但合起来构成了一份完整的并发约束清单。LLM 默认会把"独立操作"和"可并行"画等号，把这套约束完全交给 LLM 推断是危险的，所以这些规则被显式写在了协议里。

整理一下这一组 skill 在并发上画的几条线：

隔离级别 什么操作必须串行 出处
同一广告组下 创意的新建 / 更新 / 删除 tencentads-creatives 头部"串行执行约束"
同一广告 不能同时跑两条 update-adgroup-general standard-update 隐含——_verify 回查协议假设串行
同一账户 视频 / 图片上传与同账户的创意创建之间有先后依赖 creatives 工作流零的转码轮询
跨账户 默认可并行 update-adgroup-batch.mjs 接受 tasks[] 数组
跨 skill 鉴权失效是全局事件，并发任务都要回到 auth 重试 tencentads-auth 失效兜底协议

几条值得读出来的取向。

第一，互斥粒度按"业务实体"划而不是按"API 接口"划。同一广告组下的创意操作要串行，不是因为 dynamic_creatives_* 这几个接口本身不支持并发，而是因为"一个广告组下的创意集合"是一个业务一致性边界——并发新建可能命中创意数量上限、并发更新可能让 _verify 读到中间态。把粒度划在业务实体上而不是接口上，是 skill 层在替业务系统兜底"幂等性 / 顺序性"。

第二，批量入口的存在不等于批量内部并发。update-adgroup-batch.mjs 接受 tasks[] 是为了减少 round-trip，不是为了在一个进程里把多个 update 真正并行打出去。SKILL.md 没有承诺批量内部的执行顺序——这意味着如果两个 task 改的是同一个 adgroup_id（极端情况），结果是未定义的。这条没明说的边界，是任何接入方在生产环境前必须自己想清楚的。

第三，鉴权失效是个全局信号，不是单 task 信号。如果一个 swarm 里有 3 个 agent 并行跑 tencentads-*，其中一个先撞上 AUTH_EXPIRED，按协议它应该停下来回 tencentads-auth 重新走鉴权——但另外两个 agent 此刻也一定会撞上同一个错误，因为凭据是账户级共享的。SKILL.md 没有写"撞到 AUTH_EXPIRED 时通知其他并发 task 暂停"这种协调逻辑，意味着 swarm 模式下三个 agent 会各自重复一遍鉴权流程，最坏情况下让用户在对话里输三次 API Key。要避开这个，需要在 agent 平台层加一个"鉴权事件去重"——这件事 skill 这一层做不到。

第四，LLM 不被允许自行判断"这两件事独不独立"。整个 swarm 里最危险的 prompt 反模式是 LLM 看到"用户说要给广告 A 改预算、给广告 B 暂停"就理所当然地"那我并行调两个 update 吧"。standard-update 用 update-adgroup-batch.mjs 把这件事正面承下来：要批量就走批量接口、不要在 agent 层面自己开并行。这是把"并发与否"这个决策从 agent 收回到 skill 协议本身。

把这四条放在一起看，整套 skill 给 swarm 模式的隐性建议是——任何并发都要在 skill 协议里被显式批准，没有显式批准就默认串行。这跟通用编程里"默认尝试并行、必要时加锁"的取向正好相反，但对 LLM 编排是更安全的姿态。

几个值得保留警惕的点

SKILL.md 写得好不等于实装就到位。这一组 skill 里有几处需要标注：

重试和跳过规则会真实丢字段。 「每个脚本调用最多重试 1 次，2 次仍失败立即跳过」是实用主义的处理，但代价是最终 adgroups/add 请求体里可能少一个字段。SKILL.md 直白承认了这一点：「宁可在最终请求中缺少某个字段，也不要耗尽所有轮次。」对调用方来说，需要心里有数——这种"成功"是带条件的，下游的报错处理不能假设字段都齐。

Mock not found 这个错误码在多个 SKILL.md 里被当成已知态来兜底。 智投、常规、创意三个 skill 在错误处理一节都明确写了"若返回 ‘Mock not found’ 时直接跳到下一步"。这是从字面措辞推断出来的信号：脚本在某些路径上是有 mock 路由的，具体哪些接口已经接到真实 API、哪些还在 mock，文档里没有 single source of truth，要回看脚本源码或实际跑一遍才知道。

自动推断的兜底链条要看清。 比如 wechat_channels 的自动推断走"广告组上下文 marketing_asset_outer_spec → adgroup_context 参数 → …"的多级 fallback，每一级失败时的 fallback 行为需要回看脚本源码才能完全确认。SKILL.md 给的描述够用，但不是穷尽的。

management 的 query-fields 字典查询是必走步骤。 文档反复强调"禁止猜测字段名！违反此规则 = 查询失败！"，但凡用户要的指标不在常用映射表里，就得先 --query-fields 查一遍。这个限制会让"agent 一句话查报表"的体验变成"agent 先查字典再查报表"，对延迟敏感的场景需要提前知道。

这些都不是问题，是工程取舍。但如果有人计划把这套 skill 接到生产 agent 工作流里，这几条需要写到接入指南的醒目位置。

谁应该用、谁应该等

把它放回 agent skill 这一层来看，谁该用？

适合现在就接入的：

  • 已经有腾讯广告投放团队，想把"账号查询、广告改预算、看报表"这种重复操作交给 agent 跑的。这是 management + update 的最佳场景，单 skill 调用就能解决，决策面窄、出错代价低。
  • 想做"投放 agent 助手"原型的团队。create 这条线的 7 步流程已经把决策点全部明示出来了，照着流程做一个原型是非常顺的。
  • 在做自己的 skill 体系、需要参考"如何给 LLM 写一份不让它瞎来的 SKILL.md"的人。这一组 skill 在这件事上提供了相当多可抄的细节——四元组防猜测、可选组件不主动加、查不到不占位、枚举不改写、fail-closed 协议等等。

应该再等一等的：

  • 想把 create 流程接到无人值守的生产 pipeline 里、并且对"漏一个字段"零容忍的。重试 1 次跳过的策略对原型友好，对生产需要补充更细的错误降级。
  • 不在 Linux / macOS 工作环境，且不愿意在 PowerShell 上走 --base64 通路的。Windows CMD / PowerShell 5.x 那一套引号事故每个 skill 都要踩一遍。
  • 期待文档里写的所有"自动推断"都已经实装到位、不需要回查源码就能上手的。SKILL.md 各文件 frontmatter 里声明的版本还在 0.5.0（多数）/ 0.2.0（standard-update）量级，部分能力以"协议优先、实现追赶"的状态存在。

跟 MCP / OpenAPI Plugin / Function Calling 的横向对比

读到这里，一个很自然的问题是：让 agent 调真 API 这件事，业界已经有 MCP server、OpenAPI plugin、纯 function calling 几条成熟的路线，腾讯广告这套 SKILL.md + 黑盒脚本的方案到底站在哪个位置上？

把四种范式摆在一起，差异不在"能不能调通接口"，而在"把’决策’和’执行’切在哪个位置"。这是 agent 工程里被讨论得比较少、但实际决定项目能不能上生产的那个轴。

维度 Function Calling（裸 schema） OpenAPI Plugin MCP Server 这套 SKILL.md + 脚本
协议载体 JSON Schema OpenAPI YAML / JSON stdio / sse 工具列表 Markdown 文档 + Node 脚本
决策面收窄手段 schema 里的 enum、required 同左，加 description 工具的 description prompt 协议本身（fail-closed、四元组防猜测、查不到就跳过）
字段从"事实"反查 不内置 不内置 不内置 内置（get-rules / get-component-list / query-fields）
失败兜底 LLM 自己重试 LLM 自己重试 可定义 error schema 标准错误码 + 跨 skill 恢复流程
跨调用状态 无 无 可在 server 里维护 落盘到本地文件，agent 不感知
单元测试 / mock 各家自定义 各家自定义 协议无规定 Mock not found 错误码兜底
安装成本 几乎零 一份 OpenAPI 一个 server 进程 一组 skill 目录 + npm 包

几个轴上值得展开。

协议载体的差别比看起来重要。 Function calling 和 OpenAPI 用的是结构化 schema，schema 的表达力终究是有限的——你可以写 enum、写 description、写 required，但你写不了"如果用户说’拉新’但 enum 里有 MARKETING_SUB_GOAL_ACQUIRE_USER 和 MARKETING_SUB_GOAL_NEW_CUSTOMER，必须先调 get-rules 看当前账号准入哪一个"这种条件性、动态、跨调用的规则。MCP 工具的 description 是自然语言但通常很短，写不下完整的执行协议。SKILL.md 用 markdown 写完整的协议文档，牺牲了机器可读性，换来了表达条件性规则的能力——这是这套范式相对前三种最核心的取舍。

"决策面收窄"是这套范式真正的差异化能力。 前三种范式都默认 LLM 是个理性 agent：你给它正确的 schema，它就会按 schema 调对接口。这个假设在简单 CRUD 场景下成立，在腾讯广告这种"枚举几十个、字段互锁、单位易错、单条调错就花钱"的场景下完全不成立。SKILL.md 的做法是把"LLM 在哪些点最容易出错"先识别出来（四元组、可选组件、占位 ID、单位换算），然后针对每个易错点写一条显式的反模式禁令。这件事用 OpenAPI 也能写在 description 里，但写完了 LLM 不会真的把它当协议执行——markdown 段落里的"严禁 / 必须 / 绝对不可"对 LLM 的约束力远高于 schema 的 description 字段。

MCP 跟这套范式最像，但仍然不一样。 MCP server 也是黑盒进程、也用工具列表对外暴露能力、也可以在 server 内部读凭据文件——架构上跟 node scripts/xxx.mjs 几乎是同构的。但 MCP 协议不规定"工具调用之外的协议文档"：MCP server 给 agent 的是一份"我有哪些工具、每个工具接什么参数"，给不了"agent 在调这些工具之前要做什么决策、之后要做什么校验、错误时要走什么恢复流程"。这些只能写在 prompt / system message 里，但又没有标准位置。SKILL.md 把这些显式地组织起来，是 MCP 之上的一层"agent 工程协议层"。换句话说，MCP 解决了"工具怎么暴露"，SKILL.md 解决了"agent 怎么用工具"——这两件事不冲突，理论上 SKILL.md 完全可以建在 MCP 之上。

反过来，SKILL.md 比不过 MCP 的地方也很清晰。 MCP 的工具列表是机器可读的，agent 框架可以拿来做工具发现、按 capability 过滤、跨 server 组合。SKILL.md 是一份给人 / 给 LLM 看的 markdown，工具发现完全靠 description 的关键词匹配，跨 skill 组合靠 SKILL.md 之间的"互相提一句"。规模一上来（七个 skill 还能管，七十个就乱了），这套靠人写文档协调的范式就开始吃力。

生态位预测： 这套 SKILL.md + 黑盒脚本的范式，最适合**“接口数量中等（10-50 个）、单次调错代价高、决策树深"的领域——广告投放、ERP / CRM 写操作、金融下单、医疗 EHR 写入。MCP 更适合"接口数量很大（数百个）、单次调错代价低、组合多变"的领域——文件系统、数据库查询、知识库检索。Function calling 适合"接口数量少、决策面浅、原型阶段”**。OpenAPI plugin 在 agent 工程语境下其实是个尴尬的中间态——比 function calling 重，又没 MCP 灵活，没有 SKILL.md 的协议表达力。

把腾讯广告这套放回坐标系里，它的位置就清楚了：它选了一条"协议文档为中心、脚本为执行器、agent 只做有限决策"的路线，本质上是给"高风险、决策树深"的领域生造了一套约定。这条路线没有标准化、没有 SDK、没有跨项目复用机制——但它在它所处的领域里是当下最稳的解。如果有一天 MCP 协议补上"工具调用前后的执行协议"这一层，或者出现一个标准化的"agent skill 描述格式"（不只是 description 字段），这种 SKILL.md 范式可能会被吸收掉。在那之前，它仍然是一个值得被仔细读的样本。

结语

这一组 tencentads-* skill 真正的价值不是"agent 能投腾讯广告了"，这件事用 MCP server + 一份 OpenAPI 描述也能做。它的价值在于把"如何让 LLM 在生产 API 面前不瞎来"这件事写成了可复用的协议：脚本是黑盒、agent 只做业务决策、枚举来自接口、字段必须可溯源、查不到就跳过、可选组件不主动加、绝对值和相对调整解耦、写完必须回查、fail-closed 而不是 fail-open。

这九条放在一起，构成了一份相当完整的"如何写一份让 LLM 调真接口而不出事"的参考。即使不接腾讯广告这个具体业务，把这七个 SKILL.md 当作 skill 设计的对照样本来看，仍然有可抄走的细节。

SKILL.md frontmatter 里声明的版本号告诉读者一件事：这套设计还在演化，部分能力的实装在追赶文档。值得持续观察，但当生产工具直接押注，时机还没到。
