JVM 的内存模型与线程

Java 内存模型（Java Memory Model，JMM）是理解 Java 并发的地基。它回答的不是“CPU 缓存长什么样”，也不是“JVM 堆、栈、方法区如何分布”，而是一个更关键的问题：当多个线程读写共享变量时，哪些结果是被 Java 语言允许观察到的，哪些推理是成立的。

这篇文章只做一件事：把 JMM、happens-before、volatile、final、synchronized 的语言语义讲清楚，并把它们和硬件直觉、JVM 实现细节区分开。对象头、锁升级、Monitor 实现、AQS、线程池、结构化并发都很重要，但它们不拥有本文的定义权。

如果你读完之后能稳定回答下面四个问题，这篇文章的目标就达到了：

  1. 为什么单线程看起来正确的代码，在多线程下会失效？
  2. happens-before 到底是什么，它能保证什么，不能保证什么？
  3. volatile、final、synchronized 各自提供什么语义边界？
  4. 为什么“看起来像缓存刷新”的口语模型，不能直接代替 JLS 规范？

一、先把边界说清楚：JMM 到底是什么

1.1 JMM 不是 JVM 运行时内存区域

很多文章会把 JMM 和“堆、栈、方法区、本地方法栈”混在一起讲，但这其实是两个完全不同的问题：

  • JVM 运行时内存区域讨论的是对象、栈帧、类元数据放在哪里。
  • JMM 讨论的是多线程读写共享变量时，程序允许观察到什么结果。

也就是说，前者回答“东西放哪”，后者回答“线程之间怎么看到彼此的修改”。这两个主题有关，但不是一回事。

1.2 JMM 也不是 CPU 缓存结构图

JMM 里常见的“主内存 / 工作内存”是一种规范层抽象，不是对 L1/L2/L3 Cache、寄存器或 store buffer 的逐项映射。它的目的不是帮你画硬件图，而是提供一个可推理的语言模型，让你知道什么时候一个线程的写对另一个线程必须可见。

因此，下面这两种说法都不够准确：

  1. “JMM 就是主内存和工作内存之间的 flush / reload 过程。”
  2. “JMM 就是 CPU 缓存一致性协议的 Java 版本。”

这两种说法都只能算帮助入门的直觉，不是规范本身。

1.3 JMM 真正解决的问题

JMM 的核心任务，是在编译器优化、处理器乱序执行、缓存可见性延迟都真实存在的前提下，给程序员一套可以稳定推理的并发语义。

它要解决的就是三件事：

  • 原子性：一个操作会不会被“拆开看见”？
  • 可见性：一个线程的写，另一个线程什么时候能看到？
  • 有序性：两个操作在别的线程眼里，顺序会不会改变？

JLS 对这些问题的回答，不是“帮你关掉所有优化”，而是“允许优化存在，但必须在同步边界上满足指定语义”。

二、为什么单线程正确，多线程却可能出错

2.1 性能来自分层缓存和乱序执行

现代处理器和主内存之间存在巨大的延迟鸿沟。处理器为此引入了多级缓存、写缓冲、乱序执行、流水线优化；编译器和 JIT 也会进行重排序，只要单线程观察结果不变，这些优化就是合法的。

这就是经典的 as-if-serial 语义：

在单线程程序中，只要最终结果与按源码顺序执行一致，编译器和处理器就可以自由重排内部步骤。

问题在于，多线程程序并不只关心“本线程最后算出来是不是对的”，还关心“另一个线程能不能在正确的时机看到我的写入”。

2.2 硬件的一致性，不等于语言级可推理性

以 MESI 一类缓存一致性协议为例，它解决的是“多个处理器缓存如何保持一致”的问题。但这并不自动等于：

  • 你的 Java 源码顺序会原样暴露给其他线程；
  • 没有同步的共享变量访问也能稳定推理；
  • 编译器、JIT、运行时都不会做影响观察结果的优化。

所以，硬件一致性是背景，JMM 才是程序员真正依赖的契约。

2.3 一个典型的失效例子

单看源码，很多人会自然地以为：既然 ready = true 写在后面，另一个线程一旦看到 ready == true，就一定也能看到 value == 42。

但如果这里没有同步边界，这个推理并不成立。原因不是“CPU 坏了”，而是 JMM 没有为这两个线程之间建立 happens-before 关系。一旦没有这条边，另一个线程就没有义务看到你希望它看到的顺序。

三、JMM 的核心抽象：共享变量、动作和数据竞争

3.1 JMM 关注的是共享变量

JMM 关注的是多线程之间可能共享的数据，包括：

  • 实例字段
  • 静态字段
  • 数组元素

局部变量、方法参数、线程私有栈帧里的中间值，本身不属于 JMM 争议的中心，因为它们天然不在线程之间共享。

3.2 “主内存 / 工作内存”是一套抽象动作模型

JLS §17.4.1 里使用了 8 个抽象动作来描述变量如何在主内存与线程可见状态之间流动：

动作 含义
read 从主内存读取变量值
load 将读取值放入线程可用状态
use 把值交给执行引擎使用
assign 将执行结果写回线程可用状态
store 将值准备写回主内存
write 真正写入主内存
lock 获取变量相关的独占语义
unlock 释放独占语义

这组动作的重要性不在于“你能不能把它们映射到某条机器指令”，而在于它们告诉你：Java 规范从一开始就在把线程可见性问题看成一个抽象语义问题，而不是具体 CPU 指令清单。

3.3 数据竞争：为什么“没同步”是危险信号

当两个线程并发访问同一个变量，且至少有一个是写，并且这些访问之间没有足够的同步关系时，就会出现 data race（数据竞争）。

数据竞争的危险，不是“程序一定立刻炸掉”，而是：

  • 结果可能依赖运行时机；
  • 某些执行结果在测试里很难复现；
  • 你的直觉推理会失去依据。

3.4 DRF-SC：正确同步的程序，看起来像顺序一致

JMM 的一个核心结论是：

如果一个程序是 data-race-free 的，那么它的执行结果可以被视为是**顺序一致（sequentially consistent）**的。

这句话非常关键。它意味着：

  • JMM 并不承诺“所有并发程序都像顺序执行”；
  • JMM 承诺的是“正确同步的并发程序，可以按接近顺序执行的方式推理”。

这也是为什么并发编程最重要的第一步，从来不是“背多少 API”，而是先判断：我的共享变量之间，到底有没有建立足够的同步边界。

四、happens-before：JMM 的推理主轴

4.1 happens-before 不是“真实时间先后”

happens-before 是 JMM 中的一个偏序关系。如果操作 A happens-before 操作 B，含义是：

  1. A 的结果对 B 可见；
  2. A 在语义上先于 B。

但它不是：

  • 物理世界里的绝对先后；
  • CPU 时钟上的先后；
  • “A 先跑完，B 才开始”的简单口语版描述。

更准确地说，happens-before 是一种“你可以放心据此做可见性推理”的关系。

4.2 happens-before 的八条常用规则

规则 内容
程序次序规则 单线程中，前面的操作 happens-before 后面的操作
监视器锁定规则 对同一把锁的 unlock happens-before 后续 lock
volatile 规则 对同一 volatile 变量的写 happens-before 后续读
线程启动规则 Thread.start() happens-before 被启动线程中的动作
线程终止规则 线程内所有操作 happens-before 其他线程检测到其终止
线程中断规则 interrupt() happens-before 目标线程检测到中断
对象终结规则 构造函数结束 happens-before finalize() 开始
传递性 A happens-before B，B happens-before C，则 A happens-before C

真正写代码时，最常用的是前五条；理解 interrupt、join、start 的语义时，后面几条也非常重要。

4.3 用一条 happens-before 链推导可见性

推导链如下：

  • A happens-before B（程序次序规则）
  • B happens-before C（volatile 规则）
  • C happens-before D（程序次序规则）
  • 因此 A happens-before D（传递性）

所以，一旦线程 B 看到 ready == true，它就必须能看到 value == 42。

这里最重要的不是“volatile 像不像缓存刷新”，而是：你已经建立了一条完整的 happens-before 链。

4.4 没有 happens-before，就不要替自己脑补

如果上面的 ready 不是 volatile，也没有锁或其他同步边界，那么“看到 ready == true 就一定看到 value == 42”这个推理就站不住。

JMM 并不会替你脑补一个同步关系。它只承认你明确建立出来的边。

五、三个关键保证：原子性、可见性、有序性

5.1 原子性：哪些操作不会被“拆开看见”

JMM 保证某些基本读写具备原子性，但这不等于所有你写下的表达式都是原子的。

另一个容易被忽略的规范点是：**从 JLS 规范层面看，非 volatile 的 long / double 仍允许被拆成两个 32 位操作处理。**很多现代 HotSpot 实现通常会做得更强，但实现更强不等于规范已经改写。

5.2 可见性：一个线程的写，另一个线程什么时候必须看到

可见性靠的不是“好运气”，而是同步边界。Java 里最常见的三种可见性来源是：

机制 能力
volatile 对该变量建立写→读的可见性边
synchronized 通过 unlock -> lock 建立可见性与互斥
final 对正确构造且未逸出的对象提供初始化安全

5.3 有序性：允许重排，但不能穿透同步边界

JMM 不是全面禁止重排序，而是要求：只要你通过同步原语建立了 happens-before，重排序就不能破坏这条关系对外的观察结果。

因此，讨论有序性时最好保持两个层次：

  1. 规范层：哪些观察结果被允许；
  2. 实现层：编译器和处理器可能如何通过屏障、指令选择、锁语义来实现它。

内存屏障是理解实现的好工具，但它属于实现手段，不是你写并发代码时唯一可信赖的推理入口。

六、三个关键工具：volatile、final、synchronized

6.1 volatile：可见性与有序性，不提供互斥

volatile 最核心的语义只有两条：

  1. 对同一变量的 volatile 写 happens-before 后续的 volatile 读；
  2. 它能阻止部分会破坏这条边的重排序。

因此 volatile 非常适合：

  • 状态位
  • 一次写、多次读的发布信号
  • “最后写一个标志、最先读这个标志”的守卫模式

但它不适合：

  • i++ 这类复合读改写操作
  • 保护多个变量共同组成的不变式
  • 需要互斥进入临界区的场景

上面这种“一个线程发信号、另一个线程观察信号”的场景，是 volatile 的典型用法。

6.2 final：初始化安全，不等于“永远线程安全”

final 最重要的并发价值，不是“它不可变”，而是它提供了初始化安全（initialization safety）：

  • 如果对象在构造过程中没有让 this 逸出；
  • 并且构造完成后再把引用发布出去；

那么其他线程一旦拿到这个对象引用，就能正确看到这些 final 字段在构造器中写入的值。

但 final 也有边界：

  • 它主要保护的是构造期间写入的 final 字段；
  • 它不自动让整个对象图都线程安全；
  • 如果构造期间 this 提前发布，初始化安全就可能被破坏。

6.3 synchronized：互斥 + 可见性 + 有序性

synchronized 的语义比 volatile 更强，因为它同时提供：

  • 互斥：同一时刻只有一个线程进入受同一监视器保护的临界区；
  • 可见性：前一个持有者的写，对后一个持有者可见；
  • 有序性：不能跨越锁边界破坏对外可见结果。

这里真正被锁住的不是“代码块本身”，而是对象监视器（monitor）。实例同步方法锁的是 this；静态同步方法锁的是对应的 Class 对象。

关于对象头、Mark Word、锁膨胀、偏向锁历史、Monitor 细节，这些都属于实现与成本问题，应该放到《线程安全与锁优化》里展开，而不是在本文里和 JMM 语义混在一起。

7.1 volatile 发布

7.2 锁保护的共享状态

这里的关键不是“锁让线程排队”，而是：对同一把锁的 unlock -> lock 建立了可靠的可见性边。

7.3 双重检查锁定为什么必须配合 volatile

new Singleton() 在实现层并不是一个不可再分的神秘动作。没有 volatile 时，其他线程理论上可能先看到“引用已经写入”，却还没看到完整初始化后的对象状态。volatile 在这里的价值，不是互斥，而是禁止破坏发布语义的重排序，并建立可见性边。

7.4 线程启动与线程终止也是同步边界

八、JMM 与 java.util.concurrent 的关系

JMM 不是只服务于 volatile 和 synchronized。java.util.concurrent 包里的很多抽象，底层都在利用 JMM 的可见性规则。

官方文档明确给出过一条很重要的内存一致性约束：

一个线程在把对象放入并发集合或提交给并发组件之前的操作，对另一个线程成功获取该对象后的操作可见。

这类规则解释了为什么：

  • 把任务提交给线程池后，工作线程能看到提交前构造好的任务状态；
  • Future.get() 返回之后，调用线程能看到任务线程完成的写入；
  • 并发容器和同步器不仅是“方便 API”，也是“带内存语义的并发边界”。

但这些属于“JMM 如何被上层并发库使用”。线程池参数、FutureTask 状态流转、AQS、结构化并发，应该分别在它们自己的文章中展开。

九、最容易讲错的地方

下面这些表述，最好直接视为危险信号：

  1. “JMM 就是 JVM 堆栈模型。”
  2. “happens-before 就是时间上先执行。”
  3. “volatile 保证原子性。”
  4. “synchronized 锁的是代码块。”
  5. “long/double 在 Java 里天然总是原子读写。”
  6. “只要 CPU 缓存一致，就不用考虑 JMM。”
  7. “看到一个线程已经把标志位改了，就一定能看到它之前所有普通写。”——除非这条标志位本身建立了同步边。

并发文章最常见的失败，不是信息少，而是把规范、硬件、实现、经验四个层次混在一起说，最后每句话都“有点像对的”，但连起来没法稳定推理。
