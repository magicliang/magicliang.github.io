Java 结构化并发

结构化并发（Structured Concurrency）不是又一个“并发工具类大全”话题，而是在回答一个更根本的问题：当一个任务拆成多个并发子任务时，这些子任务的生命周期、失败路径、取消路径和可观测关系，能不能像单线程里的函数调用一样清晰。

如果没有这层结构，代码虽然也能并发跑起来，但任务之间的亲属关系只存在于程序员脑子里：谁替谁工作、谁失败后该取消谁、谁该等谁结束，都要手工拼出来。一旦超时、失败或中断发生，线程泄漏、取消延迟和错误收束混乱就会一起冒出来。

所以本文只聚焦一条主线：作用域化任务树。你会看到：为什么 ExecutorService + Future 的非结构化模型容易把收尾逻辑写散；结构化并发怎样把 fork、join、取消和收束限定在一个词法作用域里；它和虚拟线程、线程池、CompletableFuture 分别是什么关系；以及 JDK 19 到 JDK 26 的 API 为什么会一路演进。

一、为什么 Java 需要结构化并发

1.1 问题不在“能不能并发”，而在“并发之后怎么收尾”

  1. findUser() 失败时，fetchOrder() 谁来取消？
  2. 调用方线程被中断时，两个子任务怎么感知？
  3. 一个子任务已经没意义了，另一个还在阻塞 I/O，谁来回收？
  4. 线上线程 dump 里，怎么一眼看出这两个线程是同一个请求拆出来的？

这就是结构化并发要解决的核心痛点：不是把“并发执行”做出来，而是把“并发关系”做实。

1.2 非结构化并发为什么容易出问题

ExecutorService 很强大，但它的设计目标不是表达任务树，而是提供一个通用执行器。它允许：

  • 一个线程创建执行器；
  • 另一个线程提交任务；
  • 第三个线程等待结果；
  • 任务和等待者之间没有强制父子关系。

这类自由度在很多场景下是优点，但它也意味着：运行时不会替你维护任务之间的亲缘关系。

于是，一旦你在心里认为“这两个 Future 是同一个请求的子任务”，你就已经在手工维护一棵逻辑任务树了，只是 API 本身没有把这棵树显式建出来。

这里最容易被低估的一点是：逻辑上的父子关系如果没有落到运行时模型里，就会退化成一堆分散在异常处理、finally、日志、监控和超时回调里的“约定俗成”。

也就是说，非结构化并发真正麻烦的地方不只是代码长，而是：

  • 成功路径通常很短，看起来甚至挺优雅；
  • 一旦进入失败、超时或取消分支，真正的任务关系却散落在多个位置；
  • 你以为自己写的是“一次请求拆两个子任务”，运行时看到的却只是“两个互不相干的异步任务”。

这就是为什么很多旧代码在 happy path 上没有任何问题，但只要线上进入超时、重试、下游局部失败，就开始暴露线程泄漏、取消延迟和错误收束混乱。问题并不一定出在单个 API 用错，而是出在任务树只存在于人的脑内，没有存在于程序结构里。

1.3 结构化并发的核心类比

Martin Sústrik 提出了 structured concurrency 这个术语，Nathaniel J. Smith 则把它讲得更形象：很多传统并发原语之于并发控制流，类似 goto 之于顺序控制流。

这个类比的意思不是“goroutine、thread、future 都不好”，而是：

  • 如果并发子任务可以随意启动、随意逃逸、随意由别的线程收尾；
  • 那么代码的结构就无法真实反映任务的生命周期结构。

而结构化并发想恢复的，正是这种对应关系：

如果一个任务拆出多个并发子任务，那么这些子任务也应该回到同一个代码块里完成收束。

二、结构化并发到底在约束什么

2.1 词法作用域就是并发边界

结构化并发最重要的原则是：

子任务的生命周期不能超过父任务中启动它们的那个词法作用域。

这意味着父任务在离开该作用域之前，必须把以下事情做完：

  • 该 fork 的都 fork 完；
  • 该等待的都 join 完；
  • 该处理的异常已经处理；
  • 该取消的子任务已经被取消并等待结束。

如果你把这条原则和普通函数调用类比，会更容易理解：函数调用的子过程不能“偷偷活到调用者返回之后”再决定要不要把异常扔回来。结构化并发只是把这种边界控制，从单线程调用栈扩展到了多线程任务树。

2.2 结构化并发不是“自动并发”，而是“自动收束”

很多人第一次看结构化并发，会把注意力放在 fork() 上。但真正值钱的不是“能 fork”，而是：

  • join 是成组发生的，不是散落在各个 Future 上；
  • 异常是按作用域收束的，不是每个任务各抛各的；
  • 取消是按任务树传播的，不是调用方自己追着 Future 调 cancel()；
  • close 必须等待子任务结束，不能留下后台线程继续游荡。

从工程角度看，它更像一种“收尾协议”，而不是单纯的并发启动协议。

2.3 三个直接收益：可靠性、可读性、可观测性

结构化并发最直接的收益有三个：

  1. 可靠性：减少线程泄漏、取消延迟、失败后遗留任务。
  2. 可读性：代码块本身就表达了哪些任务属于同一组。
  3. 可观测性：线程 dump 和运行时层面可以重建任务树，而不是只看到一堆彼此无关的线程。

这三者其实是同一件事的不同表现：运行时终于知道“哪些线程是在替同一个父任务工作”。

三、Java 里的核心抽象：StructuredTaskScope

3.1 它回答的不是“在哪执行”，而是“如何成组收束”

在 Java 里，结构化并发的核心 API 是 StructuredTaskScope。它不是通用线程池，也不是异步编排 DSL，而是一个并发子任务作用域。

它关注的是：

  • 在一个作用域里 fork 多个子任务；
  • 以一个整体 join 这些子任务；
  • 根据成功、失败、超时等策略收束结果；
  • 在退出作用域前确保剩余子任务都结束。

3.3 fork、join、close 三件事不能拆开理解

结构化并发最容易被误读的一点，是把它当成“比线程池更优雅的 submit”。这不对。

它真正的契约是这三步必须构成闭环：

  1. fork：把子任务纳入当前作用域；
  2. join：把子任务作为一个整体等待或收束；
  3. close：保证作用域退出前没有未结束的子任务遗留。

如果只看 fork()，那它和普通“起一个线程”没有本质区别。真正让结构成立的，是 join + close 把任务生命周期锁回了词法作用域里。

3.4 结构化使用是被运行时约束的

Java 的这个 API 不是“建议你这样用”，而是会在运行时强制一些结构规则，例如：

  • scope 有 owner thread；
  • fork / join / close 只能由 owner thread 调用；
  • fork 完子任务后必须 join；
  • 作用域提前退出时会触发异常，而不是默默放过。

这点非常重要。结构化并发的价值恰恰来自“少给你自由”，因为那些自由通常会变成资源泄漏和混乱收尾。

如果换个角度看，这其实就是结构化并发和许多旧式并发工具最大的哲学差异：

  • 旧模型更像“给你足够多原语，你自己拼”；
  • 结构化并发更像“先限制任务树必须长成什么样，再在这个边界内并发”。

这种设计刚接触时会让人觉得它“没那么灵活”，但工程经验恰恰说明：在请求级 fan-out/fan-in 场景里，少一点随意逃逸的灵活性，往往正是可靠性的来源。

四、错误收束：为什么它比“多个 Future 各自抛异常”更好用

4.1 结构化并发强调的是作用域级 outcome

在非结构化模型里，每个 Future 都有自己的成功或失败状态，调用方需要自行决定：

  • 先等哪个；
  • 谁失败了要不要取消别人；
  • 最终向上抛哪个异常；
  • 其他异常要不要吞掉、记录、聚合。

结构化并发把问题换了个角度：子任务当然各自有结果，但父任务首先关心的是“这个作用域最终是成功还是失败”。

也就是说，它优先让你处理 scope-level outcome，而不是把你先扔进一堆离散 Future 状态里。

4.2 默认策略：全部成功，否则失败

默认 open() 的语义就是最常见的那一类：

  • 所有子任务都成功，join() 正常返回；
  • 任一子任务失败，作用域失败，并尽快取消其他未完成子任务。

这很适合典型的聚合式调用：用户信息、订单信息、库存信息都必须拿到，少一个都不能继续。

4.3 “短路”是这里最有工程价值的设计

结构化并发里非常关键的一点是 short-circuiting：

  • 一个任务已经失败，其他任务的结果就不再有意义；
  • 一个任务已经先成功拿到可用答案，其他备用任务就没必要继续跑。

这种“结果已定，其他任务尽快停下”的语义，在旧的 Future 心智模型里往往要手工补，而在结构化并发里它是第一等公民。

4.4 它不是 Python 式“异常组总论”

需要特别分清的一点是：Java 结构化并发强调的是作用域结果收束，而不是像某些语言那样把多个异常完整打包成一个异常组作为主叙事。

在 Java 当前设计里，更核心的思路是：

  • 决定 scope 的成败；
  • 在需要时把某个失败原因作为主异常向上传递；
  • 其他任务则配合取消和收尾。

所以这里的重点是 single outcome for the scope，而不是“把所有子任务错误都变成并列主角”。

五、取消传播：真正落地仍然依赖中断

5.1 结构化并发没有发明新的取消机制

这是最容易被说大的地方之一。结构化并发并没有创造一个全新的、神奇的取消系统。OpenJDK 的 JEP 也明确说了：

它不是要替换现有的 thread interruption 机制。

也就是说，结构化并发提供的是：

  • 取消关系的组织方式；
  • 取消信号的传播路径；

而取消真正落地到子任务线程时，仍然主要体现为：中断（interrupt）。

5.2 取消传播链条是什么样的

在结构化作用域里，常见的取消来源有三种：

  1. 某个子任务失败，触发作用域短路；
  2. 作用域 owner thread 被中断；
  3. 作用域超时或按策略被取消。

随后发生的事情是：

  • 作用域进入取消状态；
  • 尚未完成的子任务线程收到中断；
  • join() 被唤醒或转入失败/超时结果；
  • close() 等待尚未停下的子任务真正结束。

这就是为什么我们说：scope-level cancellation 最终还是通过 thread interruption 落地。

5.3 这也意味着子任务必须“合作式响应”

结构化并发不会把不可中断阻塞 magically 变成可中断。若子任务：

  • 长时间阻塞在不可中断 I/O；
  • 吞掉中断后继续死循环；
  • 把 InterruptedException 吃掉却不恢复中断状态；

那么作用域的关闭仍然会被拖慢，甚至无限拖住。

所以结构化并发并不意味着“取消一定立刻成功”，它只是把取消意图的传播关系做对了。至于响应速度，仍然取决于任务代码是否遵守中断协作规则。这一点应和《Java 并发编程笔记》里的中断章节一起理解。

5.4 timeout、deadline 不应在本文里展开成总论

结构化并发当然能和超时配合，但这里最好把边界说清：

  • 结构化并发负责的是作用域内任务如何一起收束；
  • timeout / deadline 负责的是时间约束如何触发取消或失败。

JDK 25/26 的 API 已支持在 open(...) 时配置 timeout；更早的预览 API 则常见 joinUntil(...)。但这些都只是在结构化作用域上挂接时间约束，并不等于“结构化并发 = 超时机制总论”。

timeout、deadline、cancellation、interruption 之间的统一语义，应放到《跨语言超时机制全解析》里集中讲。

六、与虚拟线程的关系：互补，而不是互相替代

6.1 虚拟线程解决“线程贵不贵”，结构化并发解决“线程乱不乱”

这两个能力经常一起出现，但解决的是两类不同问题：

  • 虚拟线程回答：我能不能很便宜地为每个并发任务分配一个线程？
  • 结构化并发回答：这些线程之间的父子关系、取消关系和收尾关系能不能被正确表达？

一句话概括就是：

虚拟线程提供充足线程资源，结构化并发提供可管理的任务结构。

6.2 为什么它们天然搭配

结构化并发最适合的，就是那种“一个请求拆多个 I/O 子任务并发跑”的场景。而这种场景恰好也是虚拟线程最擅长的：

  • 子任务数量可能多；
  • 大量等待 I/O；
  • 希望保留阻塞式编程模型，而不是把代码写成层层回调或异步链。

所以在 Java 的实际叙事里，经常是：

  • 一个请求由一个虚拟线程承载；
  • 该请求内部用 StructuredTaskScope 再 fork 若干虚拟线程；
  • 最终形成一棵可观察、可取消、可收束的任务树。

6.3 但虚拟线程并不会自动带来结构

如果你只是把线程池换成 Executors.newVirtualThreadPerTaskExecutor()，然后继续到处 submit、到处拼 Future，那么你得到的只是“更便宜的非结构化并发”。

线程便宜了，不代表关系就清楚了。

因此，虚拟线程与结构化并发是互补关系，不是替代关系，更不是“用了虚拟线程就不需要结构化并发”。

6.4 pinned 问题也不会因为结构化并发自动消失

如果虚拟线程因为同步块、native 调用或不可挂起阻塞而发生 pinned，结构化并发本身并不会替你修好这个问题。它只能让任务树更清楚，不会改写底层阻塞行为。

所以把结构化并发和虚拟线程配合使用时，仍然要保留对 pinned、阻塞点和中断响应性的基本判断。

七、与线程池、CompletableFuture 的关系：层级不同，不是简单替代

7.1 它不是 ExecutorService 的新版皮肤

OpenJDK 明确把这件事说得很清楚：结构化并发不是为了替换 java.util.concurrent 里已有的所有并发构件。

这意味着它和线程池的关系，不是“旧 API vs 新 API”，而是“不同层级的工具”：

  • 线程池关注共享执行资源如何治理；
  • 结构化并发关注一次任务拆分后的子任务关系如何治理。

因此，线程池更像资源层，结构化并发更像任务作用域层。

这组区分看起来抽象，但如果把问题改写成工程语言就会很直观：

  • 当你在问“线程够不够、队列会不会堆爆、拒绝策略怎么配”，你处理的是资源池问题；
  • 当你在问“这组并发子任务是不是必须一起结束、一个失败后另外几个该不该停”，你处理的是任务树问题。

两者当然会相遇，但相遇不等于替代。把结构化并发理解成“线程池 2.0”，会让读者天然期待它回答线程复用、队列、吞吐治理这些问题；而它其实并不打算负责这些。

7.2 什么时候更像线程池的问题，什么时候更像结构化并发的问题

可以用一个简单判断来区分：

  • 如果你在想“线程数多少、队列怎么排、拒绝策略是什么”，这更像线程池问题；
  • 如果你在想“这几个并发子任务是不是同一个父请求的一部分，失败后该不该一起停”，这更像结构化并发问题。

很多服务端 fan-out/fan-in 请求，其实后者更核心。

7.3 它也不等于 CompletableFuture

CompletableFuture 解决的是异步阶段图编排：结果如何 thenApply、thenCompose、allOf、anyOf。结构化并发解决的是阻塞式子任务树收束：任务在哪里 fork，在哪里 join，如何以一个作用域为单位失败或取消。

它们最大的差别不是“谁更高级”，而是编程模型不同：

  • CompletableFuture 更偏阶段图和异步回调拼接；
  • 结构化并发更偏阻塞式、顺序读写风格下的并发拆分。

再说得更透一点：它们的“默认阅读方向”都不一样。

  • 读 CompletableFuture 代码时，你通常顺着阶段依赖图往后看：前一个 stage 完成后，下一个 stage 怎么接；
  • 读结构化并发代码时，你更像在看一个局部调用栈被横向展开：这里 fork 了几个子任务，随后在同一个作用域 join 回来。

这就是为什么两者都能表达“并发”，却很难互相替代。前者擅长异步编排，后者擅长把本来就属于同一作用域的一组子任务写得像同步逻辑一样可读。

7.4 一个更准确的定位

因此更准确的表述应是：

  • 线程池：资源执行模型；
  • CompletableFuture：异步编排模型；
  • 结构化并发：作用域化任务树模型。

它们可以协同，但不该混成一个“并发万能 API 排行榜”。

八、内存语义与上下文传播：结构化并发不是脱离 JMM 的魔法

8.1 结构化并发也有明确的 happens-before 边界

JDK 文档明确给出了 StructuredTaskScope 的内存一致性语义，大意可以概括为：

  • owner thread 在 fork 子任务之前的操作，happens-before 子任务线程中的操作；
  • 子任务线程中的操作，happens-before owner thread 成功通过 Subtask.get()、Subtask.exception() 或 join() 获取结果之后的操作。

这很重要，因为它说明结构化并发并不是绕过 JMM，而是在 JMM 之上建立了更具体的并发边界。

8.2 这类语义的价值是什么

它的工程意义在于：

  • 父任务在 fork 前准备好的上下文，对子任务可见；
  • 子任务完成后写出的结果，对父任务后续处理可见；
  • 你不用自己再围绕“这些结果什么时候可见”额外拼装一层同步协议。

当然，这并不意味着你就可以无视共享可变状态。JMM 的基础定义权仍然在《JVM 的内存模型与线程》那篇文章里；这里只强调结构化并发把哪些 happens-before 边显式提供了出来。

8.3 ScopedValue 让上下文传递更自然

结构化并发与 ScopedValue 的配合也非常自然：父任务当前线程上的绑定值，会被作用域内 fork 出去的子任务继承。

这件事的意义，不只是“少传几个参数”，而是：

  • 请求上下文、租户信息、trace id 等作用域型上下文，终于可以和任务树一起传播；
  • 不需要像 ThreadLocal 那样再担心线程复用导致的污染模型；
  • 顺序代码向并发 fan-out 重构时，调用上下文更容易保持一致。

所以在 Java 的新并发叙事里，虚拟线程、结构化并发、ScopedValue 其实是在共同组成一套更顺手的服务端并发模型。

九、API 演进：为什么这篇文章不能只写某一个版本的代码

9.1 时间线先拉直

Java 结构化并发从 JDK 19 开始一路演进：

JEP JDK 阶段 关键信息
JEP 428 19 Incubator 首次引入结构化并发
JEP 437 20 Second Incubator 继续孵化并适配 ScopedValue 继承
JEP 453 21 First Preview fork() 返回 Subtask，不再返回 Future
JEP 462 22 Second Preview 继续预览
JEP 480 23 Third Preview 无实质 API 改动
JEP 499 24 Fourth Preview 无实质 API 改动
JEP 505 25 Fifth Preview 重大 API 重构：open(...) + Joiner
JEP 525 26 Sixth Preview 在新模型上继续细化

截至 JDK 26，它仍然是 preview API，并未正式转正。

9.2 最大的分水岭发生在 JDK 25

前几轮预览里，大家更熟悉的是这样的写法：

  • new StructuredTaskScope.ShutdownOnFailure()
  • new StructuredTaskScope.ShutdownOnSuccess<>()
  • join() / joinUntil(...)
  • 通过子类或 shutdown policy 来表达收束策略

JDK 25 开始，主入口变成：

  • StructuredTaskScope.open()
  • StructuredTaskScope.open(joiner)
  • StructuredTaskScope.open(joiner, configOperator)

也就是说，策略从“继承某个预设子类”转向了“通过 Joiner 组合出作用域结果”。

9.3 为什么这次重构很关键

这次重构的意义在于，Java 团队开始更明确地把重点放在：

  • scope 的 outcome 是什么；
  • join 返回什么；
  • 何时 short-circuit；
  • timeout 如何作为配置进入作用域；
  • 自定义策略如何通过 Joiner 表达。

换句话说，API 从“几个看起来很好懂的预设类”演进到“更抽象但更稳定的作用域结果模型”。

9.4 写文章时应该如何处理版本差异

因此，面对结构化并发这个主题，最稳妥的写法不是死盯某一版示例代码，而是分成两层：

  1. 概念层保持稳定：作用域、取消传播、失败短路、可观测性、和虚拟线程的关系。
  2. API 层注明版本：JDK 21-24 是旧预览接口，JDK 25-26 是 open + Joiner 模型。

这样读者不会把历史 API 当成“现行定稿版”，也不会因为 API 换皮就误以为核心思想变了。

十、典型使用场景：它最擅长什么，不擅长什么

10.1 最适合：请求内 fan-out / fan-in

结构化并发最擅长的是这种场景：

  • 一个请求进入系统；
  • 需要并行访问多个下游；
  • 所有结果要在当前请求作用域内汇总；
  • 任意失败、超时或上游取消都应触发整体收尾。

例如：

  • 用户主页聚合多个下游服务；
  • 搜索请求并发查索引、缓存、画像服务；
  • 从多个冗余数据源 race 一个最快成功结果。

10.2 也适合：层级明确的子任务树

除了典型聚合式 I/O，它也适合那些天然是任务树的问题：

  • 一层任务再拆二层子任务；
  • 各层都希望保留清晰的父子边界；
  • 每层都想把取消和失败关在自己的词法作用域内。

这也是为什么它和 ScopedValue、虚拟线程放在一起时特别顺手。

10.3 不适合：长期存在的共享工作池

如果你的需求本质上是：

  • 维护一个长期存在的后台执行资源池；
  • 接受来自许多不相关调用方的任务；
  • 重点是队列、限流、拒绝策略、隔离和吞吐治理；

那么这更像线程池问题，而不是结构化并发问题。

结构化并发不应该被误用成“把所有后台任务都包进一个 scope”——那样反而会和它强调的词法边界相冲突。

10.4 也不负责流式通道通信

OpenJDK 也明确把这一点列为非目标：结构化并发不是 channels / streaming between threads 的总方案。它处理的是任务关系，不是线程之间持续流式交换数据的全部模型。

十一、实践中的几个高频误区

11.1 误区一：用了虚拟线程，就不需要结构化并发

错。虚拟线程解决的是线程成本，不解决任务关系。没有结构，廉价线程照样可以泄漏、照样可以收尾混乱。

11.2 误区二：结构化并发会自动杀死所有慢任务

错。它传播取消和中断，但不能保证不可中断阻塞立刻结束。子任务仍然需要合作式响应中断。

11.3 误区三：它会彻底替代线程池和 CompletableFuture

错。它们解决的问题层次不同：结构化并发不是线程资源治理器，也不是异步阶段图总框架。

11.4 误区四：API 长这样，概念就只属于这个版本

错。JDK 21-24 和 JDK 25-26 的 API 入口差异很大，但真正稳定的概念主线一直没变：词法作用域、短路收束、取消传播、可观测任务树。

11.5 误区五：有了作用域，JMM 就不用管了

错。结构化并发只是提供更明确的并发边界，不会推翻 Java 内存模型。共享可变状态的正确性判断，仍然要回到 happens-before、可见性和同步语义上。

十二、这篇文章与系列其他文章的分工

  • 本文负责：结构化并发的动机、作用域、失败短路、取消传播、可观测性、与虚拟线程/线程池/CompletableFuture 的关系，以及 JDK 19-26 的 API 演进。
  • 《Java 并发编程笔记》负责：线程状态、wait/notify、park/unpark、interrupt、join、虚拟线程 pinned 等线程原语与生命周期问题。
  • 《Java 线程池笔记》负责：Executor、FutureTask、ThreadPoolExecutor、ScheduledThreadPoolExecutor、ForkJoinPool、CompletableFuture 的执行与编排模型。
  • 《JVM 的内存模型与线程》负责：JMM、happens-before、volatile、final、synchronized 的语言语义。
  • 《跨语言超时机制全解析》负责：timeout、deadline、cancellation、interruption 的统一时间语义。

边界一旦固定，结构化并发就不再承担“线程池总论”“中断总论”“超时总论”这些职责，而是专注于任务树怎样以作用域为单位正确收尾。
