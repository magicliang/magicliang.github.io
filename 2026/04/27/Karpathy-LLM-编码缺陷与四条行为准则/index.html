LLM 编码工具的"自作主张"问题

如果你长期用 Claude Code、Cursor、Copilot 之类的工具写代码，应该有过这种感觉：模型并不是"不会写"，而是"写得太多、太聪明、太自作主张"。让它修一个 bug，它顺手把无关代码风格也改了；让它加一个简单功能，它给你抽象出三层接口和一套配置系统；让它"重构 X"，它没问清楚 X 的边界就开始动手。

Andrej Karpathy 在 X 上发过一段对这类问题的精炼归纳，forrestchang 把它整理成了一个 GitHub 仓库 andrej-karpathy-skills，提供一份 60 多行的 CLAUDE.md，外加官方 Claude Code Plugin 的安装方式。这个仓库的内容可以同时用于 Claude Code、Cursor 以及任何支持 CLAUDE.md 形态的 LLM 编码工具。

这篇文章做三件事：

  1. 把仓库背后的诊断（LLM 编码到底"差"在哪四个地方）拆开讲清楚
  2. 把对应的四条准则（Think Before Coding / Simplicity First / Surgical Changes / Goal-Driven Execution）逐条解析，给出可被 LLM 引用的判定语
  3. 把这套准则在本地的工程化落地路径走通：作为 skill 安装到 ~/.claude 体系或项目级 ./skills/ 目录的实操过程

一、Karpathy 观察到的四个系统性偏差

仓库 README 引用了 Karpathy 的三段原话（见 Karpathy 原帖）。这三段对应四种独立的失效模式：

“The models make wrong assumptions on your behalf and just run along with them without checking. They don’t manage their confusion, don’t seek clarifications, don’t surface inconsistencies, don’t present tradeoffs, don’t push back when they should.”

“They really like to overcomplicate code and APIs, bloat abstractions, don’t clean up dead code… implement a bloated construction over 1000 lines when 100 would do.”

“They still sometimes change/remove comments and code they don’t sufficiently understand as side effects, even if orthogonal to the task.”

偏差一：替用户做隐藏假设

模型遇到歧义时不发问，而是悄悄选一个解释往下走。这不是模型"不愿问"，而是它的训练目标里"给出一个完整答案"的权重远高于"承认我不确定"。

举一个常见例子：用户说"加一个导出用户数据的功能"。可能的歧义至少有：导出全部还是分页？导出哪些字段（隐私敏感）？是浏览器下载文件，还是后台任务+邮件通知，还是 API 返回数据？典型的 LLM 反应是直接挑一个解释（通常是最像 demo 的那个）写出来。

[PATTERN] 歧义被吞掉的本质：模型把"看起来完整的代码"当成最终交付物，而不是把"和用户对齐过的需求"当成最终交付物。

偏差二：偏爱过度抽象

让 LLM 写一个折扣计算函数，它经常会先抽出 DiscountStrategy 抽象基类、PercentageDiscount / FixedDiscount 子类、DiscountConfig 配置 dataclass、DiscountCalculator 上下文，然后告诉你"这样以后扩展新的折扣类型很方便"。

这不是错的——但它解决的是未来可能存在的问题，而代价是现在的代码理解成本翻了 5 倍。仓库里的 EXAMPLES.md 把这种过度抽象的实例和精简版本做了直接对比，对比之下，"简单版"几乎在所有维度都更优：更易读、更易测、更快写完，将来真有第二种折扣类型时再重构也来得及。

[PATTERN] 过度抽象的本质：模型混淆了"经过验证的设计模式是好的"与"在所有场景下都该使用设计模式"。模式是对真实复杂度的解药，不是给一次性代码贴的勋章。

偏差三：改动外溢

被要求修一个 bug，顺手把附近的代码风格、引号种类、变量命名、缺失的类型注解、不规范的 docstring 一并"改进"了。

仓库里给了一个真实案例：用户只是说"修复空 email 导致校验器崩溃的 bug"，模型却同时把 username 长度校验、字母数字校验也加上了，还把变量名、注释、缩进全部"统一"了一遍。代码 review 时这种 PR 最难处理——审阅者无法确定哪些改动是为了修 bug、哪些是顺带"美化"的。

[PATTERN] 改动外溢的本质：模型从训练数据里习得了"良好实践"的偏好，但没有内化"在维护既有代码时，每一行变更都需要被解释"这条工程纪律。

偏差四：目标不可验证

让 LLM “fix the auth system”，它会回答：“好的，我会回顾代码、识别问题、做出改进、测试改动”——然后开始改一堆东西，最后告诉你"应该好了"。

问题在于"应该好了"不是一个可验证的状态。Karpathy 的关键洞察是：

“LLMs are exceptionally good at looping until they meet specific goals… Don’t tell it what to do, give it success criteria and watch it go.”

LLM 在"有明确成功标准"的循环里非常擅长收敛。在"成功标准模糊"的任务里，它要么过早声称完成、要么不停打扰用户要澄清。

[PATTERN] 目标不可验证的本质：把祈使句任务（“修 bug”、“加校验”、“重构 X”）当成动词执行，而不是先把它翻译成可验证的目标（“写一个能复现 bug 的失败测试 → 让它通过”）。

二、四条准则逐条解析

仓库的核心交付物是一份精简的 CLAUDE.md，把上面四种偏差对应到四条可被 LLM 引用的行为准则。这四条准则的措辞非常短，但每一条背后都有非常具体的判定语。

准则 1：Think Before Coding（写代码前先想清楚）

口诀：Don’t assume. Don’t hide confusion. Surface tradeoffs.

落到实操，这一条要求模型在动手前完成四个动作：

  • 显式声明假设：不确定就直接问
  • 多种解释并列展示：歧义存在时不要默默选一个
  • 存在更简单方案就说出来：必要时反驳用户的方案
  • 遇到不清楚的地方就停下：明确说出哪里不清楚，请求澄清

这一条最容易被误读为"凡事都要问用户"。准则的真实要求其实更精确：只在存在真实歧义时问。如果用户说"把这个变量名从 data 改成 payload“，没有歧义，直接做。如果用户说"加个导出功能”，不问就是失职。

[PATTERN] 区分"显式需求"和"隐式假设"的判定标准：你是否能用一句话把当前的行动描述出来，并且这句话只引用用户的原话和代码里已有的事实？做不到，就说明你正在用假设填补空白，必须先把假设说出来。

准则 2：Simplicity First（先求简单）

口诀：Minimum code that solves the problem. Nothing speculative.

具体边界：

  • 不实现没要求的功能
  • 不为单次使用的代码做抽象
  • 不引入没要求的灵活性或可配置性
  • 不为不可能的场景写错误处理
  • 200 行能压成 50 行，就重写

这条准则最有杀伤力的是最后那个反问句：“Would a senior engineer say this is overcomplicated?” If yes, simplify. 这句话之所以有效，是因为它把审美权交给了一个虚拟的资深工程师，而不是模型自己的"代码完整性焦虑"。

需要注意的是，这条准则不反对设计模式，反对的是在没有真实需求驱动时就引入设计模式。当一种新的折扣类型真的出现时，再把简单函数重构成策略模式完全合理。

[PATTERN] 复杂度只有在当下问题已经容纳不下时才有价值。提前抽象等于提前付出维护成本，而且常常付错——因为你不知道未来真正的扩展方向是什么。

准则 3：Surgical Changes（外科手术式修改）

口诀：Touch only what you must. Clean up only your own mess.

落到具体行为：

  • 编辑既有代码时，不"改进"相邻代码、注释、格式
  • 不重构没有坏的东西
  • 匹配既有代码风格，即使你认为有更好的写法
  • 注意到无关的死代码，告诉用户，但不要删
  • 自己的修改产生的孤儿（unused imports / 变量 / 函数）要清理掉
  • 既有的死代码不要主动删（除非用户明确要求）

判定标准非常清晰：每一行改动都应该能直接追溯到用户的请求。如果某一行改动你解释不出来"这是为了满足用户哪句话"，就该回滚。

这条准则反的是模型的训练偏好。训练数据里大量"良好实践"鼓励顺手优化、补类型、加 docstring，这些在写新代码时是优点，在维护代码时是噪声。准则强制模型区分这两种场景。

[PATTERN] 维护既有代码的核心纪律是"小改动 + 高可解释性"。当你想"顺便改一下"时，记住：审阅你 PR 的人没有你的上下文，每一处看起来无关的改动都会让 review 时间和回归风险翻倍。

准则 4：Goal-Driven Execution（目标驱动执行）

口诀：Define success criteria. Loop until verified.

把祈使句任务翻译成可验证目标的对照：

  • “Add validation” → “Write tests for invalid inputs, then make them pass”
  • “Fix the bug” → “Write a test that reproduces it, then make it pass”
  • “Refactor X” → “Ensure tests pass before and after”

对于多步任务，要求陈述一个简短计划，每一步都带验证：

这条是整套准则的灵魂，也直接呼应 Karpathy 的核心洞察：模型在明确成功标准下非常擅长循环。强成功标准让 LLM 可以独立循环，弱成功标准（“make it work”）会导致它不停打扰用户。

这一点对 Claude Code、Cursor 这样的 agent 形态尤其重要。它们的工作模式本来就是"工具调用循环"，循环的退出条件越清晰，agent 的自主性就越高。如果你只说"修好 auth"，agent 不知道何时该停；如果你说"把这个失败测试变绿，且其他测试不能挂"，agent 知道何时该停。

[PATTERN] 给 LLM 任务时，永远问自己一个问题："完成"是一个可以被脚本检查出来的状态，还是一个我需要肉眼判断的状态？ 前者 LLM 能独立完成，后者必然带来反复返工。

三、四条准则之间的关系

这四条不是平行的清单，它们之间有时序和依赖关系。

阶段 起作用的准则 失效后果
接到任务、还没动手 准则 1（Think Before Coding） 跑偏整个任务的方向
决定写多少代码 准则 2（Simplicity First） 写出 5 倍于必要量的代码
编辑既有代码时 准则 3（Surgical Changes） PR 包含大量噪声改动，review 不动
判断"完成"与否 准则 4（Goal-Driven Execution） 过早声称完成，或不停打扰用户

值得注意的是，准则 1 和准则 4 互为镜像：准则 1 在入口处澄清歧义、声明假设，准则 4 在出口处用可验证标准判定完成。中间两条规范执行过程：准则 2 控制写多少代码，准则 3 控制改多少范围。

[PATTERN] 一个完整的 LLM 编码任务可以被建模为：入口澄清（1）→ 量级克制（2）→ 半径克制（3）→ 出口验证（4）。任何一环失守，最终交付物的可维护性都会显著下降。

四、把这套准则装成本地 skill

仓库提供了三种安装方式：Claude Code Plugin、单文件 CLAUDE.md、Cursor 项目规则。但仓库里其实还有一个更隐蔽的形态——skills/karpathy-guidelines/SKILL.md，这是一个标准的 Anthropic 风格 skill，可以直接被任何兼容 skill 协议的工具加载。

下面记录我的本地落地路径：4.1 是位置选择，4.2 是我实际跑的最小命令序列，4.3 是带治理工具链的完整规范流程（首次安装会跳过其中的备份与恢复步骤），4.4 是 SKILL.md 与 DESIGN.md 的分工。

4.1 选择安装层级

skill 有两个常见的安装位置：

  • 项目级：<项目根>/skills/<skill-name>/，只对当前项目生效
  • 用户级（全局）：~/.aone_copilot/skills/<skill-name>/ 或 ~/.claude/skills/<skill-name>/，对所有项目生效

karpathy-guidelines 这种通用编码态度准则两种位置都合理。我选择把它装到一个专门管理 skill / agent / prompt 的本地 git 仓库 claude-configurations 的 skills/ 目录下，这样可以版本化、可同步、可团队共享。

如果你已经有一套本地 skill 治理工具链（比如 skill-installer + generate-skill-doc），流程会更规范：

  1. 检测安装目录：优先 ./skills/（项目级），无则询问用户
  2. 下载新版到临时目录，校验 SKILL.md frontmatter
  3. 备份旧版 DESIGN.md（如有）
  4. 完整清除旧目录（避免残留过期文件）
  5. 复制新版到目标位置
  6. 恢复 DESIGN.md 历史
  7. 触发 generate-skill-doc，生成或更新 DESIGN.md + 版本记录
  8. 清理临时文件，输出安装报告

为什么不"覆盖式更新"而要"清除+复制"？因为 skill 的目录结构会演化（比如新增了 references/ 子目录、删除了某些文件），覆盖式更新可能留下幽灵文件，行为变得难以预测。

4.4 SKILL.md 与 DESIGN.md 的分工

值得专门提一下的是 SKILL.md 和 DESIGN.md 的角色分工：

  • SKILL.md：给 LLM 看的操作指令，会被加载到上下文，token 必须紧凑
  • DESIGN.md：给人看的设计说明，不被 LLM 加载，记录"为什么这么设计"和版本历史

版本号只放在 DESIGN.md 里，不写进 SKILL.md frontmatter。这是为了避免每次升级都让 LLM 上下文里多一行无意义的 token。这个设计来自本地 generate-skill-doc 工具链的约定，但放在任何 skill 治理体系下都是好实践。

[PATTERN] 任何"准则型 skill"都应该考虑这种双文件结构：操作语义（机器读）和设计意图（人读）必须分离，否则二者会互相污染——要么 SKILL.md 被设计讨论灌水变长，要么 DESIGN.md 被压缩成 LLM 友好的清单失去解释力。

五、这套准则不能解决的问题

最后必须澄清边界，否则就违反了准则 1 自己。这套准则不是：

  • 不是代码审查工具：它不会告诉你"这段代码有 N+1 查询"或"这里缺了空指针检查"。代码质量审查应该走专门的 reviewer skill。
  • 不是 lint / 格式化器：它不约束代码风格本身，只约束模型在改动时不要主动改风格。
  • 不是测试框架的替代品：准则 4 强调"写测试先于实现"，但具体怎么写测试、用哪个测试框架，是 TDD 类 skill 的职责。
  • 不是对琐碎任务的强制约束：仓库 README 明确说了 “These guidelines bias toward caution over speed. For trivial tasks, use judgment.”。改个 typo、写一行明显的修复时，硬套四条准则等于自缚手脚。

这一点很关键：准则的价值在非平凡任务上。准则的代价是每次任务多花一些时间澄清和规划，所以平凡任务不值得付这个代价。

六、和已有 skill 生态的关系

如果你的本地 skill 库里已经有一些"编码态度类"skill（比如 test-driven-development、systematic-debugging、verification-before-completion、brainstorming、receiving-code-review、pua-debugging），需要回答一个问题：为什么还要装 karpathy-guidelines？

这些 skill 都是特定阶段触发的：

  • brainstorming 在创意工作之前激活
  • test-driven-development 在写实现代码之前激活
  • systematic-debugging 在遇到 bug 时激活
  • verification-before-completion 在声称完成之前激活
  • pua-debugging 在反复失败 2 次以上时激活

karpathy-guidelines 的覆盖面是整段编码任务期间常驻的态度准则，触发面更宽。具体地：

  • 它的准则 1（Think Before Coding）和 brainstorming 部分重叠，但 brainstorming 是需求阶段，准则 1 是动手前那一刻
  • 它的准则 4（Goal-Driven）和 TDD 重叠，但 TDD 给具体流程，准则 4 给态度理由
  • 它的准则 2（Simplicity First）和准则 3（Surgical Changes）在已有 skill 里没有直接等价物

简单说：准则 2 和准则 3 是 LLM 高频翻车但没被其他 skill 专门覆盖的盲区。这就是为什么这个 skill 值得独立存在，而不是被合并进已有 skill。

[PATTERN] 在判断一个新 skill 是否值得引入时，关键问题不是"它和现有 skill 有重叠吗"，而是"它覆盖的盲区里，有多少是高频翻车点"。准则 2 和准则 3 命中的就是这种盲区。

七、使用建议

基于上面的分析，我对这套准则的使用建议是：

  • 个人使用：直接装 skill 形态。这种"声明式准则"最适合做成 skill，比项目级 CLAUDE.md 更灵活，可以跨项目复用。
  • 团队使用：装 skill 形态 + 在团队基线 CLAUDE.md 里追加项目特定规则。仓库 README 也明确推荐了这种合并方式。
  • 新项目起步：用 curl 直接拉一份到项目根目录的 CLAUDE.md，简单粗暴。
  • 已有 CLAUDE.md 的项目：用 >> 追加到末尾，注意检查和已有规则是否冲突。

无论选哪种形态，都要给自己一个验收标准（这恰恰是准则 4 的应用）：装上之后，diff 里的无关改动是否变少了？模型问澄清问题的时机是否提前到实现前而不是返工后？ 这两个指标如果没改善，那么要么是触发场景没对上，要么是模型对该 skill 的指令遵从度不够，需要进一步调优措辞。
