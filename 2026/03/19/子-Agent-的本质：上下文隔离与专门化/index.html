子 Agent 的本质：上下文隔离与专门化

Created2026-03-19|Updated2026-03-24|技术
|Word Count:2.3k|Reading Time:7mins|Post Views:

“子 Agent”是多 Agent 系统里最常被提起、也最常被说混的概念之一。它常被误解为能力更弱的 Agent、主 Agent 的上下文分叉，或者固定听命于上级的“下属角色”。

这些直觉都抓到了一部分现象，却没有说中它的本质。本文只回答一个问题：子 Agent 到底是什么，它与 Tool、Skill 以及主 Agent 的关系边界在哪里？

三种直觉，三种误解

在给出定义之前，先把三个最常见的误解拆开。

误解一：子 Agent 是能力弱化的 Agent

这种直觉来自“子”字的字面含义——更小、更弱、更受限。但 LangChain 官方文档明确指出：

“An interesting aspect of this approach is that sub-agents may have the exact same capabilities as the main agent.”

子 Agent 可以拥有与主 Agent 完全相同的能力。它不是弱化版，不是受限版，更不是“工具的高级包装”。

误解二：子 Agent 是拥有更小上下文的 fork

这种直觉来自操作系统的进程模型——子进程从父进程 fork，继承部分状态，但资源受限。这个类比只在“相对独立”这一点上有帮助，真正的关键词不是“更小”，而是“独立且干净”。

Anthropic 官方工程博客描述其多 Agent 研究系统时写道：

“Subagents facilitate compression by operating in parallel with their own context windows, exploring different aspects of the question simultaneously before condensing the most important tokens for the lead research agent.”

子 Agent 拥有独立的上下文窗口，而不是主 Agent 上下文的子集。这个窗口是新的、干净的，不是“缩小版主上下文”。

误解三：子 Agent 是听从指挥但能力更强的下属

这种直觉来自组织层级模型——主 Agent 指挥、子 Agent 执行。但这个类比的问题在于，它把“子 Agent”理解成了一种固定身份，而不是一种任务关系。

真正的问题在于：一个 Agent 是否是“子 Agent”，取决于它在当前任务中的角色，而不是它的固有属性。

子 Agent 的真实本质

综合 Anthropic、Claude Code、LangChain 的定义，子 Agent 的本质可以压成一句话：

子 Agent 是一个完整的、专门化的 Agent 实例，运行在独立的上下文窗口中，由主 Agent 通过工具调用方式协调；它的核心价值在于上下文隔离，而不是能力弱化。

五个核心特征

1. 上下文隔离

这是子 Agent 最核心的价值。LangChain 的表述最直接：

“This provides context isolation: each subagent invocation works in a clean context window, preventing context bloat in the main conversation.”

每次调用子 Agent，都是在一个干净的上下文窗口里开始工作。主 Agent 的历史对话、工具调用记录、中间推理过程，都不会自动灌进来。这防止了主 Agent 的上下文膨胀。

2. 完整的 Agent 能力

子 Agent 不是 Tool。Tool 是单一功能的函数，被动执行，没有决策能力，也没有 Agent Loop。子 Agent 则是完整的 Agent 实例，拥有：

  • 独立的推理-行动循环（Agent Loop）
  • 自主规划和决策能力
  • 多步骤任务执行能力
  • 执行期间的短期记忆

3. 无状态执行

子 Agent 是无状态的——它不保留跨调用的长期记忆。每次被主 Agent 调用，都是一次新的执行；跨调用的状态通常由主 Agent 或外部记忆承接。

4. 专门化任务处理

子 Agent 通过 system prompt 或角色约束承担专门职责，可以针对某一类任务聚焦。这种专门化不是能力削弱，而是能力聚焦。

5. 不能递归派发子 Agent

Claude Code 官方文档明确指出：

“Subagents cannot spawn other subagents, so Agent(agent_type) has no effect in subagent definitions.”

子 Agent 不能再派发子 Agent。这是一个刻意的设计约束，用来防止无限递归和控制流失稳，保证行为可预测。

子 Agent 与 Skill 的边界

子 Agent 不能递归派发，但 Skill 可以递归触发。这恰好说明两者不是同一层机制。

这里需要区分两个概念：

概念 本质 递归能力
子 Agent 完整的 Agent 实例，独立上下文窗口 ❌ 不能递归派发
Skill system prompt 的动态注入，上下文内的行为扩展 ✅ 可以递归触发

Skill 本质上是提示词注入，而不是 Agent 实例创建。它发生在同一个上下文窗口内，不会生成新的 Agent Loop。

因此，Skill 的递归与子 Agent 的递归不是一回事。前者是上下文内扩展，后者是新 Agent 实例创建。这个差别也解释了为什么 Agent 创建会被严格约束，而 Skill 组合通常更自由。

“子 Agent”是关系描述，不是能力描述

这是本文最关键的判断。

当我们说某个 Agent 是“子 Agent”时，描述的是它在当前任务中的协作位置，而不是它在系统里的永久身份。

一个最小例子就够了：

在这个任务里，搜索 Agent、研究 Agent、验证 Agent 都是“子 Agent”。但如果用户下一轮直接让“研究 Agent”去做文档调研，它又可以作为独立 Agent 存在。也就是说：

  • “研究”是它的专长；
  • “子 Agent”是它在当前任务里的关系位置。

这就是为什么“子 Agent”首先是关系描述，而不是能力描述。

子 Agent vs Tool：本质区别

既然子 Agent 不是弱化版 Agent，那么它和 Tool 的区别到底是什么？

维度 Tool 子 Agent
本质 单一功能的函数/方法 完整的 Agent 实例
上下文 无独立上下文 拥有独立上下文窗口
决策能力 无，被动执行 有，可自主规划
循环能力 无 拥有 Agent Loop（推理-行动循环）
记忆 无 执行期间有短期记忆
复杂度 单步操作 多步骤复杂任务
状态 无状态 执行期间有状态，调用间无状态

何时用 Tool，何时用子 Agent？

  • Tool：任务是单步的、确定性的、不需要推理的，例如读文件、执行命令、调用 API。
  • 子 Agent：任务需要多步骤推理，并且值得为它分配一个独立上下文。

如果想看 Coding Agent 的整体循环、messages 状态载体、system reminder 和 TODO 外化机制，请直接看《Coding Agent 架构祛魅：从 Claude Code 到 OpenCode 的真实实现》。本文只处理“子 Agent 是什么”这个问题。

设计含义

理解子 Agent 的本质，对多 Agent 系统设计至少有三条直接影响：

1. 不要把“子 Agent”设计成永久层级

如果一个 Agent 永远只能被另一个 Agent 调用、永远不能独立承担任务，这往往说明你把任务关系固化成了架构关系。更好的做法是让 Agent 先拥有独立职责，再在具体任务中形成主从协调。

2. 选择子 Agent 的核心理由是上下文隔离

当你考虑是否要用子 Agent 时，最重要的问题不是“这个任务需要多强的能力”，而是“它值不值得拥有一个独立上下文”。如果任务足够简单，用 Tool 即可；如果任务需要推理但不需要隔离，放在主 Agent 内也可以；只有当任务既需要多步骤推理、又需要隔离上下文时，子 Agent 才真正必要。

3. Skill 解决行为扩展，子 Agent 解决任务委托

这两种机制解决的是不同层面的问题。Skill 扩展的是当前 Agent 的行为能力；子 Agent 委托的是独立任务执行。前者更像“让我学会一个新动作”，后者更像“把这一段工作交给一个拥有独立上下文的执行者”。

结语

子 Agent 的本质是拥有独立上下文的专门化 Agent 实例，而不是能力更弱的 Agent，也不是某种永久下级。

几个关键结论：

  • 子 Agent 是完整的 Agent，可以拥有与主 Agent 相同的能力；
  • 上下文隔离是核心价值，不是能力限制；
  • 子 Agent 不能递归派发，而 Skill 递归不等于 Agent 递归；
  • “子 Agent”是关系描述，不是能力描述；
  • 主子关系是动态的、任务级别的，不是静态架构约束。

理解了这些，多 Agent 系统的很多设计取舍就会变得清晰：什么时候该用 Tool，什么时候该派子 Agent，什么时候只是给当前 Agent 注入一个 Skill，而不是再创建一个新实例。
