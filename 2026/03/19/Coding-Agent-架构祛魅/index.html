Coding Agent 架构祛魅：从 Claude Code 到 OpenCode 的真实实现

Created2026-03-19|Updated2026-03-24|技术
|Word Count:3.5k|Reading Time:11mins|Post Views:

围绕 Coding Agent 的讨论，常见两种极端：要么把它神化成“会自己完成软件开发的智能体”，要么把它贬成“不过是提示词工程”。两种判断都抓不住重点。

本文只回答一个问题：一个 Coding Agent 真正的实现骨架是什么，Claude Code、OpenCode、pi 这些产品又分别在这个骨架上做了什么取舍？

一个循环统治一切

先说结论：所有 Coding Agent 的核心，都是一个 while(tool_use) 循环。

把表层 UI、品牌包装、工具数量全部剥掉之后，剩下的最小骨架大致就是这样：

没有神秘的“自主模块”，也没有独立的停止管理器。模型自己决定什么时候停：当它不再调用工具时，循环自然结束。

这个设计有一个很重要的好处：模型既可以通过工具推进任务，也可以通过“输出纯文本而不调用工具”来提问、解释或汇报进度，不需要额外的对话子系统。

messages 数组：真正的状态载体

上面伪代码中的 context，在主流 LLM API 里的实际落点，通常就是一个 messages 数组。对一个 Coding Agent 来说，状态并不神秘：它的历史、工具调用结果、系统提醒、用户输入，最终都以消息的形式堆进这里。

这个数组会不断增长，因此框架必须回答一个更具体的问题：历史消息怎么保留？

策略 描述 适用场景
无限叠加 保留所有消息，数组持续增长 短任务、大 context 窗口
压缩摘要 用 LLM 总结历史，替换原始消息 超长任务（Claude Code 的 compaction 机制）
滑动窗口 丢弃旧消息，保留最近 N 条 长对话、有限 context 窗口

终止条件的最简实现也因此很清楚：不需要专门的 stop 工具，让模型的“不调用工具”本身成为停止信号，就是最低耦合的做法。

Claude Code：把循环打磨成工程系统

工具集：精简，而不是全能

Claude Code 的工具设计并不追求“覆盖一切”，而是在最小闭环上做取舍。按现有公开分析，可以把它概括为四类：

类别 工具 设计意图
命令行 bash、glob、grep、ls bash 需要用户审批，其余不需要
文件操作 read、write、edit、multi_edit、notebook_read、notebook_edit Jupyter 单独处理，因为原始格式极长
网络 web_search、web_fetch 基础信息获取
控制流 todo_write、task 计划管理与子 Agent 调度

没有额外的 critic 人格，没有庞杂的记忆数据库，也没有层层嵌套的工作流 DSL。这种简单不是能力不足，而是为了让主循环稳定。

TODO 列表：外化的计划

Claude Code 很常见的早期动作之一，是先写一个结构化 TODO 列表：

这个设计解决了一个关键问题：模型在长任务里会逐渐失去对计划的稳定追踪。TODO 列表把计划外化成可读写对象，而不是把“接下来要做什么”全压在模型的上下文记忆里。

更重要的是，框架还能在工具返回结果里反复提醒模型维护 TODO。这说明计划管理不是一次性 prompt，而是运行时持续注入的约束。

这也是 Coding Agent 的一个通用原则：把计划写到外部结构里，比让模型“自己记住计划”更可靠。 如果想看外部记忆的完整分层与文件形态，请直接看《智能体外部记忆：文件标准现状与超越 Harness 的更广泛用途》。本文不展开记忆分层本身。

System Reminder：程序化的上下文注入

Claude Code 会在运行时不断附加 <system-reminder> 之类的上下文块，而且内容会随状态变化：

  • 会话开始：注入基本行为约束（不要主动创建文件、不要做多余的事）
  • TODO 列表为空时：提醒模型考虑是否需要创建计划
  • TODO 列表更新后：把最新的任务状态注入上下文

这不是“把 prompt 写长一点”，而是程序化的上下文管理：根据当前状态，精确控制模型每一步能看到什么。

子 Agent：架构上的逃逸阀

当任务复杂到单个上下文窗口装不下，Claude Code 会通过 task 工具派发子 Agent。对架构来说，子 Agent 的意义主要有两个：

  1. 把一段工作移到独立上下文里执行；
  2. 让多个可拆分任务并行推进。

在 Claude Code 的实现里，子 Agent 不继承主 Agent 的完整上下文历史，而且被刻意限制为不能继续派发子 Agent。这里的重点不是“子 Agent 到底是什么”，而是：当主循环本身快要被上下文塞爆时，子 Agent 是最重要的架构逃逸阀。

如果想看子 Agent 的定义、常见误解，以及它与 Tool、Skill 的边界，请直接看《子 Agent 的本质：上下文隔离与专门化》。本文只说明它在架构里的作用。

安全检查：用小模型做守门员

一个鲜为人知的细节：Claude Code 在执行 bash 命令前，会把命令发给 Claude Haiku（最小最快的模型）做安全检查，判断命令是否读取或修改了敏感文件。

这个设计体现了一个重要的工程权衡：用便宜的小模型做更确定的判断，不让安全检查拖慢主循环。Haiku 的输出是结构化的 XML，而不是自然语言，也降低了解析不确定性。

异构模型协作：主循环用强模型做复杂推理，安全/分类等确定性任务用小模型，是成本与能力的最优分配。

Claude Code 的核心竞争力：上下文工程

上文提到的每一个机制——system reminder、TODO 外化、子 Agent、消息压缩——本质上都在做同一件事：精确控制模型每一步看到什么信息。这也是为什么 Coding Agent 的核心工作，与其说是“会不会写 prompt”，不如说是“会不会做上下文工程”。

这和总纲文里的 PE / CE / HE 三级关系是对得上的：在 Coding Agent 这里，真正重要的控制面已经不只是一句 prompt，而是整个上下文注入与任务推进系统。如果想看这条母定义，请直接看《驾驭工程全景：从 Prompt Engineering 到 Context Engineering 再到 Harness Engineering 的三级跃迁》。本文不再展开总纲层讨论。

OpenCode：交互很好，但 Agent Core 还不够深

OpenCode 是一个开源 Coding Agent，以终端 TUI 为核心交互界面。它的优点和短板都很明显，因此很适合作为对照组。

OpenCode 的设计亮点

事件驱动架构：模块间通过 Event Bus 通信，客户端通过 SSE 订阅事件流，实现实时更新。这让 TUI 的响应性非常好。

声明式 Agent 配置：OpenCode 的"Agent"本质上是配置模板，定义了工具权限、执行权限、system prompt 和模型参数。内置三个 Agent：

  • build：完整权限，适合实际开发
  • plan：只读权限，适合分析和规划
  • general：搜索优化，适合代码研究

多 LLM 提供商支持：通过统一 API 层支持 OpenAI、Anthropic、Google、Amazon 等，这是 Claude Code 不具备的灵活性。

OpenCode 的问题：Agent Core 的深度不足

OpenCode 的 TUI 做得确实好——差异渲染、低闪烁、响应迅速。但如果回到“Coding Agent 的骨架是什么”这个问题，就会发现它在 Agent Core 上还有明显欠缺：

上下文管理缺失：OpenCode 没有类似 Claude Code 的动态 system reminder 机制，也没有成熟的上下文压缩策略。当会话变长，模型行为会退化，但框架层面没有对应的补偿机制。

工具调用兼容性问题：OpenCode 通过统一 AI 抽象层处理工具调用，但对自托管模型的兼容性存在明显缺陷，使用非主流提供商模型时工具调用稳定性较差。

“Agent”更多是配置，不是真正的多 Agent 架构：OpenCode 的 Agent 系统更像 system prompt 和权限模板切换，而不是真正意义上的多 Agent 协调。

system prompt 过长：大量“你是一个工程师，你应该……”之类的角色设定，会占用宝贵上下文，而且未必真的提升决策质量。

这也说明一个常见反模式：把角色设定写得更长，不等于 Agent 会更好。 如果它挤占了真正有用的任务信息，反而会削弱表现。

pi：极简主义对照实验

Mario Zechner 独立开发的 pi 是一个有趣的对照实验。他刻意做减法：

  • 没有内置 TODO 列表：认为这是不必要的复杂性
  • 没有 plan 模式：不需要专门的规划阶段
  • 没有 MCP 支持：认为 MCP 增加了不必要的抽象层
  • 没有子 Agent：单一上下文，简单可控
  • 默认 YOLO 模式：不需要用户逐步确认

pi 的核心贡献之一，是把很多“看似必须存在”的部件砍掉，只保留跑通主循环真正需要的部分。

pi 的意义不在于它一定优于主流产品，而在于它证明了一点：Coding Agent 的骨架并不复杂，真正复杂的是如何把这个循环打磨得稳定、可控、可持续。

行业的真实问题：缺乏有意义的 benchmark

当前 Coding Agent 领域还有一个现实问题：缺乏真正贴近工程交付的 benchmark。

现有的 benchmark（如 SWE-bench）测试的是特定类型的 GitHub issue 修复，与真实开发场景差距很大。这导致：

  • 工具的"优化"可能是针对 benchmark 的过拟合，而非真实能力提升
  • 无法量化"sysprompt 变长"对实际任务的影响
  • 不同工具之间的比较缺乏客观基准

一个真正有价值的 benchmark，应该同时包含需求理解、代码修改、验证、回归检测，而不是只看单点修 bug。否则，很多“功能很多”的系统和“循环打磨得更好”的系统，根本比不出差别。

为什么这些架构会持续收敛

回到本文主问题，可以得到一个更克制也更重要的结论：不同产品的 Coding Agent 之所以越来越像，不是因为大家在互相抄作业，而是因为它们都在解决同一组底层问题——如何让模型在长任务里维持状态、可靠调用工具、必要时拆分子任务，并且在合适的时候停下来。

一旦问题被表述成这样，很多实现会自然收敛到相似骨架：

  • 用循环而不是一次性调用推进任务；
  • 用 messages 承载运行时状态；
  • 用外化计划减少上下文漂移；
  • 用压缩、提醒或子 Agent 处理长任务压力。

这也是为什么 Coding Agent 真正值得比较的，不是界面有多少按钮、系统里有多少名词，而是：它是否把这套最小骨架打磨得足够稳定。 如果想从更高层理解这种收敛为什么会发生，请直接看《驾驭工程全景：从 Prompt Engineering 到 Context Engineering 再到 Harness Engineering 的三级跃迁》。本文到这里为止，只保留与架构骨架直接相关的结论。

结语

把神秘感剥掉之后，Coding Agent 的核心其实并不玄妙：

  • 一个持续运行的工具调用循环；
  • 一个承载状态的 messages 数组；
  • 一套围绕上下文管理展开的工程机制；
  • 若干在复杂任务下才逐步出现的逃逸阀，例如 TODO、压缩、子 Agent 和验证闭环。

所以，Coding Agent 真正值得研究的，不是“它像不像人”，而是“它怎样把一个 LLM 循环打磨成可持续交付的工程系统”。

原文作者：Ryan Lopopolo，OpenAI 技术人员。本文记录了 OpenAI 内部一个工程团队历时五个月、以"零人工编码"方式构建并交付真实软件产品的完整经验。 在过去五个月里，我们的团队一直在进行一项实验：构建并交付一款软件产品的内部 beta 版，其中没有一行代码是人工编写的。 该产品有内部日常活跃用户和外部 Alpha 测试者。它经历了交付、部署、故障和修复的整个过程。与众不同的是，每一行代码 — 从应用逻辑、测试、CI 配置、文档、可观察性到内部工具 — 全都是由 Codex 编写的。据估计，我们只用了手工编写代码所需的大约 1/10 的时间就完成了这项工作。 人类掌舵。智能体执行。 我们有意选择这一限制，以便构建必要的内容，从而将工程速度提升数个数量级。我们用了几周的时间来交付最终达到一百万行代码的项目。为此，我们需要了解，当软件工程团队的主要工作不再是编写代码，而是设计环境、明确意图和构建反馈回路，从而使 Codex 智能体能够可靠地工作时，会发生哪些变化。 这篇文章要说的是，在我们与智能体团队一起从零开始打造一款全新产品的过程中，所...
