驾驭工程全景：从 Prompt Engineering 到 Context Engineering 再到 Harness Engineering 的三级跃迁

Created2026-03-21|Updated2026-03-24|技术
|Word Count:2.3k|Reading Time:7mins|Post Views:

2020 年，我们学会了跟模型说话；2025 年，我们学会了给模型喂信息；2026 年，我们开始给模型搭脚手架。Prompt Engineering、Context Engineering、Harness Engineering 不是并列热词，而是一个控制面不断外移的过程：PE ⊂ CE ⊂ HE。

本文只回答一个问题：为什么 AI 工程的控制面会从 Prompt 外移到 Context，再外移到 Harness？

为什么 Prompt Engineering 不再足够

早期使用大语言模型时，最直接的问题是：怎样提问，模型才会给出更好的回答？

于是，Prompt Engineering 成为第一阶段的核心技能。人们研究 Few-shot、Chain-of-Thought、输出格式控制，本质上都在解决同一个问题：怎样让单次调用的输入更有效。

但当任务从“翻译一段文字”变成“帮我分析代码库、修改模块并验证结果”时，问题就变了。困难已经不再只是“怎么说”，而是：

  • 模型需要知道哪些信息；
  • 哪些信息要先给，哪些后给；
  • 信息过多时如何裁剪；
  • 任务跨多个步骤和多个会话时，状态如何延续。

这意味着 Prompt Engineering 仍然重要，但它已经不足以解释现代 AI 工程。

一张图看懂三级跃迁

理解这条演进路径，先记住下面这张图：

三者的关系不是替代，而是超集：

  • PE 解决表达问题；
  • CE 在 PE 之上，开始管理信息环境；
  • HE 在 CE 之上，开始管理整个任务生命周期。

一个最小类比如下：

  • PE：你学会怎么跟厨师下指令；
  • CE：你学会在正确时间把正确食材和菜谱放到厨师面前；
  • HE：你开始管理整个厨房的交接、质检、库存和任务推进。

如果把这个类比换成工程任务，也很好理解：PE 关心“这次怎么说清楚”，CE 关心“这次推理到底该看到哪些信息”，HE 则开始接管“这个任务如何跨时间、跨步骤、跨失败持续推进”。

类比只用这一次，后面回到严格定义。

第一层：Prompt Engineering 解决了什么

Prompt Engineering 的核心，是优化单次 LLM 调用的输入表达方式。

它解决的问题包括：

  • 输出不稳定；
  • 推理步骤不清；
  • 指令遵循不精确；
  • 新任务缺乏示例时表现差。

用一句话概括：PE 教会了我们如何跟模型说话。

一个最小例子是：

  • 糟糕提示：“帮我分析一下这段代码。”
  • 更好的提示：“请用三点说明这段代码的职责、依赖和潜在风险。”

两者的区别，是表达方式影响了模型的输出结构。

为什么 PE 不够

PE 的边界也很清楚。它主要处理“如何说”，但不处理“模型还需要知道什么”。

当任务进入多步骤场景时，单条 prompt 已经不够了。模型除了看 prompt，还会看：

  • 系统提示词；
  • 工具定义；
  • 检索结果；
  • 对话历史；
  • 先前步骤留下的状态。

这时，工程问题开始从“措辞优化”进入“信息供给优化”。

第二层：Context Engineering 为什么出现

Context Engineering 关注的是：在正确的时间，以正确的格式，把正确的信息交给模型。

所以它不再只是 prompt 技巧，而是信息环境设计。

它至少涉及四类动作：

  1. 选择上下文：哪些信息值得进入当前窗口；
  2. 写入上下文：哪些中间结果要被记录下来，供后续使用；
  3. 压缩上下文：上下文变长时如何保留高信号部分；
  4. 隔离上下文：不同任务是否应由不同 Agent 分担。

一个最小例子是代码任务：

  • 不是把整个仓库塞给模型；
  • 而是只给它当前模块、相关调用方、必要文档和工具结果。

同一个任务如果放在 HE 视角下，问题又会再往前走一步：假设模型已经拿到了正确上下文，但它中途失败、会话中断、或者修改后没有通过验证，那么这时要回答的就不再是“当前窗口里该放什么”，而是“任务状态如何恢复、检查点如何继续、完成条件由谁判断”。这已经超出了 CE 的能力边界。

这时你已经不只是在“写 prompt”，而是在设计一个动态信息供给系统。

为什么 CE 仍然不够

CE 比 PE 强得多，但它仍然主要处理单个上下文窗口内的问题。

也就是说，CE 可以把一次推理组织得非常好，却不能天然保证一个长任务能跨多轮持续完成。它擅长优化“这一次模型怎么看”，但不直接接管“整个任务怎么收口”。

真正的长任务会继续暴露新的限制：

  • 任务跨多个会话时，状态如何恢复；
  • 出错后如何回退并继续，而不是全部重来；
  • 模型怎么知道现在是否真的完成；
  • 多个 Agent 如何在不同上下文之间协作。

换句话说，CE 已经开始像系统设计，但还没有真正接管任务生命周期。

第三层：Harness Engineering 为什么成为新阶段

Harness Engineering 解决的是：如何让智能体持续、可靠、可验证地完成长任务。

它的关注点不再是“当前窗口里放什么”，而是“整个系统如何运转”。

因此，HE 往往包括这些能力：

  • 状态持久化；
  • 验证闭环；
  • 失败恢复；
  • 环境管理；
  • 约束与规则；
  • 多 Agent 编排。

一个最小例子足以说明它为什么出现：

如果一个任务要持续几小时、跨多个会话推进，只会做上下文选择还不够。你还必须回答：

  • 上一轮做到哪了；
  • 哪些检查必须通过；
  • 中断后如何继续；
  • 什么时候才允许退出。

这时，问题已经不是信息组织，而是任务系统设计。

HE 与 CE 的本质区别

这是最容易被混淆的一点，因此需要先给出一句清晰区分：

  • Context Engineering：优化模型在当前推理时看到的信息；
  • Harness Engineering：管理智能体在完整任务生命周期中的行为、状态、验证与恢复。

可以把它们的区别压成一句：

CE 管“当前窗口里该看到什么”，HE 管“整个任务该如何持续完成”。

这里最重要的区别在于：HE 不是“更大的 Context Engineering”，而是把 CE 纳入任务生命周期控制之后形成的系统层能力。

因此，CE 是 HE 的组成部分，但 HE 不止于 CE。

三级关系的真正含义

现在可以把三者关系说得更精确：

  • PE 的核心问题是：如何说；
  • CE 的核心问题是：模型知道什么；
  • HE 的核心问题是：系统如何运转并完成任务。

所以，PE 不会消失。哪怕在 HE 时代，prompt 依然重要。但它已经不再是最高控制面。

同样，CE 也不会消失。Harness 本身往往就要实现很多 CE 策略：选择、压缩、隔离、记忆注入。只是到了 HE，这些能力不再是目标本身，而是任务系统的一部分。

  • 《SDD 与超级个体：AI 时代的人机协作范式》解释为什么规范会成为协作基础设施。
  • 《告别 Vibe Coding：用 OmO 构建可靠的 AI 工程系统》解释 HE 如何在工程系统层面落地。
  • 《智能体外部记忆：文件标准现状与超越 Harness 的更广泛用途》解释状态、记忆与文件标准。
  • 《在智能体优先的世界中利用 Codex》作为固定案例锚点，展示 agent-first 工程的真实组织实践。

结语：控制面的外移

这三级跃迁的本质，不是“AI 热词换了一波”，而是AI 工程的控制面在持续外移。

  • 一开始，人们优化的是一句 prompt；
  • 接着，人们优化的是一个上下文窗口里的信息环境；
  • 现在，人们开始优化整个任务系统。

所以，当我们不再问“AI 能不能写代码”，而开始问“如何让 AI 持续、可靠、可验证地完成复杂任务”时，答案就不再只是 Prompt Engineering，而是这条完整的演进路径：

先学会说话，再学会喂信息，最后学会搭脚手架。
