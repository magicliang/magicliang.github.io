告别 Vibe Coding：用 OmO 构建可靠的 AI 工程系统

Created2026-03-18|Updated2026-03-24|技术分享
|Word Count:2.6k|Reading Time:7mins|Post Views:

过去一年，AI 编程工具从对话式代码生成器进化为能够自主执行复杂任务的智能代理。但真正的挑战从来不只是“让 AI 写出代码”，而是：如何让 AI 持续、可靠、可验证地完成工程任务。

本文只回答一个问题：为什么 Vibe Coding 无法支撑复杂工程，而像 OmO 这样的工程系统可以。

为什么 Vibe Coding 在复杂工程里会失效

Vibe Coding 并非一无是处。对于快速验证想法、搭原型、试交互，它很高效。

问题在于：当任务进入复杂工程时，它的工作方式会迅速暴露结构性缺陷。一个最小例子是这样的：

  • 你让 AI 先改登录流程；
  • 中途发现边界情况没说清；
  • 于是你补 prompt、补上下文、重新解释约束；
  • 改到后面，测试没跑、类型没查、历史状态也散落在聊天记录里。

这时，人并没有退出执行循环，只是变成了一个高强度的“上下文接力员”和“错误兜底员”。

表面上看，AI 参与度很高；但实际上，真正持续维持任务稳定的人仍然是你。它把很多本该由系统承担的隐性工程动作，重新压回了人的临场记忆和即时判断里。

所以，Vibe Coding 真正的问题不是模型不够强，而是它缺少工程控制面。

Vibe Coding 的五个根问题

这组问题可以被压缩成五类，而且这五类问题几乎解释了它为什么难以支撑复杂交付。

1. 上下文接力问题

LLM 的自然工作单元是会话，但软件开发的工作单元是项目。任务一旦跨越多个会话，就必须有人负责“把上一轮发生了什么重新讲给下一轮听”。如果没有外部状态和交接机制，人就会一直被迫站在回路中。

2. 单窗口单角色问题

Vibe Coding 往往把规划、搜索、执行、验证都塞进同一个对话窗口里。这样做的问题不是“看起来不够专业”，而是角色没有分工、上下文没有隔离、并行也无从发生。

3. 完成定义模糊

在 Vibe Coding 里，“done” 常常意味着“看起来差不多”。但真实工程需要更硬的判断：

  • 测试是否通过；
  • 类型是否正确；
  • 构建是否成功；
  • 业务验收标准是否满足。

没有这些边界，任务就无法真正收口。

4. 缺少失败恢复

复杂任务中断、跑偏、或者卡住是常态。没有状态持久化、任务分解和断点续传机制，就只能依靠人工重新组织上下文，从头再来。

5. 缺少历史知识与自动纠偏

代码只能表达“是什么”，却很难表达“为什么是这样”和“未来往哪走”。如果系统没有规则层、历史知识层和自动纠偏机制，AI 就会在看似合理的局部优化中不断积累技术债。

这些问题为什么不能靠更强模型解决

这是最关键的一步判断：这些问题不主要是智力问题，而是系统问题。

更强的模型也许会更会写、更会推理，但它仍然需要回答这些工程问题：

  • 状态放在哪；
  • 谁来判断任务是否完成；
  • 出错后怎样恢复；
  • 不同角色如何协作；
  • 哪些约束必须被强制执行。

例如，一个更强的模型也许能更聪明地猜测“上一步大概做到了哪里”，但它仍然不能凭智力本身替系统提供持久状态、检查点恢复和客观完成条件。这些问题不是“推理得更好”就会自动消失，而是必须由工作流和工程系统显式回答。

换句话说，Vibe Coding 的天花板不是“模型还不够聪明”，而是“系统没有替复杂任务提供足够的控制面”。

如果你希望从更高层理解这种控制面为何会从 Prompt 外移到 Harness，可以直接看《驾驭工程全景：从 Prompt Engineering 到 Context Engineering 再到 Harness Engineering 的三级跃迁》。本文不再重复定义那套模型，只专注于工程系统如何回答这些问题。

OmO 的价值，不在于它“多了几个 Agent”，而在于它把 AI 编程从对话模式改造成了工作流模式。

真正的区别不在界面长什么样，而在于它能不能承接状态、强制验证，并在任务被打断之后继续把工作接上。

  • 拆任务；
  • 选路径；
  • 管状态；
  • 跑验证；
  • 组织多角色协作；
  • 防止“看起来完成”冒充“真的完成”。

OmO 如何逐项对付这五个问题

这里需要把问题与系统回应一一对应起来。

这并不是把原本简单的问题“流程化”成官僚系统，而是把那些原本一直由人类在脑子里手工维护的工程动作外化出来：计划写到哪里、状态记到哪里、失败后从哪里继续、什么条件下才允许说“完成”。

Vibe Coding 的问题 OmO 的系统回应
上下文接力 用计划、任务状态和持久化文件承接跨轮次状态
单窗口单角色 用专业化角色分工，把规划、研究、执行、验证拆开
完成定义模糊 用验证 checkpoint 和验收标准收口
缺少失败恢复 用任务拆解、断点续传、可回滚状态减少“从头再来”
缺少历史知识与自动纠偏 用规则、项目知识和角色互相制衡降低偏差传播

一个最小映射例子如下：

  • 如果问题是“任务跨会话后上下文丢了”，回应不是“再写一个更长 prompt”，而是把状态写到计划、任务清单或外部文件里；
  • 如果问题是“AI 说自己完成了”，回应不是“更相信它一点”，而是要求它过 LSP / build / test / acceptance criteria 这些客观门槛。

同样一个任务，如果仍停留在对话模式里，你通常会不断补 prompt、补上下文、凭感觉判断“差不多完成了”；而一旦进入工作流模式，系统就会要求计划、状态、验证门槛和恢复点都明确存在。两者的差别，不只是更严谨，而是“完成”这件事第一次有了系统意义上的定义。

为什么验证与验收是系统中心

复杂工程里，最危险的不是“AI 一开始写错”，而是“AI 看起来写对了，但系统没有办法及时发现它其实没完成”。

所以，真正把 OmO 与普通对话式 AI 编程拉开差距的，不是角色名字，而是验证与验收被前置成了系统中心。

这里至少有三类最小验证：

  • LSP diagnostics：即时发现类型和静态错误；
  • build / test：判断代码是否真的可运行；
  • acceptance criteria：判断行为是否真的符合目标。

一个最小例子是：

  • 模糊目标：“把登录流程优化一下”；
  • 可验证目标：“保留现有 SSO；新增邮箱登录；关键路径 E2E 通过；LSP 0 error；关键 API p95 < 200ms”。

只有后者，才让“完成”变成机器和人都能判断的条件。

OmO 适合什么，不适合什么

更适合的场景

  • 多步骤、长周期、需要多次验证的任务；
  • 存量系统改造，而不是一次性小实验；
  • 多角色协作、边界复杂、需要明确 done 定义的工程任务。

不适合的场景

  • 一次性的小改动；
  • 只求快速原型、不追求长期可维护性；
  • 任务本身没有明确验收标准，或者没有必要建立完整工作流。

这点很重要：OmO 不是“所有任务都该上”的答案。它是复杂工程条件下的回答，而不是一切 AI 编程行为的默认形态。

结语：不是让 AI 更像人，而是让工程系统更能容纳 AI

Vibe Coding 的问题，从来不只是“生成质量还不够高”，而是：它把复杂工程长期交付所需的状态管理、任务分工、验证机制和恢复能力，统统留给了人类临场兜底。

OmO 这类系统的意义，恰恰在于把这些隐性的工程动作显式化、结构化、系统化。

所以，这篇文章真正想说的不是“OmO 很强”，而是：

当任务复杂到一定程度，问题就不再是‘AI 会不会写’，而是‘系统能不能让 AI 稳定交付’。

这也是为什么这篇文章更接近一篇工程系统宣言，而不是工具使用说明。
