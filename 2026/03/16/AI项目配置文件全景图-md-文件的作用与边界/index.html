AI 项目配置文件全景图：md 文件的作用与边界

Created2026-03-16|Updated2026-03-24
|Word Count:3k|Reading Time:9mins|Post Views:

AI 编程工具普及之后，一个很现实的问题开始出现：项目里到处都是 .md 文件。AGENTS.md、CLAUDE.md、rules/、openspec/……它们分别负责什么，边界又在哪里？

本文只回答一个问题：AI 项目里的这些配置文件分别该负责什么，以及它们为什么不能混用。

这篇文章的核心判断是：这些配置文件可以用“核心问题”和“时态属性”两个维度来区分。 前者决定它们在回答什么问题，后者决定它们适合承载什么类型的信息。

理解这两个维度，就能在配置文件的迷宫里找到方向。

为什么一个 README 已经不够了

传统软件项目往往把大部分高层说明压进 README.md。但在 AI 编程时代，项目需要同时面向两类读者：人类和 智能体。

人类通常需要高层概览，例如“这个项目是做什么的”；智能体则需要结构化、可执行的上下文，例如“代码在哪里、怎么跑起来、哪些约束不能违反”。这两类需求差异很大，单一文件很难同时承载。

于是，项目配置文件开始分化。理解这些文件，本质上就是理解 AI 编程协作中的职责边界。

一个典型 AI 项目的配置全景

一个典型的 AI 辅助项目，配置大致会长成这样：

三类基础配置：AGENTS.md、CLAUDE.md 与 rules/

先用一张表概括三者关系：

配置文件 核心问题 类比
AGENTS.md “谁来做？怎么协作？” 项目地图 + 协作协议
CLAUDE.md “项目级上下文是什么？” 单兵作战手册
rules/ “遇到 X 情况，执行 Y” 场景应对卡片

1. AGENTS.md：项目地图与协作协议

定位：项目级地图与协作协议。
核心职责：定义“谁来做”和“怎么协作”。

典型内容：

  • 项目概述与技术栈
  • 目录结构与模块划分
  • 常用命令（构建、测试、运行）
  • 编码规范与命名约定
  • 服务接口清单（Providers & Consumers）
  • 中间件配置（缓存、数据库、RPC、MQ）
  • 关键依赖版本
  • 反模式清单（已知技术债务）

关键特征：

  • 可以按目录层级嵌套；
  • 强调模块、入口、命令、约束与协作关系；
  • 更像“地图”，而不是长篇百科全书。

2. CLAUDE.md：项目级上下文与行为约束

定位：Claude Code 体系中的项目级指令文件。
核心职责：在系统提示与用户输入之间，提供项目级上下文和行为约束。

典型内容：

  • 项目上下文（技术栈、架构决策、关键依赖）
  • 编码规范（缩进、命名、注释语言）
  • 行为准则（错误处理策略、用户交互时机）
  • 工具使用规范（构建工具、测试框架）
  • 项目特定的最佳实践

关键特征：

  • 更强调当前智能体循环的项目级上下文；
  • 与 Claude Code 运行机制深度绑定；
  • 更像单个智能体的长期作战手册。

与 AGENTS.md 的本质区别：

  • AGENTS.md 更强调项目地图、协作入口和跨模块规则；
  • CLAUDE.md 更强调单个智能体在当前项目里应如何行动。

3. rules/：场景化规则片段

定位：场景化规则目录。
核心职责：为某一类具体场景提供额外指导。

这个例子想说明的不是“AGENTS.md 要写得多长”，而是它最适合承担三种职责：

  1. 项目导航：帮助智能体快速定位代码与命令；
  2. 协作约束：让跨模块关系有据可查；
  3. 稳定背景：承载那些不会随着单次任务频繁变化的信息。

AGENTS.md 与 SDD / OpenSpec 知识库的区别

一个常见误解是：AGENTS.md 就等于 SDD 知识库。它们不是一回事。

维度 AGENTS.md SDD 知识库
定位 项目地图 / 说明书 规范与变更体系
内容范围 项目结构、技术栈、规范 项目上下文 + 规范 + 变更 + 决策
变更追踪 弱 强（由 changes/ 目录承载）
适用场景 快速上手新项目 长期维护的企业级项目

关键点在于：两者可以共存，但不要混用。AGENTS.md 负责地图和稳定背景；SDD / OpenSpec 负责规范、提案和变更流。

实践中的边界规则

与其列出大量“理想栈组合”，不如记住几条边界规则：

  1. 要回答“项目是什么、哪里有什么、有哪些长期约束” → 优先写进 AGENTS.md；
  2. 要回答“当前这个智能体在本项目里该怎么行动” → 优先写进 CLAUDE.md；
  3. 要回答“遇到某类场景时怎么办” → 拆进 rules/；
  4. 要回答“系统现在是什么、这次要变成什么” → 放进 openspec/ / 规范体系；
  5. 要记录任务进展、计划、记忆分层 → 这已经进入外部记忆问题，应该看《智能体外部记忆：文件标准现状与超越 Harness 的更广泛用途》。

这也解释了一个常见误区：AGENTS.md 和 CLAUDE.md 虽然都可能包含规则、命令、目录和约束，但它们的主问题并不相同。前者偏“地图与协作”，后者偏“当前智能体的项目级行动上下文”。

结语

理解 AI 项目中的这些文件，关键不是记住名字，而是看清它们各自回答的问题：

flowchart TB
    A[AGENTS.md\n项目地图与协作协议]
    B[CLAUDE.md\n项目级行动上下文]
    C[rules/\n场景化规则]
    D[openspec/\n规范与变更容器]

    A --> B --> C --> D

如果把“外部记忆”理解为“系统记住什么”，那么这篇文章关心的则是“每类文件分别该负责什么”。一旦职责和时态都清楚了，配置文件就不会再互相争抢角色。
