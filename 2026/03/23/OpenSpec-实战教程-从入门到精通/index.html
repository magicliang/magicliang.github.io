为什么需要 OpenSpec

在 AI 编程时代，一个反复出现的问题是：AI 写出的代码和你想要的不一样。不是 AI 不够聪明，而是你没有把"要什么"说清楚——或者说清楚了，但 AI 在执行过程中遗忘了、偏离了。

OpenSpec 正是为解决这个问题而生的工具。它的核心理念可以用一句话概括：在写代码之前，先让人类与 AI 就规范达成共识（align before code）。

OpenSpec 是一套语义驱动开发（SDD, Spec-Driven Development）的规范体系，通过结构化的 md 文件来管理项目知识、能力规范和变更提案。它不是一个重量级的项目管理工具，而是一个轻量的"人机对齐协议"——用最小的文档开销，换取最大的 AI 输出质量。

设计哲学

OpenSpec 的设计哲学可以归纳为四个关键词：

  • Fluid not rigid：流动而非僵化——规范是活的文档，随项目演进而更新
  • Iterative not waterfall：迭代而非瀑布——不要求一次写完所有规范，支持渐进式完善
  • Easy not complex：简单而非复杂——最小化文档开销，只写必要的内容
  • Brownfield-first：存量优先——专为已有项目设计，不要求从零开始

支持的 AI 工具

OpenSpec 支持 20+ 主流 AI 编程工具，包括：Claude Code、Cursor、Windsurf、GitHub Copilot、Cline、CodeBuddy、Codex、Continue、Gemini、RooCode、Trae 等。这意味着你不需要绑定某个特定的 AI 工具——OpenSpec 产出的规范文件是通用的。

核心概念

理解 OpenSpec 需要掌握六个核心概念。它们之间的关系构成了整个规范驱动开发的骨架。

Specs（行为契约）

Specs 是描述系统当前行为的规范文件。它们回答的问题是：“系统现在能做什么？”

每个 Spec 使用 SHALL/MUST 等语义约束关键词来定义能力边界，采用 Gherkin 风格的场景描述：

Changes（变更提案文件夹）

Changes 是一个文件夹，而非单个文件。每个变更提案对应 openspec/changes/ 下的一个子目录，包含该变更的所有相关工件（Artifacts）。

Artifacts（工件）

每个 Change 文件夹下包含四类工件：

工件 文件 作用 时态
Proposal proposal.md 定义"为什么改、改什么、影响范围" 对齐态
Specs specs/ 本次变更涉及的规范片段（Delta） 增量态
Design design.md 技术方案设计（可选） 设计态
Tasks tasks.md 实现任务清单（后端/前端/测试分项） 迭代态

为什么先产生 proposal，而不是直接写 spec？

这是初学者最常见的困惑：既然叫 OpenSpec，不应该以 spec 为核心吗？为什么 /opsx:propose 命令先产生 proposal.md，然后才附带产生 specs/（Delta Spec）和 tasks.md？

答案在于 OpenSpec 的核心设计理念——align before code（先对齐再编码）。三个工件各有明确的职责分工：

  • proposal.md（对齐态）：回答"为什么做"和"做什么"。它是人类可读的需求文档，用自然语言描述变更的动机、范围和影响。它的标准结构是 ## Why（为什么改）+ ## What Changes（改什么）+ ## Capabilities（能力变更）+ ## Impact（影响范围）。注意：proposal 不包含验收标准——验收标准是 spec 的职责，通过 Gherkin 场景的 GIVEN/WHEN/THEN 来表达。proposal 的核心价值是消除歧义——让 PM、研发、AI 三方在动手之前就达成共识。如果直接跳到 spec，你会发现 AI 写出的 spec 可能精确但方向错误
  • specs/（增量态）：回答"系统行为的精确定义"。Delta Spec 使用 SHALL/MUST 等语义约束关键词和 Gherkin 场景，将 proposal 中的自然语言需求翻译为可验证的行为契约。它是 spec 的增量片段（ADDED/MODIFIED/REMOVED），archive 后会合并进主 specs/
  • tasks.md（迭代态）：回答"具体怎么实现"。它将 spec 中的行为契约拆解为 AI 可逐条执行的任务清单，包含后端、前端、测试等分项

Spec 确实是核心，但 proposal 是 spec 的前提。 这个顺序体现了 OpenSpec “Iterative not waterfall” 的设计哲学——不要求一步到位写出完美的 spec，而是通过 proposal → spec → tasks 的渐进式细化，让每一步都有人工审查的机会。跳过 proposal 直接写 spec，就像跳过需求分析直接写代码一样危险。

Delta Specs（增量规范）

Delta Specs 是 OpenSpec 最精妙的设计之一。它不要求你重写整个规范，而是只描述本次变更的增量：

  • ADDED：新增的需求
  • MODIFIED：修改的需求
  • REMOVED：删除的需求

这种增量设计让每次变更的范围清晰可控，也让 AI 能精确理解"这次要改什么"。

依赖是使能器（enabler），不是门控（gate）。也就是说，你可以在 proposal 还没完全定稿时就开始写 specs，但 specs 的质量会受限于 proposal 的完整度。

Archive（归档）

当一个变更完成后，/opsx:archive 的当前定义并不是“无条件自动合并然后归档”这么简单，而是一个带检查与选择的收口动作：

  1. 先检查 artifact 是否完成
  2. 再检查 tasks.md 是否还有未勾选项
  3. 若存在 Delta Specs，则先评估与主 Specs 的差异，并询问你是否现在执行 sync
  4. 最后再把整个 Change 目录移动到 changes/archive/YYYY-MM-DD-<change-name>/

也就是说，Spec 合并与 Archive 在体验上是强相关的，但语义上并不是同一个动作：合并主 Specs 对应的是 /opsx:sync，Archive 负责最终归档与保留历史记录。

目录结构全景

完整的目录结构：

各文件的定位与职责

config.yaml：项目配置（可选）

定位：项目的结构化上下文入口
核心职责：为 AI 提供项目上下文（context）和项目规则（rules），提升 proposal 生成质量

关键特征：

  • 完全可选——不填也能正常使用 OpenSpec，AI 会从代码库推断上下文
  • 填充后可显著提升 AI 生成 proposal 的质量，尤其对复杂项目
  • 替代了早期版本中的 project.md（Markdown 自由文本 → YAML 结构化数据）
  • 是一次性投入、持续受益的配置——填好后无需频繁维护

specs/：能力规范集合（通过 sync 持续更新）

定位：已同步到主 Specs 的能力规范集合
核心职责：定义系统的"当前事实"

关键特征：

  • 代表系统"当前是什么"
  • 使用语义化约束语言
  • 支持模块化管理

⚠️ 常见困惑：为什么 specs/ 目录一直是空的？

如果你做完了几个需求，但 openspec/specs/ 目录里一直没有文件，最常见的原因不是“propose 失效”，而是你还没把 Delta Specs 同步进主 Specs。

当前本地命令定义里，主 Specs 的更新动作由 /opsx:sync 负责；/opsx:archive 在归档前会评估 sync 状态，并推荐你执行 sync，但也允许你跳过。因此更准确的理解是：

  • changes/<change-id>/specs/：本次变更的 Delta Specs
  • openspec/specs/：已经同步过的主 Specs

只有当你执行了 sync（无论是单独 /opsx:sync，还是在 archive 流程中选择“现在同步”），这些 Delta 才会体现在 openspec/specs/ 中。

changes/：变更提案目录

定位：变更提案的存储目录
核心职责：管理从需求到实现的变更流程

关键特征：

  • 每个需求一个独立子目录
  • 支持变更历史追溯（archive）
  • 与 specs/ 目录形成"增量 + 基线"的关系

与其他配置文件的关系

OpenSpec 不是孤立存在的，它与 AI 编程生态中的其他配置文件形成互补关系：

维度 AGENTS.md CLAUDE.md rules/ openspec/
核心问题 “谁来做？怎么协作？” “项目级上下文是什么？” “遇到 X，执行 Y” “系统是什么？要变成什么？”
定位 多 Agent 编排配置 分层提示机制的 L2 项目配置 场景化规则片段 语义驱动规范体系
生成方式 /init 自动生成 手动创建 手动创建 openspec init 自动生成骨架
维护方式 自动更新 手动维护 手动维护 proposal 驱动增量更新

核心洞察：两者可以共存——用 AGENTS.md 提供快速上下文，用 OpenSpec 知识库驱动规范化的开发流程。

时态属性：理解文件的深层框架

这些文件各自适合承载什么"时态"的信息？这里的"时态"不是语法概念，而是工程概念——一份文档描述的是系统的过去（现状存量）、当下（需求对齐）还是未来（迭代步骤）？

时态 描述 典型载体
存量态（系统现状） 系统当前是什么样的 AGENTS.md、config.yaml、specs/spec.md
对齐态（需求共识） 本次要做什么、影响范围是什么 proposal.md、PRD、用户行为文档
迭代态（执行步骤） 具体怎么做、分几步完成 tasks.md、系统行为 Gherkin

这个分类的工程含义：

存量载体的核心要求是准确。AGENTS.md 里的目录结构、config.yaml 里的项目上下文、specs/spec.md 里的能力定义，必须与代码库的实际状态保持同步。一旦存量信息过时，后续所有基于它生成的 proposal 和 tasks 都会偏离现实。这是 GIGO（Garbage In, Garbage Out）在 AI 工程中最隐蔽的表现形式。

对齐载体的核心要求是可读性。proposal.md 和用户行为文档必须让 PM 和研发都能读懂、都能确认。它的价值不在于技术精确，而在于消除歧义——让人类在 AI 开始写代码之前就达成共识。

迭代载体的核心要求是可执行性。tasks.md 是 AI 的直接输入，必须足够具体，使 Agent 能够按步骤执行，而不需要再做大量推断。

不要用迭代载体（tasks.md）来记录存量，也不要用存量载体（AGENTS.md）来表达对齐意图。混用时态是 AI 工程中常见的配置混乱来源。

完整实操步骤（以"新增下单接口"为例）

第零问：OpenSpec 需要先建立知识库吗？

可选但推荐。 从 v1.2.0 开始，OpenSpec 的知识库载体从 project.md 演变为 config.yaml，且变成了完全可选的。

你可能见过其他 SDD 方案（Spec Kit、Kiro、Taskmaster AI）要求先建立知识库，本质上都是在解决同一个问题：让 AI 在生成规范之前，先理解这个项目是什么、用什么技术栈、有什么约束。OpenSpec 的做法是：

其他 SDD 方案的"知识库" OpenSpec 的对应物 是否必须
项目背景文档 openspec/config.yaml（context 字段） 可选
项目规则/约束 openspec/config.yaml（rules 字段） 可选
架构决策记录（ADR） openspec/specs/<capability>/design.md 通常在 sync 后成为主 Specs 的一部分
已有功能的规范 openspec/specs/<capability>/spec.md 通常在 sync 后成为主 Specs 的一部分
AI 行为规则 AGENTS.md / CLAUDE.md 取决于你使用的 AI 工具

零配置也能用：即使你不填 config.yaml，OpenSpec 仍然可以正常工作——AI 会从代码库本身推断上下文。但填充 config.yaml 可以显著提升 AI 生成 proposal 的质量，尤其是对于复杂项目。

一个很实用的特点是：config.yaml 可以作为一次性投入、持续受益的项目上下文入口——只需在 openspec init 之后填好，后续每个功能的 spec.md 会随着每次 sync 持续沉淀，成为持续增长的知识库。

步骤五：验证实现与规范的一致性（推荐）

在当前 OpenSpec 定义里，/opsx:verify 不是“跑一遍测试”的别名，而是生成一份结构化核对报告。它至少会从三个维度审视这次变更：

  • Completeness：任务有没有做完、需求有没有落地
  • Correctness：实现是否偏离 spec、场景是否有对应实现或测试
  • Coherence：实现是否违背 design，或明显偏离项目既有模式

步骤六：同步主 Specs（推荐）

核心原则：Actions, not Phases

OpenSpec 官方文档（workflows.md）明确提出了一个关键设计哲学：Actions, not Phases（动作，而非阶段）。

这意味着：依赖是使能器（enabler），不是门控（gate）。你不会被"锁"在某个阶段里。在 apply 阶段发现 spec 有问题？完全可以回退修改 spec，然后继续 apply。

Propose 阶段的人机协作模式

OpenSpec 官方 concepts.md 明确定义了 Propose 阶段的人机协作模式：

  1. Human provides intent, context, and constraints.（人类提供意图、上下文和约束）
  2. Agent converts this into behavior-first requirements and scenarios.（AI 将其转化为行为优先的需求和场景）
  3. Agent keeps implementation detail in design.md and tasks.md, not spec.md.（AI 将实现细节放在 design.md 和 tasks.md 中，而非 spec.md）
  4. Validation confirms structure and clarity before implementation.（验证确认结构和清晰度后再实现）

简而言之：人类负责"说什么"，AI 负责"写出来"。proposal.md、spec.md、design.md、tasks.md 这些文件都是 AI 工具生成的产物——但它们不是"不可触碰的圣物"，而是人机协作的中间产物。

Review 发现问题后的三种修正路径

当你审查 Propose 阶段的产出物发现问题时，有三种修正路径，按推荐程度排序：

路径一：使用 /opsx:explore 让 AI 修正（⭐ 推荐）

这是 OpenSpec 设计中最自然的修正路径之一，但要注意它的真实边界：/opsx:explore 的核心是探索、澄清、可视化、对比方案，不是实现代码。它可以在你要求时帮助更新 artifact，但“更新 artifact”是记录思考结果，不是“进入编码模式”。

为什么推荐先从 explore 开始？

  • 它强制你先思考清楚，而不是匆忙进入实现
  • 它可以读取现有 change artifacts 和代码库，帮助你发现遗漏、歧义和隐藏复杂度
  • 当结论稳定后，你再决定是否把这些结论落到 proposal/spec/design/tasks 中

/opsx:explore 的典型使用场景：

  • 需求不明确，需要先调研再决定方案
  • 审查后发现 scope 需要调整（缩小或扩大）
  • 技术方案需要更详细的设计
  • 发现遗漏的边界情况或场景

路径二：自然语言对话让 AI 更新特定文件

路径三：人工直接编辑 md 文件（⚠️ 允许但需谨慎）

OpenSpec 的 md 文件就是普通的 Markdown 文件，你完全可以直接编辑它们。官方 concepts.md 明确说明了每种 artifact 的更新时机：

Artifact 什么时候应该更新
proposal.md Scope 变化、Intent 更清晰、Approach 根本性转变
spec.md 发现遗漏的场景、需求描述不精确、验收标准不完整
design.md 实现发现方案不可行、发现更好的方案、依赖或约束变化
tasks.md 任务拆分不合理、发现遗漏的步骤、顺序需要调整

手工编辑的风险：

  • 你可能只改了 spec 但忘了同步更新 tasks，导致 AI 在 apply 阶段按照旧的 tasks 执行
  • 你可能破坏了 spec 的格式（如 SHALL/MUST 关键词、Gherkin 场景格式），导致 AI 无法正确解析
  • 你可能引入了 proposal 和 spec 之间的不一致

如果你选择手工编辑，建议：

  1. 告诉 AI 你手工改了哪些 artifact
  2. 让 AI 检查 proposal / spec / design / tasks 是否仍然一致
  3. 在 apply 后用 /opsx:verify 看看是否出现新的不一致

为什么必须"先修 Spec 再修代码"？

这个顺序不是教条，而是有深层的工程原因：

1. Spec 是 AI 的"记忆"

归档后，spec 会合并进 openspec/specs/，成为系统的"当前事实"。如果你只修了代码不修 spec，下次开发新功能时，AI 读到的 spec 仍然是错误的——它会基于错误的 spec 生成新的错误代码。这就是 Spec Drift（规范漂移），是 SDD 工作流中最隐蔽的技术债。

2. Spec 是验收标准的唯一来源

如果 spec 说"库存不足返回 409"，但代码实际返回 503，那到底谁是对的？如果 spec 没有定义超时场景，那超时行为就是"未定义行为"——任何实现都"不算错"。先修 spec 就是先明确"什么是对的"，然后让代码对齐这个标准。

3. 闭合循环防止问题复现

⚠️ 关于 /opsx:repair 命令

社区已在 GitHub issue #821 提出了 /opsx:repair 命令的提案，旨在将上述手动 Repair Loop 自动化。该命令的预期行为是：

  1. 分析缺陷的根因（实现问题 vs spec 问题）
  2. 如果是实现问题：只修代码，保持 spec 不变
  3. 如果是 spec 问题：同步更新 spec → design → tasks → 代码
  4. 确保修复后的状态可以直接 archive

OpenSpec Commands / Skills：如何理解这套生态

从你下载的本地文件来看，OpenSpec 这套体系可以理解成两层：

  • .qoder/commands/opsx/*.md：用户显式调用的命令入口
  • .qoder/skills/openspec-*：命令背后的工作流技能定义

两者大多是一一对应的：命令负责“怎么触发”，skill 负责“触发后该怎么做”。因此，理解 OpenSpec 最新行为时，比起背一些过时的仓库统计数字，更重要的是看这些命令和 skill 的职责边界。

Skills 的本质

在 AI 编程工具的生态中，存在四种核心机制，Skills 是其中之一：

机制 触发方式 典型用途 特点
Skills 自动激活（基于描述匹配） 领域知识注入、工作流自动化 上下文感知，零摩擦触发
Slash Commands 手动调用 可重复工作流、参数化任务 显式控制，适合标准化流程
MCP 外部服务集成 数据库、API、文件系统访问 突破沙箱，连接真实世界
Hooks 事件驱动 质量门禁、自动化检查 在特定时机自动执行，不可绕过

用户应该直接操作 Skill，还是让 Copilot 自动调用？

这是一个关键的使用决策，答案取决于你的使用场景：

让 Copilot 自动调用（推荐的默认方式）

适用场景：日常开发中，你只需要用自然语言描述需求，AI 工具会根据 Skill 的描述自动匹配并激活。

优点：零摩擦，不需要记忆命令，自然语言交互。

直接使用 Slash Commands（需要精确控制时）

适用场景：当你需要精确控制工作流的某个步骤，或者自动匹配不够准确时。

优点：精确控制，可重复，适合标准化流程。

本地可见的核心 Skills

你给我的 .qoder/skills 里，当前能直接读到 11 个 OpenSpec 相关 skills。按工作流地位，大致可以分成两组：

类型 Skill 职责

核心 openspec-continue-change 继续创建下一个 artifact

核心 openspec-ff-change 快速生成 apply-ready artifacts

三个最容易讲错的点

1. /opsx:new 不是“生成 proposal 的简写”
它只负责创建 openspec/changes/<name>/，展示第一个 artifact 的说明，然后停下来。后续要靠 /opsx:continue 逐步推进。

2. /opsx:explore 不是“实现前的半自动修文器”
它的本质是 thinking mode。你可以让它捕获结论到 artifact，但它本身不该写业务代码。

3. /opsx:archive 不是“自动合并 specs 的别名”
当前定义里，真正负责 Delta → 主 Specs 合并的是 /opsx:sync。/opsx:archive 会评估 sync 状态并推荐你同步，但允许你跳过后直接归档。

新手的实际建议

  • 想快：直接 /opsx:propose
  • 想稳：/opsx:new 然后反复 /opsx:continue
  • 想先想清楚：先 /opsx:explore
  • 做完实现别急着归档：先 /opsx:verify，再 /opsx:sync，最后 /opsx:archive
  • 第一次上手：优先试 /opsx:onboard，它本身就是一份“带实操的教程”

注意事项与最佳实践

⚠️ 必须注意的陷阱

1. config.yaml 很重要，但不是绝对前提

config.yaml 的确会显著影响 proposal 质量，但从当前本地定义看，它始终被当作“增强上下文的可选入口”，而不是“没有它就无法工作”的前置条件。

2. AGENTS.md / CLAUDE.md 属于 AI 工具层，不应和 OpenSpec 工件混为一谈

无论它们由哪个具体工具生成，本质上都属于 AI 工具自己的项目级配置文件，而不是 OpenSpec 的 proposal / specs / design / tasks 这套工件。两者可以共存：前者提供工具级上下文，后者承载规范驱动开发流程。

3. 归档很重要，但别把 sync 和 archive 混成一个动作

很多人在代码合并后直接 archive，却忽略了 /opsx:sync。这样 change 虽然归档了，但主 Specs 可能并没有同步到最新状态。更稳妥的习惯是：verify → sync → archive。

4. 不要跳过人工审查

OpenSpec 的核心价值在于"人机对齐"。如果你跳过了 proposal 和 spec 的人工审查，直接让 AI 实现，那就失去了 SDD 的意义——你只是在用一个更复杂的方式让 AI 自由发挥。

5. 混用时态会导致混乱

不要在 tasks.md（迭代载体）里记录系统现状，也不要在 specs/spec.md（存量载体）里写待实现的需求。每种文件有其明确的时态定位，混用会让 AI 和人类都感到困惑。

6. 警惕 Spec Drift（规范漂移）

Apply 后发现 bug 时，很多人的本能反应是"直接让 AI 修 bug"。但如果 bug 的根因是 spec 定义不完整或有误，只修代码不修 spec 会导致 Spec Drift——代码的实际行为与 spec 的定义产生偏差。归档后，错误的 spec 会沉淀进知识库，下次 AI 读到它时又会生成错误的代码。正确的做法是：先判断根因，如果是 spec 问题，先修 spec 再修代码。 详见本文「Apply 后的 Spec 缺陷闭合循环」章节。

最佳实践

1. 渐进式采用

不需要一次性把所有功能都写成 Spec。可以从新功能开始，逐步为已有功能补充规范。OpenSpec 的 brownfield-first 设计哲学就是为此而生。

2. 让 AI 帮你填 config.yaml

config.yaml 虽然需要人工审查，但初稿完全可以让 AI 扫描代码库后自动生成。你只需要修正不准确的地方。

3. 保持 Spec 的粒度适中

每个 Spec 应该对应一个独立的"能力"（capability），而不是一个函数或一个类。粒度太细会导致 Spec 数量爆炸，粒度太粗会导致 Spec 失去指导意义。

4. 定期验证规范一致性

使用 /opsx:verify 定期检查 change artifacts 与实现的一致性。它关注的不是单一“是否通过”，而是 completeness、correctness、coherence 三个维度。

5. 善用 archive 的历史记录

changes/archive/ 目录保存了所有已完成变更的历史。当你需要回溯“为什么做了这个改动”时，archive 里的 proposal / design / tasks / delta specs 都可能有价值。不要把它简单理解成“只剩下 spec 与 design 的精简区”。

6. 善用 /opsx:explore 进行迭代审查

不要把 propose 和 apply 当作“一锤子买卖”。/opsx:explore 的价值在于帮你把问题想清楚、把歧义问出来、把方案比较透；想清楚之后，再把结论落回 artifact。这不是“效率低”，而是 SDD “Iterative not waterfall” 的体现。

7. Apply 后发现问题，先定位根因再修复

Apply 后发现缺陷时，先判断是"实现偏离了 spec"还是"spec 本身有问题"。如果是前者，直接修代码；如果是后者，必须先修 spec 再修代码，否则会产生 Spec Drift。修复后建议使用 /opsx:verify 验证 spec 与代码的一致性，确认无误后再归档。

⚠️ archive 后 proposal.md 会消失吗？

就你这次给我的本地 archive.md / openspec-archive-change 定义来看，archive 的动作是把整个 openspec/changes/<name>/ 目录移动到 openspec/changes/archive/YYYY-MM-DD-<name>/。这意味着 proposal.md、design.md、tasks.md、delta specs 会随目录一起进入 archive，而不是只保留其中一部分。

所以，至少对你当前这套本地定义而言，“archive 后 proposal 会被吞掉”这个说法应当删除。更准确的表述是：主 Specs 通过 sync 沉淀系统当前事实，archive 则保留这次 change 的完整历史上下文。

问题的本质：Gherkin 场景 ≠ 量化验收标准

前文提到，OpenSpec 的 spec 文件使用 Gherkin 风格的场景（WHEN/THEN）来表达验收标准。但细心的读者会发现，Gherkin 场景擅长描述"行为"，却不擅长表达"量化指标"。

  1. "不超过 5 秒"是定性描述，不是可自动验证的量化指标——AI 无法直接从这个描述生成一个断言 assert(responseTime < 5000)

OpenSpec 官方文档的沉默

查阅 OpenSpec 官方 concepts.md，你会发现它明确指出：

“Good scenarios: Are testable (you could write an automated test for them)”

但官方文档没有进一步说明如何表达量化验收标准。这是一个设计上的"留白"——OpenSpec 提供了框架，但没有规定具体的验收标准表达格式。

解决方案二：使用独立的 Acceptance Criteria 章节

对于复杂的验收标准，可以在 spec.md 中增加专门的章节：

优点：

  • 结构化、机器可解析
  • 清晰区分功能性和非功能性验收标准
  • 便于追溯（每个 AC 有唯一 ID）

缺点：

  • 不是 OpenSpec 官方推荐格式
  • 需要团队约定这种写法

解决方案三：在 design.md 中定义验收测试方案

对于需要专门测试框架的验收标准（如性能测试、压力测试），建议在 design.md 中定义测试方案：
