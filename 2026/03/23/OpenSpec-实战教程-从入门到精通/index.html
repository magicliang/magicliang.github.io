为什么需要 OpenSpec

在 AI 编程时代，一个反复出现的问题是：AI 写出的代码和你想要的不一样。不是 AI 不够聪明，而是你没有把"要什么"说清楚——或者说清楚了，但 AI 在执行过程中遗忘了、偏离了。

OpenSpec 正是为解决这个问题而生的工具。它的核心理念可以用一句话概括：在写代码之前，先让人类与 AI 就规范达成共识（align before code）。

OpenSpec 是一套语义驱动开发（SDD, Spec-Driven Development）的规范体系，通过结构化的 md 文件来管理项目知识、能力规范和变更提案。它不是一个重量级的项目管理工具，而是一个轻量的"人机对齐协议"——用最小的文档开销，换取最大的 AI 输出质量。

设计哲学

OpenSpec 的设计哲学可以归纳为四个关键词：

  • Fluid not rigid：流动而非僵化——规范是活的文档，随项目演进而更新
  • Iterative not waterfall：迭代而非瀑布——不要求一次写完所有规范，支持渐进式完善
  • Easy not complex：简单而非复杂——最小化文档开销，只写必要的内容
  • Brownfield-first：存量优先——专为已有项目设计，不要求从零开始

支持的 AI 工具

OpenSpec 支持 20+ 主流 AI 编程工具，包括：Claude Code、Cursor、Windsurf、GitHub Copilot、Cline、CodeBuddy、Codex、Continue、Gemini、RooCode、Trae 等。这意味着你不需要绑定某个特定的 AI 工具——OpenSpec 产出的规范文件是通用的。

核心概念

理解 OpenSpec 需要掌握六个核心概念。它们之间的关系构成了整个规范驱动开发的骨架。

Specs（行为契约）

Specs 是描述系统当前行为的规范文件。它们回答的问题是：“系统现在能做什么？”

每个 Spec 使用 SHALL/MUST 等语义约束关键词来定义能力边界，采用 Gherkin 风格的场景描述：

Changes（变更提案文件夹）

Changes 是一个文件夹，而非单个文件。每个变更提案对应 openspec/changes/ 下的一个子目录，包含该变更的所有相关工件（Artifacts）。

Artifacts（工件）

每个 Change 文件夹下包含四类工件：

工件 文件 作用 时态
Proposal proposal.md 定义"为什么改、改什么、验收标准" 对齐态
Specs specs/ 本次变更涉及的规范片段（Delta） 增量态
Design design.md 技术方案设计（可选） 设计态
Tasks tasks.md 实现任务清单（后端/前端/测试分项） 迭代态

Delta Specs（增量规范）

Delta Specs 是 OpenSpec 最精妙的设计之一。它不要求你重写整个规范，而是只描述本次变更的增量：

  • ADDED：新增的需求
  • MODIFIED：修改的需求
  • REMOVED：删除的需求

这种增量设计让每次变更的范围清晰可控，也让 AI 能精确理解"这次要改什么"。

Schemas（依赖图）

Schemas 定义了工件之间的依赖关系：

依赖是使能器（enabler），不是门控（gate）。也就是说，你可以在 proposal 还没完全定稿时就开始写 specs，但 specs 的质量会受限于 proposal 的完整度。

Archive（归档）

当一个变更完成（代码合并到主干）后，通过归档操作：

  1. Delta Specs 中的 ADDED/MODIFIED/REMOVED 内容自动合并进主 Specs
  2. 整个 Change 文件夹移动到 changes/archive/ 目录
  3. 主 Specs 更新为系统的最新状态

各文件的定位与职责

config.yaml：项目配置（可选）

定位：项目的结构化上下文入口
核心职责：为 AI 提供项目上下文（context）和项目规则（rules），提升 proposal 生成质量

关键特征：

  • 完全可选——不填也能正常使用 OpenSpec，AI 会从代码库推断上下文
  • 填充后可显著提升 AI 生成 proposal 的质量，尤其对复杂项目
  • 替代了早期版本中的 project.md（Markdown 自由文本 → YAML 结构化数据）
  • 是一次性投入、持续受益的配置——填好后无需频繁维护

specs/：能力规范集合

定位：已归档的能力规范集合
核心职责：定义系统的"当前事实"

关键特征：

  • 代表系统"当前是什么"
  • 使用语义化约束语言
  • 支持模块化管理

changes/：变更提案目录

定位：变更提案的存储目录
核心职责：管理从需求到实现的变更流程

关键特征：

  • 每个需求一个独立子目录
  • 支持变更历史追溯（archive）
  • 与 specs/ 目录形成"增量 + 基线"的关系

与其他配置文件的关系

OpenSpec 不是孤立存在的，它与 AI 编程生态中的其他配置文件形成互补关系：

维度 AGENTS.md CLAUDE.md rules/ openspec/
核心问题 “谁来做？怎么协作？” “项目级上下文是什么？” “遇到 X，执行 Y” “系统是什么？要变成什么？”
定位 多 Agent 编排配置 分层提示机制的 L2 项目配置 场景化规则片段 语义驱动规范体系
生成方式 /init 自动生成 手动创建 手动创建 openspec init 自动生成骨架
维护方式 自动更新 手动维护 手动维护 proposal 驱动增量更新

核心洞察：两者可以共存——用 AGENTS.md 提供快速上下文，用 OpenSpec 知识库驱动规范化的开发流程。

时态属性：理解文件的深层框架

这些文件各自适合承载什么"时态"的信息？这里的"时态"不是语法概念，而是工程概念——一份文档描述的是系统的过去（现状存量）、当下（需求对齐）还是未来（迭代步骤）？

时态 描述 典型载体
存量态（系统现状） 系统当前是什么样的 AGENTS.md、config.yaml、specs/spec.md
对齐态（需求共识） 本次要做什么、验收标准是什么 proposal.md、PRD、用户行为文档
迭代态（执行步骤） 具体怎么做、分几步完成 tasks.md、系统行为 Gherkin

这个分类的工程含义：

存量载体的核心要求是准确。AGENTS.md 里的目录结构、config.yaml 里的项目上下文、specs/spec.md 里的能力定义，必须与代码库的实际状态保持同步。一旦存量信息过时，后续所有基于它生成的 proposal 和 tasks 都会偏离现实。这是 GIGO（Garbage In, Garbage Out）在 AI 工程中最隐蔽的表现形式。

对齐载体的核心要求是可读性。proposal.md 和用户行为文档必须让 PM 和研发都能读懂、都能确认。它的价值不在于技术精确，而在于消除歧义——让人类在 AI 开始写代码之前就达成共识。

迭代载体的核心要求是可执行性。tasks.md 是 AI 的直接输入，必须足够具体，使 Agent 能够按步骤执行，而不需要再做大量推断。

不要用迭代载体（tasks.md）来记录存量，也不要用存量载体（AGENTS.md）来表达对齐意图。混用时态是 AI 工程中常见的配置混乱来源。

Profile 系统与工作模式

从 v1.2.0 开始，OpenSpec 引入了 Profile 系统，将命令集分为两个层级。你可以通过 openspec config profile 在两种模式之间切换。

Core Profile（默认，推荐起步）

Core Profile 是 OpenSpec 的默认模式，只暴露四个核心命令。这是 OpenSpec “Easy not complex” 设计哲学的直接体现——大多数项目只需要这四个命令就够了。

命令 CLI 形式 Slash 形式 作用 产出物
Propose openspec propose /opsx:propose 创建变更提案 proposal.md + specs/ + tasks.md
Explore openspec explore /opsx:explore 探索和细化提案 更新后的 proposal.md、design.md
Apply openspec apply-change /opsx:apply 执行实现 代码文件 + 测试文件
Archive openspec archive-change /opsx:archive 归档已完成的变更 更新后的 specs/、archive/

注意：CLI 命令名和 Slash 命令名略有不同。CLI 使用 apply-change 和 archive-change（带连字符），Slash 命令使用 /opsx:apply 和 /opsx:archive（简写）。

Expanded Workflow（扩展工作流）

对于更复杂的场景，可以通过 openspec config profile 切换到扩展模式，解锁额外命令：

命令 作用 典型场景
/opsx:new 创建新的变更目录 需要更精细控制提案结构时
/opsx:continue 继续未完成的变更 中断后恢复工作
/opsx:ff 快进（跳过某些步骤） 简单变更不需要完整流程时
/opsx:verify 验证规范一致性 实现完成后检查是否符合规范
/opsx:sync 同步规范与代码 代码变更后更新规范
/opsx:bulk-archive 批量归档 多个变更同时完成时
/opsx:onboard 项目上手引导 新成员加入项目时

何时需要扩展模式？ 当你发现 Core Profile 的四个命令无法满足需求时——比如需要批量归档多个变更、需要在中断后精确恢复工作状态、或者需要跳过某些步骤快速推进简单变更。大多数情况下，Core Profile 已经足够。

完整实操步骤（以"新增下单接口"为例）

第零问：OpenSpec 需要先建立知识库吗？

可选但推荐。 从 v1.2.0 开始，OpenSpec 的知识库载体从 project.md 演变为 config.yaml，且变成了完全可选的。

你可能见过其他 SDD 方案（Spec Kit、Kiro、Taskmaster AI）要求先建立知识库，本质上都是在解决同一个问题：让 AI 在生成规范之前，先理解这个项目是什么、用什么技术栈、有什么约束。OpenSpec 的做法是：

其他 SDD 方案的"知识库" OpenSpec 的对应物 是否必须
项目背景文档 openspec/config.yaml（context 字段） 可选
项目规则/约束 openspec/config.yaml（rules 字段） 可选
架构决策记录（ADR） openspec/specs/<capability>/design.md 随 archive 自动沉淀
已有功能的规范 openspec/specs/<capability>/spec.md 随 archive 自动沉淀
AI 行为规则 AGENTS.md（自动生成，不要手写） 自动生成

零配置也能用：即使你不填 config.yaml，OpenSpec 仍然可以正常工作——AI 会从代码库本身推断上下文。但填充 config.yaml 可以显著提升 AI 生成 proposal 的质量，尤其是对于复杂项目。

一个很实用的特点是：config.yaml 可以作为一次性投入、持续受益的项目上下文入口——只需在 openspec init 之后填好，后续每个功能的 spec.md 会随着每次 archive 自动沉淀，成为持续增长的知识库。

速查决策树

Skills 生态：OpenSpec 的扩展能力

OpenSpec 的 Skills 生态是其最具扩展性的特性之一。Skills 本质上是可复用的工作流模板，让 AI 在特定场景下自动执行预定义的操作序列。

Skills 的本质

在 AI 编程工具的生态中，存在四种核心机制，Skills 是其中之一：

机制 触发方式 典型用途 特点
Skills 自动激活（基于描述匹配） 领域知识注入、工作流自动化 上下文感知，零摩擦触发
Slash Commands 手动调用 可重复工作流、参数化任务 显式控制，适合标准化流程
MCP 外部服务集成 数据库、API、文件系统访问 突破沙箱，连接真实世界
Hooks 事件驱动 质量门禁、自动化检查 在特定时机自动执行，不可绕过

用户应该直接操作 Skill，还是让 Copilot 自动调用？

这是一个关键的使用决策，答案取决于你的使用场景：

让 Copilot 自动调用（推荐的默认方式）

适用场景：日常开发中，你只需要用自然语言描述需求，AI 工具会根据 Skill 的描述自动匹配并激活。

优点：零摩擦，不需要记忆命令，自然语言交互。

直接使用 Slash Commands（需要精确控制时）

适用场景：当你需要精确控制工作流的某个步骤，或者自动匹配不够准确时。

优点：精确控制，可重复，适合标准化流程。

openspec-skills 仓库

OpenSpec 社区维护了一个官方的 Skills 仓库（partme-ai/openspec-skills），包含 15 个预置 Skills，分为 3 个 plugin 分类。这些 Skills 覆盖了从需求分析到代码实现的完整开发流程。

在 SDD 工作流中的 Skill 化架构

将 Skills 应用到三层 Agent 架构：

六项常见任务的机制选择

1. 生成脚手架 → Slash Command
脚手架生成是显式触发的任务，需要用户指定项目名称、模板类型等参数，属于"可重复工作流"。

2. 生成单元测试 → Skill
单元测试生成是 TDD 工作流的自然组成部分，应该在检测到新代码或代码变更时自动激活。

3. 生成集成测试 → Skill + MCP
集成测试需要了解服务间依赖关系，可能需要访问服务目录、API 定义等外部资源。Skill 负责推理"测试什么、如何测试"，MCP 负责获取服务目录和接口定义。

4. 生成 Mock → Skill + 确定性脚本
Mock 生成分为两个阶段：决策（哪些接口需要 Mock、返回什么数据）和执行（生成 Mock 文件）。遵循"概率与确定性分离"原则——LLM 决定"What"，脚本执行"How"。

5. 实现记忆 → CLAUDE.md + PKB Skill
记忆分为两类：持久化规范（存放在 CLAUDE.md/AGENTS.md）和动态知识（通过 PKB Skill 管理）。

6. 特定代码生成 → 视情况而定
模板化代码（CRUD、DTO）用确定性脚本；业务逻辑代码用 Skill；涉及外部 API 的代码用 Skill + MCP。

核心洞察：机制选择不是"非此即彼"，而是"各司其职"。正确的做法是：

  • 让 Skills 处理"什么时候该做什么"（自动感知）
  • 让 Slash Commands 处理"用户明确要求的重复任务"（显式控制）
  • 让 MCP 处理"需要外部资源"（连接真实世界）
  • 让 Hooks 处理"强制执行的质量门禁"（不可绕过）

注意事项与最佳实践

⚠️ 必须注意的陷阱

1. project.md 是天花板，不是地板

AI 生成的 proposal 质量完全取决于 project.md 的完整度。如果你跳过了知识库填充，后续所有的提案都会缺乏上下文，导致 AI 生成的规范偏离实际。

2. AGENTS.md 不要手改

AGENTS.md 是 openspec init 自动生成的，它会在后续操作中自动更新。手动修改会导致与 OpenSpec 的状态不同步。

3. 归档是必须的，不是可选的

很多人在代码合并后忘记执行 openspec archive。不归档的后果是：Delta Specs 不会合并进主 Specs，下次开发时 AI 不知道这个功能已经存在，可能会重复实现。

4. 不要跳过人工审查

OpenSpec 的核心价值在于"人机对齐"。如果你跳过了 proposal 和 spec 的人工审查，直接让 AI 实现，那就失去了 SDD 的意义——你只是在用一个更复杂的方式让 AI 自由发挥。

5. 混用时态会导致混乱

不要在 tasks.md（迭代载体）里记录系统现状，也不要在 specs/spec.md（存量载体）里写待实现的需求。每种文件有其明确的时态定位，混用会让 AI 和人类都感到困惑。

最佳实践

1. 渐进式采用

不需要一次性把所有功能都写成 Spec。可以从新功能开始，逐步为已有功能补充规范。OpenSpec 的 brownfield-first 设计哲学就是为此而生。

2. 让 AI 帮你填 project.md

project.md 虽然需要人工审查，但初稿完全可以让 AI 扫描代码库后自动生成。你只需要修正不准确的地方。

3. 保持 Spec 的粒度适中

每个 Spec 应该对应一个独立的"能力"（capability），而不是一个函数或一个类。粒度太细会导致 Spec 数量爆炸，粒度太粗会导致 Spec 失去指导意义。

4. 定期验证规范一致性

使用 /opsx:verify 定期检查 Specs 与代码的一致性。随着项目演进，Specs 可能会与实际代码产生偏差。

5. 善用 archive 的历史记录

changes/archive/ 目录保存了所有已完成变更的历史。当你需要回溯"为什么做了这个改动"时，archive/ 里的 proposal.md 就是最好的文档。

分工说明：

  • AGENTS.md：定义项目结构、模块协作、服务接口——使多个 Agent 明确职责划分
  • CLAUDE.md：作为分层提示机制的 L2，提供项目级上下文和编码规范——为 Agent Loop 提供持久上下文
  • rules/：特定场景的精细化规则——按需加载，覆盖未涉及的细节
  • openspec/：语义驱动的规范体系——管理知识、规范和变更
